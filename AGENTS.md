# AGENTS.md

## Абсолютное правило языка

Во всем проекте любой текст, который создается или изменяется, должен быть написан на русском языке.

Это правило распространяется на:

- спецификации и требования;
- документацию;
- комментарии в коде;
- сообщения, подсказки, метки интерфейса и другие пользовательские тексты;
- README, заметки, инструкции, TODO и любые другие текстовые материалы проекта.

Если при работе создается или изменяется текстовая информация, она должна быть сформулирована на русском языке. Это правило является обязательным и не допускает исключений внутри проекта.

Технические идентификаторы не считаются пользовательским или документационным текстом. Имена веток, файлов, переменных, функций, классов, импортов, API, пакетов, команд, параметров, ключей JSON, env-переменных и других технических сущностей должны сохраняться в стандартном техническом формате, если этого требует стек, инструмент или интеграция. Комментарии в коде, пользовательские сообщения и документация при этом остаются на русском языке.

## Обязательная сверка со спецификациями

Перед выполнением любой задачи агент обязан сначала понять продуктовый контекст и действующие спецификации.

Минимальное обязательное чтение перед изменениями:

- `спецификации/бизнес-контекст.md`;
- функциональные спецификации, которые прямо относятся к запрошенной задаче;
- связанные спецификации, если задача затрагивает границы между функциями.

Агент не должен начинать реализацию, редактирование файлов или принятие продуктовых решений только на основании свободного текста запроса, если для этой области уже существует спецификация.

Если запрос пользователя соответствует спецификациям, агент должен выполнять задачу в рамках этих спецификаций и не расширять область работ без необходимости.

Если запрос пользователя противоречит бизнес-контексту или функциональным спецификациям, агент обязан остановиться и явно сообщить пользователю, что запрошенное действие выходит за рамки текущих спецификаций или конфликтует с ними. В такой ситуации агент не должен продолжать реализацию, обходить спецификацию или молча принимать новое поведение как разрешенное.

Если нужно изменить поведение, которое уже зафиксировано в спецификациях, это можно делать только по явной просьбе владельца проекта. В таком случае сначала нужно обновить соответствующие спецификации, а уже потом менять реализацию или другие материалы проекта.

Если спецификация неполная, неоднозначная или не покрывает запрошенный случай, агент должен указать на пробел и предложить уточнить или дополнить спецификацию перед выполнением рискованной работы.

Спецификации являются источником истины для продукта. Задача агента - работать внутри этих рамок, а не случайно уводить проект в сторону.

## Правила работы с GitHub

Работа над изменениями в репозитории должна идти через отдельные ветки разработки.

Когда для задачи создается или фиксируется план разработки, агент обязан до начала реализации создать отдельную ветку под этот план и перейти на нее. Новая ветка должна создаваться от актуальной ветки `main`, если пользователь явно не попросил другое основание.

Когда пользователь просит составить план разработки, агент должен сначала вернуться на `main`, обновить `main` из GitHub, создать ветку плана и только после этого создать файл плана в `планы-разработки/активные-планы/`.

Если в рабочей копии есть незакоммиченные изменения, которые мешают переходу на `main`, обновлению `main` или созданию ветки, агент должен остановиться, показать состояние рабочей копии и не переносить, не откатывать и не перезаписывать эти изменения без явного разрешения пользователя.

Имена веток планов должны использовать формат `plan/YYYY-MM-DD-korotkoe-nazvanie`. Часть `korotkoe-nazvanie` должна быть коротким ASCII-slug в нижнем регистре: латинские буквы, цифры и дефисы. В именах веток не нужно использовать русские буквы, пробелы или специальные символы.

Один план разработки должен соответствовать одной рабочей ветке. Нельзя смешивать в одной ветке независимые планы, если пользователь явно не попросил объединить работу.

## Планы разработки

Планы разработки хранятся в папке `планы-разработки/`.

Для планов используются две рабочие папки:

- `планы-разработки/активные-планы/` - планы, которые уже составлены, но еще не полностью разработаны, проверены и завершены;
- `планы-разработки/завершенные-планы/` - планы, работа по которым завершена и которые сохраняются как история разработки.

Шаблон плана хранится в `планы-разработки/шаблон-плана.md`.

Когда пользователь просит составить план разработки, агент должен создать файл этого плана в `планы-разработки/активные-планы/` по шаблону. План должен оставаться активным до тех пор, пока разработка и пользовательское тестирование не завершены.

Каждый план разработки должен фиксировать:

- цель;
- имя рабочей ветки;
- дату создания;
- связанные спецификации;
- затронутые области дорожной карты;
- что входит в план;
- что не входит в план;
- шаги разработки;
- критерии готовности;
- способ пользовательской проверки;
- риски и открытые вопросы.

Когда пользователь сообщает, что тестирование завершено, работа принята и нужно сделать push, агент обязан до commit и push переместить файл соответствующего плана из `планы-разработки/активные-планы/` в `планы-разработки/завершенные-планы/`.

После переноса плана агент должен обновить дорожную карту, а затем продолжить стандартный GitHub-процесс: проверить рабочую копию, сделать commit, выполнить push и создать pull request по прямой просьбе пользователя.

## Дорожная карта

Дорожная карта проекта хранится в папке `дорожная-карта/`.

Общая хронология завершенных работ хранится в `дорожная-карта/хронология.md`.

Работы по отдельным областям хранятся в файлах внутри `дорожная-карта/области/`. Базовые области:

- `дорожная-карта/области/студенты-и-проекты.md` - каталог студентов, карточки, связь студента с проектом и текущие статусы;
- `дорожная-карта/области/github-и-репозитории.md` - GitHub, локальные репозитории студентов, обновления, review-копии и открытие кода;
- `дорожная-карта/области/проверка-и-история.md` - лента обновлений, комментарии, статусы проверки и история ревью;
- `дорожная-карта/области/данные.md` - модель данных, локальное хранение, JSON-схемы, настройки путей и переносимость данных;
- `дорожная-карта/области/фронтенд-и-дизайн.md` - страницы, навигация, состояния интерфейса, визуальная структура и пользовательский опыт;
- `дорожная-карта/области/ии-анализ.md` - локальный ИИ-анализ, ИИ-рапорты, черновики комментариев и проверяемость результата ИИ;
- `дорожная-карта/области/инфраструктура-и-качество.md` - архитектура, структура проекта, инструменты, диагностика и тестирование.

Когда разработка по плану завершена, владелец проекта проверил результат и просит сделать push, агент обязан до commit, push и создания pull request обновить дорожную карту.

Обновление дорожной карты должно включать:

1. запись в `дорожная-карта/хронология.md`;
2. записи во всех файлах областей, которые затронуты завершенным планом;
3. создание нового файла области в `дорожная-карта/области/`, если завершенный план затронул область, которой еще нет в дорожной карте.

Записи в дорожной карте должны идти в хронологическом порядке от старых к новым, добавляться в конец соответствующих разделов и ссылаться на файл завершенного плана из `планы-разработки/завершенные-планы/`.

Если pull request еще не создан на момент обновления дорожной карты, в поле pull request нужно указывать `будет создан после push`. После создания pull request агент не обязан делать отдельный commit только ради добавления номера или ссылки pull request в уже сохраненную дорожную карту, если пользователь явно не попросил обновить эту запись.

После завершения разработки агент не должен автоматически коммитить, отправлять изменения в GitHub или открывать pull request без прямой просьбы пользователя.

Когда пользователь просит завершить работу по плану, агент должен:

1. проверить состояние рабочей копии;
2. переместить файл текущего плана из `планы-разработки/активные-планы/` в `планы-разработки/завершенные-планы/`;
3. обновить `дорожная-карта/хронология.md` и все затронутые файлы областей в `дорожная-карта/области/`;
4. сделать commit с изменениями текущего плана;
5. выполнить push рабочей ветки в GitHub;
6. создать pull request из рабочей ветки в `main`.

После создания pull request агент не должен выполнять merge самостоятельно. Merge выполняет владелец проекта вручную на GitHub.

Если pull request уже создан и пользователь просит продолжить работу или начать следующий план, агент обязан сначала вернуться на ветку `main`, обновить ее из GitHub и только после этого организовывать новый план и создавать новую рабочую ветку.

Если в рабочей копии есть незакоммиченные изменения, которые не относятся к текущему плану, агент не должен перезаписывать, переносить или откатывать их без явного разрешения пользователя. В такой ситуации нужно явно сообщить о состоянии рабочей копии и выбрать безопасное продолжение.

## Инфраструктура Codex проекта

Проектная конфигурация Codex хранится в `.codex/config.toml`.

Репозиторные навыки Codex хранятся в `.agents/skills/`.

Навык `pilot-spec-workflow` является основным рабочим навыком для задач, которые могут затрагивать продуктовые решения, спецификации, реализацию, локальное хранение, ИИ-анализ, интерфейс или документацию проекта.

Если этот навык доступен, агент должен использовать его перед выполнением таких задач.
