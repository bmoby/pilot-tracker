# Supabase-хранилище данных приложения

## Роль этой спецификации

Эта спецификация фиксирует переход Pilot Tracker с локальных JSON-файлов на онлайн-проект Supabase как основное хранилище прикладных данных.

Решение согласовано владельцем проекта. Оно меняет старую рамку первой версии: раньше прикладные данные должны были храниться только локально в JSON, теперь данные курса и история проверки должны храниться в онлайн-проекте Supabase.

При этом Pilot Tracker не становится студенческим кабинетом, командным продуктом или облачной средой исполнения кода. Приложение остается инструментом одного преподавателя.

## Цель

Перенести в Supabase данные, которые описывают:

- студентов;
- проекты и GitHub-ссылки;
- обновления проектов;
- историю проверки;
- комментарии преподавателя;
- статусы проверки;
- задания ИИ-анализа;
- ИИ-рапорты;
- краткие описания проектов, созданные ИИ.

Локальными остаются:

- студенческие репозитории;
- review-копии;
- локальный запуск Codex CLI;
- временные промпты и файлы результата ИИ-анализа;
- машинные настройки путей и локальных инструментов;
- секреты подключения к Supabase.

## Главный принцип

Архитектура становится гибридной:

```text
Supabase
  студенты
  проекты
  история обновлений
  комментарии
  статусы проверки
  задания ИИ
  ИИ-рапорты

Компьютер преподавателя
  интерфейс Next.js через localhost или защищенный сайт администратора
  репозитории студентов
  review-копии
  Git, GitHub CLI, Codex CLI, VS Code
  локальные настройки
  Supabase Auth cookie-сессия администратора
  временные файлы
```

Supabase не хранит код студентов. Код остается в GitHub и в локальных копиях на компьютере преподавателя.

## Выбранное окружение Supabase

Рабочим окружением этой ветки является онлайн-проект Supabase. Локальный Supabase через Docker, включая запуск `supabase start`, не используется как целевая база данных для разработки, проверки миграций или пользовательской работы.

Правила:

- прикладные данные Pilot Tracker записываются в онлайн-проект Supabase;
- миграции схемы применяются к онлайн-проекту Supabase через Supabase CLI или другой управляемый доступ к удаленной базе данных;
- локальный Docker не должен быть обязательным условием запуска приложения, тестов или миграций этой ветки;
- служебные файлы Supabase CLI, например `supabase/.temp/`, являются локальными артефактами и не должны попадать в Git;
- до внедрения `Т012` `service_role` ключ может использоваться только серверным слоем локального приложения для чтения и записи прикладных данных;
- после внедрения `Т012` пользовательский поток чтения и записи должен использовать сессию Supabase Auth, publishable key и RLS, а `service_role` остается только для миграций, служебных скриптов и контролируемых административных операций вне браузера;
- для применения SQL-миграций нужен доступ Supabase CLI к онлайн-проекту или отдельный безопасный доступ к удаленной Postgres-базе.

## Что хранится в Supabase

В Supabase хранятся прикладные данные курса:

- текущая рабочая когорта студентов;
- один основной проект каждого студента;
- GitHub-ссылка проекта;
- текущий снимок проекта: ветка, последний известный коммит, последний статус;
- запуски обновления проектов;
- события обновления проектов;
- снимок GitHub-ссылки, ветки и коммитов на момент обновления;
- комментарии преподавателя;
- статусы проверки обновлений;
- задания очереди ИИ-анализа;
- ИИ-рапорты;
- краткие описания проектов.

Эти данные должны быть доступны после перезапуска приложения и при открытии Pilot Tracker на другом компьютере с тем же подключением к Supabase.

## Что остается локально

Локально хранятся данные и артефакты конкретного компьютера:

- базовая папка данных;
- базовая папка репозиториев;
- базовая папка review-копий;
- команды `git`, `gh`, `codex` и `code`;
- локальная диагностика инструментов;
- локальные копии студенческих репозиториев;
- review-копии выбранных обновлений;
- временные файлы ИИ-запуска;
- локальные диагностические журналы;
- `.env.local` или другой локальный файл с секретами подключения.

Локальные настройки не являются историей курса и не должны храниться в Supabase как общий факт.

## Новый компьютер

Если преподаватель открывает Pilot Tracker на другом компьютере, Supabase сразу дает список студентов, GitHub-ссылки, историю обновлений, комментарии, статусы и старые ИИ-рапорты.

Но локальных репозиториев на новом компьютере может не быть. В таком случае приложение должно показывать состояние:

```text
Проект еще не загружен на этот компьютер.
```

Пока проект не загружен локально, недоступны:

- обновление этого проекта;
- открытие кода в VS Code;
- создание review-копии;
- новый локальный ИИ-анализ.

Приложение должно дать действие загрузки проекта на этот компьютер по GitHub-ссылке из Supabase. После загрузки проект снова можно обновлять, открывать и анализировать локальным ИИ.

Если Pilot Tracker открыт как hosted-сайт в окружении без записываемой папки проекта, чтение прикладных данных Supabase не должно зависеть от создания локального каталога `data/`. Локальные настройки в таком окружении могут использоваться как начальные значения в памяти, а операции с локальными репозиториями, review-копиями, Git, VS Code и Codex CLI остаются недоступными или требуют отдельного локального запуска.

Локальный путь проекта не должен считаться глобальным фактом курса. На разных компьютерах один и тот же проект может лежать в разных папках.

## Подключение и секреты

Подключение к Supabase выполняется из серверного слоя Next.js-приложения, из локальных серверных сценариев или из request-scoped клиента, которому передана действующая cookie-сессия администратора.

Правила:

- секретный ключ Supabase не попадает в клиентские React-компоненты;
- `service_role` ключ не используется в пользовательском клиентском потоке после внедрения `Т012`;
- браузерный и request-scoped клиент используют только publishable key или другой ключ, безопасный для публичного клиента, вместе с RLS;
- секретный ключ Supabase не коммитится в Git;
- секреты хранятся локально, например в `.env` или `.env.local`;
- интерфейс не показывает секреты целиком;
- приложение умеет проверить подключение к Supabase;
- отсутствие подключения не приводит к молчаливой записи в старые JSON-файлы как во второй источник истины.

Pilot Tracker остается инструментом одного преподавателя. После внедрения `Т012` в интерфейсе появляется только один тип учетной записи - администратор-преподаватель. Публичная регистрация, студенческие аккаунты, несколько ролей и управление пользователями из интерфейса не входят в эту модель. Проект не должен зависеть от локального Supabase-стека через Docker.

## Supabase Auth и RLS

Защищенный доступ к данным описывается в `Т012`, но хранилище должно быть совместимо с этой моделью.

Минимальные правила:

- все таблицы Pilot Tracker в схеме `public`, которые доступны через Data API, должны иметь включенный RLS;
- неавторизованный пользователь не должен читать или изменять прикладные данные;
- разрешенный администратор определяется через безопасный источник Supabase Auth, например `app_metadata`, а не через пользовательски изменяемый `user_metadata`;
- публичная регистрация не включается в интерфейсе Pilot Tracker;
- политики RLS должны разрешать чтение и запись только действующему администратору;
- если RLS отклоняет запрос, приложение показывает ошибку доступа или сохранения, а не пустое успешное состояние;
- ключ `service_role` нельзя использовать как способ обойти RLS в обычном пользовательском сценарии.

## Источник истины

После перехода источником истины для прикладных данных становится Supabase.

Правила:

- новые студенты создаются в Supabase;
- проекты и GitHub-ссылки сохраняются в Supabase;
- события обновления записываются в Supabase;
- комментарии и статусы проверки сохраняются в Supabase;
- ИИ-задания и ИИ-рапорты сохраняются в Supabase;
- локальные JSON-файлы не используются как активное хранилище после завершения миграции;
- старые JSON-файлы могут использоваться только как источник миграции и локальный архив старого состояния.

Если Supabase недоступен, создание и изменение прикладных данных блокируются. Приложение не должно показывать успешное сохранение, если запись в Supabase не выполнена.

## Минимальная модель таблиц

Первая Supabase-схема должна сохранить смысл текущей модели данных.

Минимальные таблицы:

- `auth.users` - учетная запись администратора-преподавателя управляется Supabase Auth и не создается миграцией приложения;
- `students` - студенты рабочей когорты;
- `projects` - проекты студентов и текущий снимок проекта;
- `update_runs` - запуски обновления;
- `update_events` - события обновления проектов;
- `review_statuses` - статусы проверки обновлений;
- `comments` - комментарии преподавателя;
- `ai_analysis_jobs` - задания локальной очереди ИИ-анализа;
- `ai_reports` - ИИ-рапорты;
- `project_ai_descriptions` - краткие описания проектов.

Отдельная таблица пользователей приложения не нужна для первой версии, если администратор надежно определяется через `app_metadata`. Если позже понадобится allow-list в базе, это должно быть отдельным уточнением `Т012`.

Идентификаторы сохраняются в текущем строковом формате:

```text
student_<uuid-v4>
project_<uuid-v4>
run_<uuid-v4>
update_<uuid-v4>
comment_<uuid-v4>
review_status_<uuid-v4>
ai_job_<uuid-v4>
ai_report_<uuid-v4>
```

Это упрощает миграцию старых JSON-данных.

## Миграции

Схема Supabase хранится в репозитории:

```text
supabase/
  migrations/
    <timestamp>_initial_pilot_tracker_schema.sql
```

Правила:

- изменения схемы выполняются через новые миграции;
- SQL версионируется в Git;
- миграции проверяются до переноса рабочих данных;
- ручное изменение схемы без миграции не считается устойчивым способом разработки.

## Миграция из JSON

Миграция старых JSON-данных должна быть отдельным этапом.

Минимальный процесс:

1. Прочитать текущие JSON-файлы из `data/app/`.
2. Проверить структуру и связи.
3. Выполнить пробный режим без записи.
4. Создать или проверить резервную копию исходных JSON-файлов.
5. Записать данные в Supabase с сохранением исходных идентификаторов.
6. Сверить количество записей.
7. Переключить приложение на Supabase только после успешной проверки.

Старые JSON-файлы не удаляются автоматически.

## ИИ-анализ

ИИ-анализ остается локальным:

- задание ИИ сохраняется в Supabase;
- локальный worker запускает Codex CLI;
- Codex работает в review-копии на компьютере преподавателя;
- промпт и временный файл результата остаются локальными;
- итоговый ИИ-рапорт сохраняется в Supabase;
- статус проверки не меняется автоматически.

Если Codex успешно завершился, но запись в Supabase временно не удалась, приложение должно сохранить локальный диагностический файл восстановления и явно показать, что рапорт не попал в основную историю.

## Что не входит в первый этап

- студенческий кабинет;
- несколько преподавателей;
- роли пользователей в интерфейсе;
- командная работа;
- несколько курсов или архивы когорт;
- облачный запуск ИИ;
- хранение кода студентов в Supabase Storage;
- автоматическая публикация комментариев в GitHub;
- автоматическая оценка студентов;
- real-time-обновления между устройствами;
- офлайн-режим с последующей синхронизацией.

## Критерии готовности первого этапа разработки

- добавлена Supabase-схема;
- добавлена локальная конфигурация подключения;
- секреты не попадают в Git и клиентский код;
- добавлен серверный модуль подключения;
- добавлен первый слой чтения каталога из Supabase;
- существующее приложение не сломано резким отключением JSON-хранилища;
- проверки проекта проходят.

После внедрения `Т012` дополнительно должно выполняться:

- RLS включен для таблиц Pilot Tracker;
- пользовательский поток работает через администраторскую сессию, а не через `service_role`;
- без входа данные Supabase недоступны из интерфейса;
- после выхода данные Supabase снова недоступны.
