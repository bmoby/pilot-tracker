# Локальное хранилище и файловые операции

## Цель

Зафиксировать, как первая локальная версия Pilot Tracker создает структуру `data/`, хранит локальные настройки и машинные артефакты, работает со старыми JSON-файлами при миграции, выполняет безопасные файловые записи и сообщает об ошибках хранения.

Эта спецификация опирается на модель данных `Т003` и требования `Ф005`. Она не описывает Git-команды обновления репозиториев, запуск ИИ-анализа, открытие VS Code и пользовательский интерфейс.

## Изменение после согласования `Т011`

После согласования `Т011` эта спецификация больше не описывает основное хранилище прикладных данных. Основные данные курса сохраняются в Supabase.

`Т004` остается действующей для локальных артефактов конкретного компьютера:

- локальных настроек;
- локальных копий студенческих репозиториев;
- review-копий;
- временных файлов ИИ-анализа;
- диагностических журналов;
- архивов миграции старых JSON-файлов.

Правила чтения и безопасной записи JSON остаются применимыми к локальным настройкам, диагностике и миграционным архивам, но не должны использоваться как активное хранилище студентов, обновлений, комментариев, статусов и ИИ-рапортов после перехода на Supabase.

## Границы первой версии

Локальный слой первой версии использует файловую систему преподавателя для машинных артефактов, но не является основным хранилищем прикладных данных после `Т011`.

В первую версию локального слоя входят:

- создание папок `data/app/`, `data/repositories/`, `data/review-copies/` и `data/logs/`;
- создание или обновление локального `settings.json`;
- чтение старых JSON-файлов как источника миграции;
- запись локальных настроек, диагностики и миграционных архивов;
- безопасная запись через временный файл и атомарную замену;
- проверка старых JSON-файлов по контракту `Т003` во время миграции;
- обработка ошибок файловой системы;
- переносимые относительные пути внутри `data/`.

В первую версию локального слоя не входят SQLite, локальный Supabase через Docker, собственное серверное развертывание, экспорт данных, облачная синхронизация локальных артефактов и автоматические резервные копии.

## Базовая структура папок

По умолчанию все данные приложения находятся в папке `data/` внутри папки приложения.

Минимальная структура:

```text
data/
  app/
    settings.json
    migration-archive/
  repositories/
    <student-id>/
      <project-id>/
  review-copies/
    <student-id>/
      <update-id>/
  logs/
    diagnostics.jsonl
  backups/
```

Назначение:

- `data/app/` хранит локальные настройки, диагностические данные и архивы миграции старых JSON-данных;
- `data/repositories/` хранит основные локальные копии студенческих репозиториев;
- `data/review-copies/` хранит review-копии для открытия выбранных обновлений;
- `data/logs/` хранит локальные диагностические журналы приложения;
- `data/backups/` зарезервирована для будущих локальных резервных копий.

`data/backups/` может быть создана вместе с остальной структурой, но первая версия не должна автоматически создавать резервные копии, экспортные архивы или отдельный формат выгрузки.

## Пути и переносимость

Правила путей:

- `dataRoot` по умолчанию равен `data`;
- пути `appDataPath`, `repositoriesPath` и `reviewCopiesPath` хранятся относительно `dataRoot`;
- локальный путь проекта хранится относительно `dataRoot`, например `repositories/<student-id>/<project-id>`;
- путь review-копии хранится относительно `dataRoot`, например `review-copies/<student-id>/<update-id>`;
- абсолютные пути вычисляются во время выполнения и не должны быть основным сохраненным значением для данных внутри `data/`;
- имена папок репозиториев строятся из идентификаторов, а не из имени студента или URL репозитория;
- путь не должен выходить за пределы `dataRoot` через `..`, символические ссылки или некорректную нормализацию.

Если преподаватель переносит папку `data/`, приложение должно сохранять связи сущностей за счет относительных путей. Если локальные пути стали недоступны после переноса, приложение должно показать ошибку и позволить исправить базовую папку или путь в настройках.

## Инициализация хранилища

При запуске приложение должно выполнить инициализацию хранилища до чтения пользовательских данных.

Порядок:

1. Определить корень данных `dataRoot`.
2. Создать `dataRoot`, если папка отсутствует.
3. Создать `data/app/`, если папка отсутствует.
4. Создать `data/repositories/`, если папка отсутствует.
5. Создать `data/review-copies/`, если папка отсутствует.
6. Создать `data/logs/`, если папка отсутствует.
7. Создать `data/backups/`, если принято решение создавать резервную папку заранее.
8. Создать отсутствующий `settings.json` или совместимую структуру локальных настроек.
9. Проверить доступность локальных путей и служебных папок.
10. При запуске миграции прочитать старые JSON-файлы из `data/app/` или миграционного архива.
11. Проверить структуру и связи старых JSON-данных перед переносом в Supabase.

Создание отсутствующего локального файла допустимо только для машинных настроек и служебных артефактов. Если старый JSON-файл с прикладными данными существует, но поврежден или не соответствует схеме, приложение не должно перезаписывать его пустой структурой.

## Начальные JSON-файлы

До перехода на `Т011`, если хранилище создавалось впервые, приложение создавало:

- `students.json` с коллекцией `students`;
- `projects.json` с коллекцией `projects`;
- `update-runs.json` с коллекцией `updateRuns`;
- `update-events.json` с коллекцией `updateEvents`;
- `comments.json` с коллекцией `comments`;
- `review-statuses.json` с коллекцией `reviewStatuses`;
- `ai-analysis-jobs.json` с коллекцией `aiAnalysisJobs`;
- `ai-reports.json` с коллекцией `aiReports`;
- `settings.json` с объектом `settings`.

Каждый старый JSON-файл должен содержать `schemaVersion: 1`.

После перехода на `Т011` эти файлы не создаются как активное хранилище прикладных данных. Они могут существовать как источник миграции или локальный архив старого состояния.

JSON записывается в UTF-8, с отступом в два пробела и переводом строки в конце файла.

## Чтение JSON

Правила чтения:

- читать файл как UTF-8;
- разобрать JSON стандартным парсером;
- проверить, что верхнеуровневое значение является объектом;
- проверить `schemaVersion`;
- проверить наличие ожидаемой коллекции или объекта настроек;
- проверить записи по контракту `Т003`;
- проверить связи между файлами после чтения всех коллекций.

Если файл отсутствует при первичной инициализации, он создается с пустой структурой. Если файл отсутствует после уже успешной инициализации, это считается ошибкой хранения и требует явного восстановления, потому что приложение не должно молча терять историю.

## Запись JSON

Запись JSON выполняется только после успешной валидации нового состояния.

Общие правила:

- приложение сначала изменяет состояние в памяти;
- затем валидирует измененные коллекции и связи;
- затем записывает только затронутые JSON-файлы;
- запись одного файла не должна оставлять частично записанный JSON;
- после успешной записи файл должен быть сразу пригоден для чтения после перезапуска.

Сериализация:

- UTF-8;
- отступ в два пробела;
- стабильный порядок верхнеуровневых полей: `schemaVersion`, затем коллекция или `settings`;
- перевод строки в конце файла;
- без комментариев внутри JSON.

## Безопасная запись

Для каждого JSON-файла используется безопасная запись через временный файл в той же папке.

Порядок безопасной записи:

1. Подготовить новый JSON-текст в памяти.
2. Провалидировать JSON-текст повторным разбором или эквивалентной проверкой объекта.
3. Создать временный файл рядом с целевым файлом.
4. Записать весь JSON-текст во временный файл.
5. Сбросить данные временного файла на диск, если платформа поддерживает такую операцию.
6. Атомарно заменить целевой файл временным файлом через переименование.
7. Сбросить метаданные папки на диск, если платформа поддерживает такую операцию.
8. Убедиться, что целевой файл можно прочитать и распарсить.

Временный файл должен иметь имя, которое не пересекается с пользовательскими файлами, например:

```text
.<имя-файла>.tmp-<идентификатор-процесса>-<время>
```

Если запись временного файла или переименование завершается ошибкой:

- исходный файл должен оставаться нетронутым, если операционная система это позволяет;
- временный файл можно удалить как служебный мусор;
- приложение должно вернуть ошибку записи;
- приложение не должно считать данные сохраненными.

## Согласованность нескольких файлов

Одна пользовательская операция может менять несколько JSON-файлов. Например, добавление студента создает запись студента и проекта, а обновление проекта может менять запуск, событие обновления, статус проверки и текущий снимок проекта.

Правила первой версии:

- все изменения операции сначала собираются в памяти;
- перед записью проверяется вся связанная модель данных;
- затронутые файлы записываются последовательно безопасной записью;
- параллельные записи внутри одного процесса сериализуются через очередь операций хранения;
- параллельные задания ИИ-анализа не должны читать старое состояние, а затем перезаписывать чужие новые ИИ-рапорты или статусы заданий; для таких изменений нужна сериализованная операция чтения-изменения-записи или эквивалентный безопасный механизм;
- если запись одного из файлов не удалась, приложение показывает ошибку хранения и не продолжает молча следующую пользовательскую операцию;
- после частично успешной многофайловой операции приложение должно повторно прочитать файлы и проверить согласованность.

Первая версия не реализует полноценные транзакции базы данных. Поэтому операции должны быть спроектированы так, чтобы частичная запись обнаруживалась валидатором и не приводила к скрытой потере истории.

## Создание папок репозиториев

Папка основного репозитория проекта строится по шаблону:

```text
data/repositories/<student-id>/<project-id>/
```

Правила:

- папка студента создается перед клонированием проекта;
- папка проекта создается или передается Git-команде клонирования согласно `Т005`;
- путь сохраняется в `projects.localPath` относительно `dataRoot`;
- приложение не должно создавать папки по имени студента;
- приложение не должно хранить локальные настройки, диагностические данные и миграционные архивы внутри папки студенческого репозитория;
- приложение не должно изменять файлы репозитория через операции хранения.

Если путь проекта уже сохранен, но папка отсутствует или недоступна, операция хранения возвращает ошибку пути. Правила восстановления Git-состояния описываются в отдельной технической спецификации Git.

## Создание папок review-копий

Папка review-копии строится по шаблону:

```text
data/review-copies/<student-id>/<update-id>/
```

Правила:

- review-копия относится к конкретному событию обновления;
- путь review-копии хранится или вычисляется относительно `dataRoot`;
- папка создается только для открытия кода выбранного обновления или ИИ-анализа, если такая копия нужна;
- review-копии не показываются преподавателю как отдельный пользовательский список;
- операции хранения не должны использовать review-копию как источник истории приложения;
- правила создания Git worktree или другой безопасной копии описываются в отдельной технической спецификации review-копий.

## Удаление записей и папок

Удаление записей из JSON выполняется только после пользовательского подтверждения, если удаление затрагивает историю, комментарии, статусы проверки или ИИ-рапорты.

Правила первой версии:

- удаление студента удаляет связанные записи приложения по правилам `Т003`;
- перед записью результата удаления приложение валидирует оставшиеся связи;
- удаление записей JSON выполняется безопасной записью затронутых файлов;
- физические папки основных репозиториев в `data/repositories/` не удаляются автоматически вместе со студентом;
- физические папки review-копий в `data/review-copies/` не удаляются автоматически вместе со студентом;
- если позже потребуется физическое удаление папок, оно должно иметь отдельное подтверждение и отдельное правило спецификации.

Такое разделение снижает риск случайной потери локальной копии проекта при редактировании данных приложения.

## Валидация

Валидация выполняется в трех местах:

- после чтения файлов;
- перед записью файлов;
- после безопасной записи файла, если операция должна подтвердить результат.

Минимальные проверки:

- `schemaVersion` равен `1`;
- обязательные коллекции существуют и являются массивами;
- `settings` существует и является объектом;
- идентификаторы имеют ожидаемый префикс и уникальны внутри коллекции;
- обязательные поля заполнены;
- временные метки имеют формат ISO 8601 UTC;
- статусы и режимы анализа входят в допустимые значения из `Т003`;
- ссылки между сущностями существуют;
- относительные пути не выходят за пределы `dataRoot`;
- пользовательские обязательные строки не пустые после обрезки пробелов;
- файлы JSON не содержат неожиданный верхнеуровневый тип.

Если валидация после чтения не проходит, приложение не должно автоматически исправлять файл без отдельного решения разработчика или явного действия преподавателя.

## Ошибки хранения

Ошибки хранения должны быть явными и пригодными для диагностики.

Минимальные типы ошибок:

- `storage_directory_unavailable` - папка данных отсутствует и ее нельзя создать;
- `storage_permission_denied` - нет прав на чтение или запись;
- `json_file_missing` - обязательный JSON-файл отсутствует после инициализации;
- `json_parse_error` - файл не является валидным JSON;
- `json_schema_error` - JSON не соответствует контракту `Т003`;
- `storage_consistency_error` - связи между файлами нарушены;
- `json_write_error` - файл не удалось записать;
- `json_replace_error` - временный файл не удалось заменить целевым;
- `repository_path_error` - путь основного репозитория отсутствует или недоступен;
- `review_copy_path_error` - путь review-копии отсутствует или недоступен.

Техническая ошибка должна содержать:

- тип ошибки;
- путь файла или папки, если он известен;
- операцию, во время которой произошла ошибка;
- дату и время;
- короткое сообщение для преподавателя на русском языке;
- технические детали для диагностики.

Ошибка хранения не должна удалять существующие данные и не должна заменять поврежденный файл пустым файлом.

## Поведение при поврежденном JSON

Если JSON-файл поврежден:

- приложение прекращает загрузку зависимой части данных;
- показывает ошибку с путем к файлу;
- не перезаписывает поврежденный файл;
- не создает новый пустой файл поверх поврежденного;
- не пытается угадать недостающие исторические данные;
- предлагает восстановить файл вручную из копии данных, если такая копия есть у преподавателя.

Автоматическое восстановление из `data/backups/` не входит в первую версию.

## Очистка служебных временных файлов

При запуске приложение может удалить временные файлы безопасной записи, если они соответствуют внутреннему шаблону имени и не являются текущим целевым JSON-файлом.

Правила:

- удалять можно только временные файлы в `data/app/`;
- удалять можно только файлы с внутренним шаблоном безопасной записи;
- удаление временного файла не должно касаться пользовательских данных;
- если временный файл удалить нельзя, приложение фиксирует диагностическое предупреждение, но не блокирует чтение валидных JSON-файлов.

## Резервные копии

Папка `data/backups/` зарезервирована для будущей версии.

В первой версии:

- автоматические резервные копии не создаются;
- экспортные архивы не создаются;
- восстановление из резервной копии не автоматизируется;
- безопасная запись через временный файл не считается резервной копией;
- преподаватель может вручную копировать папку `data/` средствами операционной системы.

Технический контракт данных должен оставаться пригодным для будущего резервного копирования, но первая версия не должна усложняться отдельным механизмом backups.

## Что не входит сюда

- SQLite или другая локальная база данных;
- серверная часть приложения;
- облачное хранилище локальных артефактов;
- экспорт данных или истории курса;
- автоматические резервные копии;
- Git-команды клонирования и обновления;
- правила создания Git worktree;
- запуск Codex CLI;
- публикация комментариев в GitHub;
- хранение секретов GitHub или OpenAI.

## Критерии приемки

Первая версия локального хранилища считается готовой, если приложение:

- создает структуру `data/app/`, `data/repositories/`, `data/review-copies/` и `data/logs/`;
- создает локальный `settings.json` или совместимую структуру настроек;
- читает старые JSON-файлы как UTF-8 при миграции;
- валидирует старые JSON-файлы перед миграцией;
- записывает локальные JSON-файлы безопасно через временный файл и атомарную замену;
- не перезаписывает поврежденный JSON пустым файлом;
- хранит пути проектов и review-копий относительно `dataRoot`;
- создает папки репозиториев по идентификаторам студента и проекта;
- создает папки review-копий по идентификатору студента и обновления;
- сообщает об ошибках хранения явно и на русском языке;
- не требует SQLite, собственного серверного развертывания, экспорта или автоматических резервных копий для локальных артефактов. После `Т011` прикладные данные курса хранятся в Supabase.
