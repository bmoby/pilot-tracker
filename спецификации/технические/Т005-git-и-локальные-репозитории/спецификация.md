# Git и локальные репозитории студентов

## Цель

Зафиксировать технические правила клонирования и обновления локальных копий студенческих репозиториев в первой версии Pilot Tracker.

Эта спецификация уточняет функциональные требования `Ф002` и `Ф005`: приложение получает последнюю версию ветки `main`, сохраняет результат обновления в локальной истории и не отправляет изменения в репозитории студентов.

## Границы спецификации

`Т005` описывает:

- размещение основных локальных копий студенческих репозиториев;
- клонирование репозитория при первой загрузке;
- повторное обновление через `git fetch`;
- работу только с веткой `main`;
- read-only-ограничения приложения;
- поддержку приватных репозиториев через локальную Git-аутентификацию;
- обработку ошибок Git;
- запись истории обновлений.

`Т005` не описывает:

- создание review-копий для открытия исторических версий в VS Code;
- запуск Codex CLI и ИИ-анализ;
- получение контекста pull request через GitHub CLI;
- пользовательский дизайн экранов;
- формат всех JSON-схем приложения.

Review-копии и открытие в VS Code описывает `Т006`.

## Основной принцип

Основная локальная копия студенческого репозитория является техническим зеркалом ветки `main`, которое приложение использует для обновления, сравнения коммитов и подготовки границы анализа.

Приложение не должно использовать основную локальную копию как рабочую папку преподавателя для правок, экспериментов или проверки исторических версий.

Приложение не должно:

- создавать коммиты в студенческих репозиториях;
- делать `git push`;
- публиковать комментарии в GitHub;
- создавать pull request;
- менять удаленные ветки;
- записывать собственные файлы приложения внутрь студенческого репозитория;
- автоматически исправлять грязное или поврежденное состояние локальной копии.

Допустимые изменения основной локальной копии ограничены Git-операциями, которые приводят локальную папку к состоянию `origin/main` после успешной проверки чистоты рабочей директории.

## Размещение локальных репозиториев

Все основные локальные копии студенческих репозиториев по умолчанию размещаются внутри:

```text
data/repositories/
```

Минимальная структура:

```text
data/repositories/
  <student-id>/
    <project-id>/
      .git/
      ...
```

Правила:

- `<student-id>` и `<project-id>` берутся из локальных данных приложения, а не из имени репозитория на GitHub;
- путь проекта сохраняется в данных проекта;
- путь должен быть относительным к базовой папке данных, если это возможно для переносимости;
- приложение не должно размещать основные локальные копии вне настроенной базовой папки локальных репозиториев; по умолчанию это `data/repositories/`;
- если базовая папка была перенесена, приложение должно уметь пересобрать абсолютный путь из сохраненного относительного пути и настройки `data/`;
- если сохраненный путь указывает наружу из настроенной базовой папки локальных репозиториев, приложение должно считать это ошибкой конфигурации для первой версии.

Папка `data/repositories/` хранит только основные локальные копии, которые обновляются до `main`. Review-копии должны находиться в `data/review-copies/` и описываются в `Т006`.

## Связь проекта с локальной копией

Для каждого проекта нужно хранить:

- идентификатор студента;
- идентификатор проекта;
- GitHub-ссылку на репозиторий;
- локальный путь основной копии;
- имя ветки, в первой версии всегда `main`;
- последний известный коммит `main`;
- дату последнего успешного получения состояния;
- результат последнего обновления;
- последнюю ошибку, если она есть.

GitHub-ссылка сохраняется как часть проекта и как снимок внутри каждого события обновления. Это нужно, чтобы старые события истории оставались понятными, даже если ссылка в карточке студента была изменена позже.

## Поддерживаемые URL репозиториев

Первая версия должна поддерживать стандартные GitHub-ссылки, которые локальный Git может использовать для `git clone` и `git fetch`.

Допустимые формы:

```text
https://github.com/<owner>/<repo>.git
https://github.com/<owner>/<repo>
git@github.com:<owner>/<repo>.git
```

Перед запуском Git-команд приложение выполняет базовую валидацию:

- ссылка не пустая;
- ссылка похожа на репозиторий GitHub;
- ссылка не содержит управляющих символов;
- ссылка передается в Git как отдельный аргумент, а не как часть shell-строки.

Приложение не должно собирать Git-команды через конкатенацию shell-строк. Команда и аргументы должны передаваться как массив аргументов дочернего процесса.

## Ветка `main`

Первая версия работает только с веткой `main`.

Правила:

- при клонировании приложение явно запрашивает `main`;
- при обновлении приложение явно получает `origin/main`;
- если в репозитории нет ветки `main`, обновление этого проекта завершается ошибкой;
- приложение не должно автоматически переключаться на `master` или другую ветку;
- имя ветки сохраняется в событии обновления;
- вся граница анализа для `Ф004` строится по ветке `main`.

Поддержка других веток может быть добавлена только отдельным изменением спецификаций.

## Первый clone

Если у проекта есть GitHub-ссылка, но основной локальной копии еще нет, приложение выполняет первую загрузку.

Команда:

```bash
git clone --branch main --single-branch <github-url> <local-path>
```

После успешного клонирования приложение выполняет:

```bash
git rev-parse HEAD
```

Ожидаемое поведение:

- папка проекта по умолчанию создается внутри `data/repositories/<student-id>/<project-id>/`;
- локальная копия находится на ветке `main`;
- `HEAD` соответствует последнему полученному коммиту `main`;
- полученный коммит сохраняется как новый известный коммит проекта;
- создается событие обновления с результатом первой загрузки;
- первая загрузка считается успешным обновлением проекта.

Если целевая папка уже существует, но не является ожидаемым пустым местом или ожидаемым Git-репозиторием этого проекта, приложение не должно перезаписывать ее автоматически. Нужно сохранить ошибку локального пути.

## Повторное обновление

Если основная локальная копия уже существует, приложение обновляет ее безопасной последовательностью.

Базовая последовательность:

```bash
git status --porcelain
git fetch origin main --prune
git rev-parse --verify origin/main^{commit}
git rev-list --count <last-known-commit>..origin/main
git reset --hard origin/main
git rev-parse HEAD
```

Правила:

- все команды выполняются в основной локальной копии проекта;
- перед `fetch` приложение проверяет, что папка является Git-репозиторием;
- `git status --porcelain` должен вернуть пустой вывод;
- если рабочая директория не чистая, обновление проекта останавливается;
- `git fetch origin main --prune` получает актуальное состояние удаленной ветки;
- `git rev-parse --verify origin/main^{commit}` проверяет, что удаленная ветка существует и указывает на коммит;
- количество новых коммитов считается между последним известным коммитом и `origin/main`;
- `git reset --hard origin/main` допускается только после проверки чистой рабочей директории;
- после `reset` приложение повторно читает `HEAD` и сохраняет его как новый коммит события.

Приложение не использует `git pull`, потому что первая версия не должна создавать merge-коммиты или зависеть от локальной ветки, которая могла разойтись с `origin/main`.

## Последний известный коммит

Последний известный коммит проекта хранится в данных проекта и обновляется только после успешного обновления основной локальной копии.

Для события обновления нужно сохранять:

- старый коммит, если он был известен до запуска;
- новый коммит, если он был получен;
- количество новых коммитов;
- признак наличия новых изменений;
- ветку `main`;
- локальный путь основной копии;
- GitHub-ссылку на момент запуска;
- результат Git-операции.

Если старый коммит отсутствует, событие считается первой загрузкой или первым успешным состоянием проекта. В этом случае количество новых коммитов можно сохранить как `null` или как отдельное значение первой загрузки, но событие должно иметь новый коммит.

## Переписанная история удаленного репозитория

Если `last-known-commit` больше не достижим из `origin/main`, обычный подсчет:

```bash
git rev-list --count <last-known-commit>..origin/main
```

может завершиться ошибкой или дать результат, который не отражает линейное обновление.

Для первой версии это считается ошибкой состояния репозитория.

Правила:

- приложение не должно молча заменять старый коммит новым;
- приложение не должно удалять историю Pilot Tracker;
- событие обновления сохраняется с ошибкой;
- в ошибке нужно указать, что предыдущий известный коммит не найден в истории `main` или история была переписана;
- преподаватель должен увидеть, что проект требует ручного решения.

Автоматическое восстановление после force-push не входит в первую версию.

## Read-only-режим приложения

Read-only означает, что Pilot Tracker не создает авторских изменений в студенческом коде и не отправляет изменения в GitHub.

Разрешено:

- клонировать репозиторий в `data/repositories/`;
- получать объекты Git через `fetch`;
- приводить чистую основную локальную копию к `origin/main`;
- читать Git-историю;
- читать diff;
- читать файлы проекта для проверки и ИИ-анализа.

Запрещено:

- изменять файлы проекта от имени приложения;
- записывать служебные файлы Pilot Tracker внутрь репозитория студента;
- выполнять `git add`;
- выполнять `git commit`;
- выполнять `git push`;
- выполнять `git merge`;
- выполнять `git rebase`;
- выполнять `git checkout` на произвольные ветки в основной копии;
- выполнять автоматическое исправление конфликтов или локальных изменений.

Если локальная копия стала грязной, приложение сохраняет ошибку и не пытается угадать, какие изменения можно удалить.

## Приватные репозитории

Приватные репозитории поддерживаются только через локально настроенную Git-аутентификацию преподавателя.

Правила:

- приложение не хранит GitHub-пароли;
- приложение не хранит собственный GitHub-токен для Git-операций первой версии;
- приложение не пытается обходить права доступа GitHub;
- если локальный Git может выполнить `clone` или `fetch`, приватный репозиторий обновляется как обычный;
- если доступ отсутствует, ошибка сохраняется в событии обновления;
- ошибка одного приватного репозитория не останавливает обновление остальных проектов.

Диагностика ошибки должна быть понятной для преподавателя, но не должна сохранять секреты из вывода Git.

## Ошибки Git и локальной копии

Ошибка одного проекта не должна останавливать общий запуск обновления.

Минимальные категории ошибок:

- `repo_url_missing` - у студента нет GitHub-ссылки;
- `repo_url_invalid` - ссылка не похожа на допустимый GitHub-репозиторий;
- `repo_access_denied` - репозиторий недоступен или требует аутентификации;
- `repo_not_found` - GitHub вернул отсутствие репозитория или локальный Git не смог его найти;
- `network_error` - сеть недоступна или соединение оборвано;
- `git_not_available` - команда Git не найдена или недоступна;
- `main_branch_missing` - ветка `main` отсутствует;
- `local_path_missing` - сохраненный локальный путь отсутствует;
- `local_path_not_repository` - локальная папка не является ожидаемым Git-репозиторием;
- `local_repository_dirty` - рабочая директория содержит локальные изменения;
- `local_repository_corrupted` - Git-репозиторий поврежден или команды Git не могут читать его состояние;
- `history_rewritten` - предыдущий известный коммит не найден в истории `origin/main`;
- `git_command_failed` - Git-команда завершилась ошибкой, не попавшей в более точную категорию.

Для каждой ошибки нужно сохранять:

- категорию;
- человекочитаемое сообщение;
- техническое сообщение для диагностики;
- имя Git-команды;
- код завершения, если он есть;
- дату и время;
- связь со студентом, проектом и запуском обновления.

Техническое сообщение не должно включать пароли, токены или другие секреты.

## История обновлений

Каждая попытка обновления должна создавать событие истории.

Это правило действует для:

- первой загрузки;
- успешного обновления с новыми коммитами;
- успешного обновления без новых коммитов;
- студента без GitHub-ссылки;
- ошибки Git;
- ошибки локальной копии.

Общий запуск обновления создает сводный объект запуска и отдельные события по каждому студенту. Одиночное обновление проекта со страницы студента создает одно событие проекта и может быть связано с отдельным запуском из одного проекта.

Старые события не перезаписываются. Обновляться могут только сводные поля проекта: последний статус, последний известный коммит, дата последнего успешного обновления и последняя ошибка.

## Порядок записи события обновления

Для надежности приложение должно фиксировать попытку обновления даже при ошибке.

Рекомендуемый порядок:

1. Создать запись запуска обновления со статусом `выполняется`.
2. Для каждого проекта создать или подготовить событие обновления со статусом `выполняется`.
3. Выполнить валидацию ссылки и локального пути.
4. Выполнить Git-операции, если они разрешены.
5. Записать результат события: успех, отсутствие изменений, недоступность или ошибка.
6. Обновить сводные поля проекта.
7. После обработки всех проектов завершить запись запуска обновления сводкой.

Если приложение аварийно завершилось во время обновления, при следующем запуске незавершенные события должны быть видны как незавершенные или прерванные, а не исчезать.

## Граница анализа для ИИ

После успешного получения состояния `main` приложение сохраняет границу анализа для `Ф004`.

Минимальные данные:

- локальный путь основной копии;
- ветка `main`;
- старый коммит;
- новый коммит;
- количество новых коммитов;
- дата и время фиксации границы;
- GitHub-ссылка на момент обновления.

`Т005` не выполняет ИИ-анализ и не получает контекст pull request. Эти задачи относятся к `Ф004` и `Т009`.

## Безопасность запуска команд

Git-команды должны запускаться как дочерние процессы без shell-интерполяции пользовательских данных.

Правила:

- команда и аргументы передаются отдельными значениями;
- GitHub-ссылка и локальные пути не вставляются в shell-строку;
- рабочая директория команды явно задается для существующих локальных копий;
- для clone целевой путь заранее проверяется как путь внутри настроенной базовой папки локальных репозиториев;
- stdout и stderr можно сохранять только в очищенном диагностическом виде;
- секреты и токены не должны попадать в JSON-файлы приложения.

## Что не входит в первую версию

- поддержка веток кроме `main`;
- автоматическое восстановление после force-push;
- автоматический выбор `master`, если `main` отсутствует;
- push, commit, merge или rebase студенческих репозиториев;
- публикация комментариев в GitHub;
- создание pull request;
- хранение GitHub-пароля или собственного GitHub-токена;
- встроенный просмотр diff в интерфейсе сразу после обновления;
- полноценное управление конфликтами локальной копии;
- автоматическая очистка или пересоздание поврежденных основных копий без решения преподавателя.

## Критерии приемки

Техническая реализация `Т005` считается готовой, если:

- новые репозитории по умолчанию клонируются в `data/repositories/<student-id>/<project-id>/`;
- клонирование явно использует ветку `main`;
- существующие репозитории обновляются через `fetch` и приведение чистой копии к `origin/main`;
- отсутствие ветки `main` сохраняется как ошибка;
- приложение не выполняет `git push`, `git commit`, `git merge` и `git rebase`;
- грязная локальная копия не исправляется автоматически и сохраняется как ошибка;
- приватный репозиторий обновляется при наличии локального доступа преподавателя;
- отсутствие доступа к приватному репозиторию сохраняется как ошибка одного проекта;
- каждый запуск и каждое событие обновления сохраняются в истории;
- старые события истории не перезаписываются;
- для успешного обновления сохраняются локальный путь, ветка, старый коммит, новый коммит и количество новых коммитов;
- ошибка одного проекта не останавливает обновление остальных.

## Открытые вопросы

На текущий момент открытых вопросов по `Т005` нет.

## Решения первой версии

- Основная локальная копия каждого проекта по умолчанию хранится в `data/repositories/`.
- Основная локальная копия используется только для получения актуального состояния `main`.
- Единственная поддерживаемая ветка первой версии - `main`.
- Для первого получения проекта используется `git clone --branch main --single-branch`.
- Для повторного обновления используется `git fetch origin main --prune` и приведение чистой копии к `origin/main`.
- Локальная копия считается read-only для приложения как источник кода и анализа.
- Приложение не создает коммиты и не делает push в репозитории студентов.
- Приватные репозитории поддерживаются только через локальный доступ преподавателя.
- Каждая попытка обновления оставляет запись в истории Pilot Tracker.
