# Технический запуск ИИ-анализа

## Цель

Зафиксировать технический способ запуска локального ИИ-анализа выбранного обновления проекта в первой версии Pilot Tracker.

Спецификация описывает:

- подготовку безопасного локального контекста анализа;
- запуск локального Codex CLI через `codex exec`;
- передачу промпта через stdin;
- использование схемы результата;
- сохранение финального ответа через `--output-last-message`;
- получение контекста pull request через GitHub CLI;
- сохранение ИИ-рапорта в локальном хранилище;
- ограничения режима чтения;
- базовую обработку ошибок запуска.

## Продуктовая рамка

ИИ-анализ является вспомогательным инструментом преподавателя. Он запускается только вручную из деталей конкретного обновления и не принимает итоговое решение за преподавателя.

Техническая реализация должна соблюдать решения `Ф004` и `Ф005`:

- приложение запускает только локально установленный Codex CLI;
- приложение не вызывает OpenAI API напрямую;
- приложение не поддерживает альтернативных ИИ-провайдеров или адаптеров;
- приложение не хранит API-ключи OpenAI;
- приложение не управляет авторизацией Codex напрямую;
- ИИ работает в режиме чтения;
- результат сохраняется как ИИ-рапорт внутри выбранного обновления;
- черновик комментария остается частью ИИ-рапорта и не публикуется автоматически;
- статус проверки обновления остается ручным решением преподавателя.

## Условия доступности запуска

Действие запуска ИИ-анализа доступно только для обновления, у которого есть достаточная техническая граница анализа.

Минимальные обязательные данные:

- идентификатор студента;
- идентификатор проекта;
- идентификатор обновления;
- ссылка GitHub на момент обновления, если она была указана;
- локальный путь проекта;
- путь к review-копии или возможность создать ее;
- ветка;
- старый коммит, если он известен;
- новый коммит;
- дата и время фиксации границы анализа;
- количество новых коммитов;
- команда или путь к локальному Codex CLI из настроек;
- путь к GitHub CLI из настроек, если он используется для pull request.

Если обязательных данных нет, запуск не начинается. Приложение должно сохранить или показать понятную причину недоступности, не создавая успешный ИИ-рапорт.

## Рабочая папка анализа

Codex CLI должен запускаться не в основной локальной копии проекта, которая используется для обновления `main`, а в безопасной папке анализа.

Предпочтительный вариант первой версии:

- приложение создает или переиспользует review-копию выбранного обновления;
- review-копия соответствует `new commit` выбранного обновления;
- review-копия находится внутри `data/review-copies/`;
- рабочая папка Codex указывает на эту review-копию;
- основная локальная копия проекта не переключается на исторические коммиты.

Если review-копию создать нельзя, запуск ИИ-анализа завершается ошибкой. Приложение не должно запускать Codex в основной копии проекта как запасной путь, если это может изменить состояние репозитория для будущих обновлений.

## Команда запуска Codex CLI

Базовая форма запуска:

```bash
codex exec -C <путь-к-review-копии> -s read-only --output-schema <путь-к-json-schema> --output-last-message <путь-к-файлу-финального-ответа>
```

Правила запуска:

- команда Codex берется из настроек локальных инструментов;
- рабочая папка задается через `-C`;
- режим песочницы задается как `read-only`;
- промпт передается через stdin;
- `--output-schema` указывает на локальную JSON-схему ожидаемого результата;
- `--output-last-message` указывает на временный файл, куда Codex сохраняет финальный ответ;
- приложение ожидает завершения дочернего процесса и фиксирует код выхода;
- stdout и stderr процесса используются только для диагностики и не являются основным пользовательским рапортом.

Если техническая версия Codex CLI не поддерживает один из обязательных флагов, приложение должно считать локальный инструмент несовместимым для первой версии и показать диагностическую ошибку настройки.

## Передача промпта через stdin

Промпт не передается как аргумент командной строки. Приложение записывает полный промпт во stdin дочернего процесса.

Это нужно, чтобы:

- не зависеть от ограничений длины командной строки;
- не раскрывать полный контекст анализа в списке процессов;
- стабильно передавать длинный diff, историю коммитов и контекст pull request;
- упростить диагностику через сохранение нормализованной копии входных данных в данных ИИ-запуска.

Приложение должно формировать промпт из структурированных блоков:

- задача анализа;
- граница обновления;
- данные студента и проекта;
- краткое описание проекта, если оно уже есть;
- предыдущие комментарии преподавателя по этому обновлению, если они есть;
- предыдущие ИИ-рапорты этого обновления, если анализ запускается повторно;
- локальные Git-данные по диапазону коммитов;
- контекст pull request, если он доступен;
- требования к структуре ответа;
- напоминание, что выводы ИИ должны проверяться преподавателем.

Промпт должен явно запрещать:

- менять файлы проекта;
- создавать коммиты;
- делать push;
- публиковать комментарии в GitHub;
- выставлять итоговый статус проверки;
- выдавать черновик как финальный комментарий преподавателя.

## Локальные Git-данные для анализа

Перед запуском Codex приложение может собрать технический контекст по сохраненной границе анализа.

Минимальные команды, которые допустимо выполнять в review-копии или безопасной локальной папке:

```bash
git log <старый-коммит>..<новый-коммит>
git diff <старый-коммит>..<новый-коммит>
git show <коммит>
```

Правила:

- команды должны быть read-only;
- ошибки Git не должны менять локальный репозиторий;
- если `старый коммит` отсутствует при первой загрузке, приложение должно сформировать доступный контекст от начального известного состояния проекта;
- если diff слишком большой для безопасной передачи, приложение должно передать в промпт сводку, список файлов и диагностическую отметку об усечении;
- факт усечения должен попасть в технические детали ИИ-рапорта.

## Контекст pull request через GitHub CLI

Так как студенты могут работать через pull request, приложение должно попытаться получить контекст pull request, если GitHub CLI доступен и локально авторизован.

Приложение использует локально настроенный GitHub CLI и не хранит GitHub-пароль или собственный GitHub-токен.

Минимальный контекст pull request:

- номер pull request;
- заголовок;
- описание;
- автор;
- исходная ветка;
- целевая ветка;
- состояние;
- список коммитов;
- список измененных файлов;
- комментарии обсуждения, если доступны;
- review-комментарии, если доступны.

Поиск pull request должен быть привязан к репозиторию и диапазону коммитов выбранного обновления.

Базовая стратегия первой версии:

- определить репозиторий из сохраненной GitHub-ссылки;
- получить список pull request, связанных с коммитами диапазона, если GitHub CLI позволяет это сделать;
- если точное сопоставление невозможно, использовать pull request целевой ветки `main`, который содержит `new commit`, при доступности такой проверки;
- если pull request не найден, продолжить анализ только по локальному Git-диапазону.

Ошибка GitHub CLI не блокирует ИИ-анализ diff. Она сохраняется как диагностическая часть ИИ-рапорта.

Приложение не должно выполнять через GitHub CLI действия, которые публикуют комментарии, меняют pull request, закрывают его, делают merge или изменяют репозиторий.

## Схема результата

`--output-schema` используется для стабильной структуры финального ответа Codex. Схема должна находиться в локальной технической папке приложения или в другом месте, определенном реализацией, и версионироваться вместе с кодом.

Минимальная структура результата должна позволять сохранить:

- краткое резюме обновления;
- список новых коммитов или описание диапазона;
- основные изменения;
- важные файлы для ручного просмотра;
- возможные проблемы и зоны риска;
- вопросы для ручной проверки преподавателем;
- черновик комментария преподавателя;
- данные о контексте pull request;
- предупреждения об усечении входных данных;
- технические детали запуска;
- отметку, что выводы ИИ требуют проверки преподавателем.

Схема не должна содержать поле для автоматической оценки студента или автоматического статуса проверки. Если Codex возвращает такую рекомендацию в свободном тексте, приложение не должно применять ее как статус.

## Сохранение финального ответа

Флаг `--output-last-message` должен указывать на временный файл внутри служебной папки ИИ-запуска.

После завершения процесса приложение:

- проверяет, что файл финального ответа создан;
- читает финальный ответ;
- проверяет соответствие ожидаемой структуре;
- нормализует данные для локального хранилища;
- сохраняет ИИ-рапорт внутри выбранного обновления;
- сохраняет технические детали запуска;
- удаляет или оставляет временные файлы по правилам диагностики из `Т010`.

Файл `--output-last-message` не является долгосрочным источником истины. Источник истины после успешной обработки - запись ИИ-рапорта в локальном JSON-хранилище приложения.

## Сохранение ИИ-рапорта

ИИ-рапорт сохраняется в данных приложения, описанных в `Ф005`, внутри связи с выбранным обновлением.

Минимально сохраняются:

- идентификатор ИИ-рапорта;
- идентификатор студента;
- идентификатор проекта;
- идентификатор обновления;
- дата и время начала запуска;
- дата и время завершения;
- статус: `готов` или `ошибка`;
- старый коммит;
- новый коммит;
- локальный путь проекта на момент анализа;
- путь review-копии на момент анализа;
- краткое резюме;
- важные файлы;
- описание изменений;
- возможные проблемы и зоны риска;
- вопросы для ручной проверки;
- черновик комментария преподавателя;
- полный текст рапорта или нормализованное представление результата;
- диагностические данные pull request;
- технические детали запуска;
- текст ошибки, если анализ не завершился успешно.

Повторный запуск анализа создает новый ИИ-рапорт. Старые ИИ-рапорты не перезаписываются без явного действия преподавателя.

## Ошибки запуска

Ошибка ИИ-анализа должна сохраняться как неуспешный ИИ-рапорт внутри выбранного обновления, если запуск уже начался или был запрошен преподавателем.

Минимальные ошибки:

- отсутствует локальный путь проекта;
- отсутствует review-копия и ее нельзя создать;
- локальный репозиторий недоступен или поврежден;
- отсутствует старый или новый коммит, необходимый для выбранного режима анализа;
- Git-команда анализа завершилась ошибкой;
- Codex CLI не найден;
- Codex CLI не поддерживает нужный режим запуска;
- Codex CLI не авторизован локально;
- процесс Codex завершился с ненулевым кодом;
- финальный ответ не был создан;
- финальный ответ не соответствует схеме;
- результат не удалось сохранить в локальное хранилище;
- GitHub CLI недоступен или завершился ошибкой при получении pull request.

Ошибка GitHub CLI не должна блокировать анализ локального diff. Все остальные ошибки оцениваются по месту: если без них нельзя получить проверяемый результат, запуск завершается неуспешным ИИ-рапортом.

## Диагностические данные запуска

Для каждого ИИ-запуска нужно сохранять достаточно данных, чтобы преподаватель или разработчик мог понять причину сбоя.

Минимальные диагностические данные:

- дата и время начала и завершения;
- команда Codex без секретов;
- рабочая папка;
- режим песочницы;
- путь схемы результата;
- путь временного финального ответа;
- код выхода процесса;
- краткий stderr, если он есть;
- факт доступности GitHub CLI;
- результат получения контекста pull request;
- факт усечения diff или других входных данных;
- версия локального инструмента, если она доступна без сетевого запроса.

Диагностика не должна сохранять секреты, токены, пароли или данные локального окружения, которые не нужны для понимания ошибки.

## Что не входит в первую версию

В эту техническую спецификацию не входит:

- прямой вызов OpenAI API из приложения;
- хранение API-ключей OpenAI;
- поддержка альтернативных ИИ-провайдеров;
- выбор модели внутри приложения;
- автоматический запуск ИИ после обновления проекта;
- автоматическая оценка студента;
- автоматическое изменение статуса проверки;
- автоматическая публикация комментариев в GitHub;
- создание коммитов или push из ИИ-запуска;
- полноценная проверка CI/CD студенческого проекта.

## Критерии приемки

Техническая реализация соответствует этой спецификации, если:

- ИИ-анализ можно запустить только вручную из выбранного обновления;
- Codex CLI запускается через `codex exec`;
- промпт передается через stdin;
- рабочей папкой является review-копия выбранного обновления;
- используется режим `read-only`;
- используется схема результата через `--output-schema`;
- финальный ответ сохраняется через `--output-last-message`;
- ИИ-рапорт сохраняется внутри выбранного обновления;
- повторный запуск создает новый ИИ-рапорт;
- контекст pull request получается через GitHub CLI, если он доступен;
- ошибка GitHub CLI не блокирует анализ локального diff;
- ошибки Codex и сохранения видны внутри выбранного обновления;
- приложение не изменяет файлы студенческого проекта;
- приложение не публикует комментарии и не меняет pull request;
- приложение не выставляет итоговый статус проверки автоматически.
