# Структура проекта и среда выполнения

## Цель

Зафиксировать ожидаемую организацию кода, границы модулей, среду выполнения и команды разработки для первой локальной версии Pilot Tracker.

Эта спецификация должна помочь реализовать функциональные требования без смешивания интерфейса, доменной логики, локального хранилища и интеграций с внешними инструментами.

## Контекст

Первая версия Pilot Tracker остается локальным приложением для одного преподавателя и одной активной рабочей группы студентов.

Техническая структура должна поддержать:

- список студентов и проектов из `Ф001`;
- обновление репозиториев и историю обновлений из `Ф002`;
- страницу студента, ленту обновлений, комментарии и статусы из `Ф003`;
- локальный ИИ-анализ через Codex CLI из `Ф004`;
- локальные JSON-данные, репозитории, review-копии и настройки из `Ф005`;
- платформенную рамку из `Т001`.

Структура проекта не должна зависеть от окончательного выбора интерфейса. Настольный интерфейс, локальный веб-интерфейс на `localhost` или командная строка должны использовать одни и те же сценарии приложения, модели данных, хранилище и интеграционные модули.

## Границы

Эта спецификация определяет:

- разделение кода по слоям;
- назначение основных модулей;
- правила зависимостей между слоями;
- ожидаемые команды разработки;
- требования к среде выполнения;
- соглашения по именованию и экспорту модулей;
- границы между пользовательским интерфейсом, сервисами, хранилищем и интеграциями.

Эта спецификация не определяет:

- конкретный фреймворк интерфейса;
- точную схему каждого JSON-файла;
- внутренний формат ИИ-рапорта;
- полный набор экранов;
- точные Git-обертки;
- реализацию безопасных файловых записей;
- окончательную систему тестов.

Эти детали должны быть уточнены в связанных технических спецификациях.

## Решения первой версии

- Код должен быть организован вокруг локального приложения, а не серверной системы.
- Интерфейсный слой не должен напрямую работать с файловой системой, Git, Codex CLI, GitHub CLI или VS Code.
- Все пользовательские действия должны проходить через слой сценариев приложения.
- Доменная модель должна быть независима от конкретного интерфейса и способа запуска.
- Локальное хранилище должно быть отдельным слоем, который скрывает чтение и запись JSON-файлов.
- Интеграции с Git, VS Code, Codex CLI и GitHub CLI должны быть отдельными модулями.
- Слой сценариев приложения может использовать хранилище и интеграции, но не должен зависеть от деталей отображения интерфейса.
- Команды разработки должны быть едиными и воспроизводимыми: запуск, сборка, проверки качества, тесты и подготовка данных.
- Пока интерфейс не выбран, структура должна оставлять место для одного или нескольких адаптеров интерфейса без дублирования бизнес-логики.

## Требования

### Базовая структура кода

Проект должен быть организован так, чтобы в коде явно читались следующие области:

```text
src/
  app/
  domain/
  storage/
  integrations/
  ui/
  config/
  diagnostics/
  shared/
```

Названия папок могут быть уточнены во время реализации, если выбранный стек требует другого расположения файлов. При этом смысловое разделение слоев должно сохраниться.

Назначение областей:

- `app/` хранит сценарии приложения и координацию пользовательских действий;
- `domain/` хранит сущности, статусы, правила и проверки предметной области;
- `storage/` хранит доступ к локальным JSON-файлам и файловым операциям приложения;
- `integrations/` хранит адаптеры Git, VS Code, Codex CLI и GitHub CLI;
- `ui/` хранит интерфейсные адаптеры и компоненты выбранного интерфейса;
- `config/` хранит чтение и нормализацию настроек приложения;
- `diagnostics/` хранит техническую диагностику и преобразование ошибок;
- `shared/` хранит общие типы и утилиты, которые не принадлежат одному сценарию.

### Правила зависимостей

Зависимости между слоями должны идти в одну сторону:

- `ui/` вызывает `app/`;
- `app/` использует `domain/`, `storage/`, `integrations/`, `config/` и `diagnostics/`;
- `storage/` использует `domain/` и `shared/`;
- `integrations/` использует `config/`, `diagnostics/` и `shared/`;
- `domain/` не зависит от `ui/`, `storage/` и `integrations/`;
- `shared/` не должен превращаться в место для случайной бизнес-логики.

Запрещенные зависимости:

- интерфейс напрямую читает или записывает JSON-файлы;
- интерфейс напрямую запускает Git, Codex CLI, GitHub CLI или VS Code;
- доменная модель знает о конкретных путях файловой системы;
- интеграционные модули изменяют данные приложения без сценария приложения;
- хранилище само запускает внешние процессы.

### Сценарии приложения

Сценарии приложения должны быть основным публичным программным интерфейсом для пользовательских действий.

Минимальные группы сценариев:

- студенты и проекты;
- обновление проектов;
- история обновлений;
- комментарии преподавателя;
- статусы проверки;
- открытие кода выбранного обновления;
- запуск ИИ-анализа;
- настройки и проверка локальных инструментов.

Каждый сценарий должен:

- принимать структурированный вход;
- проверять базовые предусловия;
- вызывать доменные правила;
- обращаться к хранилищу через репозитории или сервисы хранения;
- обращаться к локальным инструментам через интеграционные адаптеры;
- возвращать структурированный результат;
- сохранять ошибку в нужной предметной сущности, если это требуется функциональной спецификацией.

### Доменная модель

Доменный слой должен описывать основные сущности первой версии:

- студент;
- проект;
- запуск обновления;
- событие обновления;
- комментарий преподавателя;
- статус проверки обновления;
- ИИ-рапорт;
- краткое описание проекта;
- настройки приложения;
- ошибка или диагностическое событие.

Доменный слой должен содержать перечисления статусов, которые используются в функциональных спецификациях, и правила, которые не зависят от интерфейса.

Примеры доменных правил:

- имя студента обязательно;
- один студент связан с одним основным проектом;
- студент может существовать без GitHub-ссылки;
- статус проверки относится к конкретному обновлению;
- ИИ-рапорт относится к конкретному обновлению;
- ошибка одного проекта не отменяет общий запуск обновления;
- старые события обновления не перезаписываются.

### Хранилище

Слой хранилища должен отвечать за локальные данные приложения.

Минимальные обязанности:

- создание базовой структуры `data/`;
- чтение JSON-файлов из `data/app/`;
- запись JSON-файлов в `data/app/`;
- загрузка и сохранение студентов, проектов, обновлений, комментариев, статусов, ИИ-рапортов и настроек;
- проверка отсутствующих или поврежденных файлов данных;
- подготовка данных к восстановлению после перезапуска приложения.

Хранилище не должно:

- выполнять Git-команды;
- открывать VS Code;
- запускать Codex CLI;
- получать контекст pull request;
- решать, какой пользовательский сценарий выполнить дальше.

Подробные правила хранения должны быть описаны в `Т003` и `Т004`.

### Интеграции

Интеграционные модули должны быть адаптерами над локальными инструментами.

Минимальный состав:

- `git` - клонирование, проверка состояния, получение `origin/main`, подсчет новых коммитов и обновление основной локальной копии;
- `vscode` - открытие review-копии выбранного обновления;
- `codex` - запуск локального Codex CLI для ИИ-анализа;
- `github` - получение контекста pull request через GitHub CLI, если он используется;
- `process` - безопасный запуск дочерних процессов, если стек требует отдельной общей обертки.

Интеграционные модули должны возвращать структурированные результаты и ошибки. Они не должны напрямую показывать сообщения пользователю и не должны самостоятельно записывать историю приложения.

### Интерфейс

Интерфейсный слой должен быть адаптером над сценариями приложения.

Он отвечает за:

- ввод данных преподавателем;
- отображение списков, лент, деталей обновления и состояний ошибок;
- подтверждения опасных действий;
- запуск сценариев приложения по пользовательским действиям;
- отображение результатов сценариев.

Интерфейс не отвечает за:

- правила Git-обновления;
- формат JSON-хранилища;
- построение ИИ-промпта;
- сохранение ИИ-рапорта;
- вычисление доменных статусов вне сценариев приложения.

### Среда выполнения

Среда выполнения первой версии должна поддерживать локальные операции:

- доступ к файловой системе;
- запуск дочерних процессов;
- выполнение Git-команд;
- запуск VS Code;
- запуск Codex CLI;
- запуск GitHub CLI, если нужен контекст pull request;
- чтение и запись локальных JSON-файлов.

Если выбранный интерфейс работает в браузере, доступ к файловой системе и дочерним процессам должен выполняться через локальный слой приложения, а не напрямую из браузерного кода.

Если выбран командный интерфейс, он должен использовать те же сценарии приложения, что и графический интерфейс.

### Команды разработки

Проект должен предоставить единый набор команд. Точные имена могут зависеть от выбранного менеджера пакетов, но должны быть зафиксированы в репозитории перед началом реализации.

Ожидаемый минимальный контракт:

```text
dev          запуск локальной версии для разработки
build        сборка или проверка собираемости проекта
test         запуск автоматических тестов
lint         статическая проверка стиля кода
typecheck    проверка типов, если стек поддерживает типизацию
format       форматирование кода
check        общая проверка перед сдачей изменений
```

Если часть команд неприменима к выбранному стеку, это должно быть явно зафиксировано в техническом решении по реализации, а не молча отсутствовать.

Команды не должны требовать облачных сервисов, серверного развертывания или учетной записи внутри приложения.

### Соглашения по модулям

Модули должны быть небольшими и привязанными к предметной ответственности.

Требования:

- публичные функции сценариев приложения должны называться по пользовательскому действию;
- доменные типы и статусы должны храниться в доменном слое;
- адаптеры интеграций должны иметь единый способ возвращать успешный результат и ошибку;
- операции с путями должны проходить через слой настроек или хранилища;
- пользовательские тексты и сообщения ошибок должны быть на русском языке;
- тесты должны размещаться рядом с кодом или в отдельной папке по единому правилу проекта;
- временные файлы интеграций не должны смешиваться с JSON-данными приложения.

### Тестируемость

Структура проекта должна позволять тестировать слои отдельно:

- доменные правила без файловой системы;
- сценарии приложения с подмененным хранилищем и подмененными интеграциями;
- хранилище на временной папке данных;
- интеграции через контролируемые команды или фиктивные адаптеры;
- интерфейс через сценарии пользователя после выбора интерфейса.

Минимальная стратегия тестов должна быть уточнена в `Т010`.

## Что не входит

В эту спецификацию не входят:

- выбор Electron, Tauri, браузерного фреймворка, серверного рантайма или другого конкретного стека;
- утверждение финальной структуры экранов;
- утверждение полной структуры `data/app/*.json`;
- миграции данных;
- формат схемы ИИ-рапорта;
- точная реализация Git worktree или review-копий;
- точные команды `codex exec`;
- правила публикации пакета или установщика;
- серверное развертывание;
- облачная синхронизация;
- многопользовательский доступ.

## Критерии приемки

Структура проекта и среда выполнения считаются согласованными, если:

- в коде можно явно отделить интерфейс, сценарии приложения, доменную модель, хранилище и интеграции;
- интерфейс не работает напрямую с JSON-файлами и внешними процессами;
- сценарии приложения являются основной точкой входа для пользовательских действий;
- доменные правила можно тестировать без локальных репозиториев и файлов `data/`;
- хранилище можно тестировать отдельно от Git, VS Code и Codex CLI;
- интеграции можно заменить фиктивными адаптерами в тестах сценариев;
- проект имеет понятный контракт команд для разработки и проверки;
- структура не требует сервера, облачной базы данных или авторизации внутри приложения;
- структура позволяет позже выбрать desktop, web на `localhost` или CLI без переписывания доменной модели;
- все пользовательские тексты, сообщения ошибок и комментарии в создаваемом коде остаются на русском языке.

## Открытые вопросы

- Какой конкретный стек будет выбран для интерфейса первой версии?
- Какой менеджер пакетов и формат команд будет использовать проект?
- Нужен ли отдельный минимальный командный интерфейс для служебных операций и диагностики?
- Где после выбора стека будет располагаться точка входа локального приложения?
- Нужен ли общий пакет с типами, если проект будет разделен на несколько рабочих областей?

Эти вопросы не должны приводить к преждевременному добавлению сервера, облачного хранилища, авторизации или многопользовательского режима.
