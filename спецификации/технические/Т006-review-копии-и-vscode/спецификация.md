# Review-копии и открытие в VS Code

## Цель

Зафиксировать технические правила открытия кода выбранного обновления в VS Code без переключения основной локальной копии проекта с ветки `main`.

Эта спецификация уточняет требования `Ф003` и `Ф005`: преподаватель должен открыть код конкретного обновления на сохраненном коммите, а основная локальная копия в `data/repositories/` должна оставаться пригодной для будущих обновлений.

## Границы спецификации

`Т006` описывает:

- где размещаются review-копии;
- когда создается или переиспользуется review-копия;
- как открыть выбранный коммит в VS Code;
- почему основная копия `main` не переключается на исторические коммиты;
- как защищать основную копию от случайных изменений;
- как обрабатывать грязные review-копии;
- когда review-копии можно очищать, а когда нужно сохранять;
- какие ошибки нужно показывать и сохранять.

`Т006` не описывает:

- обновление основной копии проекта до `origin/main`;
- получение новых коммитов из GitHub;
- пользовательский дизайн страницы студента;
- формат всех JSON-схем приложения;
- запуск Codex CLI для ИИ-анализа.

Основные локальные репозитории и обновление `main` описывает `Т005`.

## Основной принцип

Открытие исторической версии проекта не должно менять основную локальную копию студенческого репозитория.

Основная копия:

```text
data/repositories/<student-id>/<project-id>/
```

остается технической копией для обновления ветки `main`.

Review-копия:

```text
data/review-copies/<student-id>/<update-id>/
```

используется для открытия конкретного коммита выбранного обновления в VS Code и для безопасного чтения кода в контексте проверки.

Приложение не должно:

- выполнять checkout исторического коммита в основной копии;
- переключать основную копию в detached `HEAD`;
- менять ветку основной копии;
- создавать коммиты из review-копии;
- делать `git push` из review-копии;
- переносить изменения из review-копии в основную копию;
- записывать комментарии преподавателя в файлы студенческого репозитория.

## Выбранная версия обновления

Версия обновления определяется `new commit`, сохраненным в событии обновления из `Ф002`.

Правила:

- если у обновления есть `new commit`, именно этот коммит открывается в VS Code;
- если обновление было первой загрузкой, открывается коммит первой загрузки;
- если обновление прошло без новых коммитов, открывается сохраненный последний коммит;
- если у обновления нет известного коммита, действие открытия недоступно;
- старый коммит используется для анализа диапазона, но открываемая папка должна соответствовать новому коммиту.

Открытие VS Code не создает новое событие обновления и не меняет статус проверки автоматически.

## Размещение review-копий

Все review-копии первой версии по умолчанию размещаются внутри:

```text
data/review-copies/
```

Минимальная структура:

```text
data/review-copies/
  <student-id>/
    <update-id>/
      .git/
      ...
```

Правила:

- `<student-id>` и `<update-id>` берутся из локальных данных приложения;
- путь review-копии сохраняется как служебное поле выбранного обновления или вычисляется из базовой папки;
- review-копии не должны создаваться вне настроенной базовой папки review-копий; по умолчанию это `data/review-copies/`;
- одна review-копия соответствует одному обновлению и одному целевому коммиту;
- если одно обновление открывается повторно, приложение может переиспользовать существующую review-копию после проверки ее состояния;
- review-копии не показываются отдельным пользовательским разделом первой версии.

Если в будущем понадобится несколько копий для одного обновления, это должно быть отдельным уточнением спецификации.

## Предпочтительный механизм: Git worktree

Предпочтительная реализация первой версии - Git worktree от основной локальной копии проекта.

Команда создания:

```bash
git worktree add --detach <review-path> <new-commit>
```

Команда проверки существующих worktree:

```bash
git worktree list --porcelain
```

Команда удаления безопасной review-копии:

```bash
git worktree remove <review-path>
```

Правила:

- worktree создается из основной локальной копии проекта;
- worktree создается в detached `HEAD` на сохраненном `new commit`;
- основная копия остается на `main`;
- создание worktree не должно выполнять `fetch`;
- если нужного коммита нет локально, приложение сначала должно сообщить ошибку доступности коммита или предложить обновить проект по правилам `Т005`;
- приложение не должно автоматически получать из сети новые данные только ради открытия старого обновления, если это может изменить состояние основной копии без явного обновления.

Worktree подходит для первой версии, потому что не требует полной повторной копии репозитория и не переключает основную рабочую директорию.

## Допустимая альтернатива: отдельная локальная копия

Если Git worktree недоступен или технически невозможен для конкретного репозитория, можно использовать отдельную локальную копию внутри настроенной базовой папки review-копий.

Минимальное правило для альтернативы:

- копия создается только локально из уже имеющейся основной копии или через безопасный Git-механизм;
- копия открывается на сохраненном `new commit`;
- копия не становится источником будущих обновлений;
- копия не делает push;
- копия не заменяет основную копию в `data/repositories/`.

Альтернатива не должна менять продуктовый смысл: преподаватель открывает код выбранного обновления, а основная копия `main` остается защищенной.

## Подготовка к открытию

Перед открытием VS Code приложение проверяет:

- у студента есть проект;
- у обновления есть `new commit`;
- основная локальная копия проекта существует;
- основная локальная копия является ожидаемым Git-репозиторием;
- нужный коммит доступен в локальной Git-базе;
- путь review-копии находится внутри настроенной базовой папки review-копий;
- существующая review-копия, если она есть, соответствует нужному коммиту;
- команда открытия VS Code доступна по настройкам `Т008`.

Если любая проверка не проходит, приложение не должно переключать основную копию и должно показать понятную ошибку.

## Создание review-копии

Если review-копия для выбранного обновления отсутствует, приложение создает ее.

Рекомендуемый порядок:

1. Прочитать событие обновления.
2. Проверить наличие `new commit`.
3. Проверить основную локальную копию проекта.
4. Проверить, что `new commit` существует локально.
5. Создать папку родителя внутри `data/review-copies/<student-id>/`.
6. Создать worktree или отдельную локальную копию в `data/review-copies/<student-id>/<update-id>/`.
7. Проверить, что `HEAD` review-копии равен `new commit`.
8. Сохранить или обновить служебную информацию о review-копии.
9. Открыть путь review-копии в VS Code.

Если создание не удалось, ошибка сохраняется в диагностике выбранного действия и показывается на странице обновления.

## Переиспользование review-копии

Если review-копия уже существует, приложение должно проверить ее перед повторным открытием.

Проверки:

```bash
git rev-parse HEAD
git status --porcelain
```

Правила:

- если `HEAD` равен сохраненному `new commit` и рабочая директория чистая, копию можно открыть повторно;
- если `HEAD` не равен сохраненному коммиту, копия считается несоответствующей;
- если копия содержит локальные изменения, она считается грязной;
- грязную копию можно принудительно восстановить до сохраненного `new commit`, потому что review-копия является воспроизводимой технической папкой для просмотра выбранного обновления;
- несоответствующую чистую копию можно пересоздать безопасно;
- несоответствующую грязную копию можно принудительно восстановить до сохраненного `new commit`;
- если принудительное восстановление не удалось, приложение должно сохранить копию и показать ошибку с указанием пути.

Приложение не должно переносить локальные изменения из review-копии в основную копию или репозиторий студента. При повторном открытии выбранного обновления приоритет имеет точное соответствие review-копии сохраненному коммиту обновления.

## Открытие в VS Code

В первой версии поддерживается только VS Code.

Команда или путь открытия определяется настройками `Т008`. Типовой вариант:

```bash
code <review-path>
```

Правила:

- открывается путь review-копии, а не путь основной копии;
- команда запускается после проверки или создания review-копии;
- если команда `code` недоступна, приложение показывает ошибку и путь, который можно открыть вручную;
- приложение не ждет закрытия VS Code как условия успешного открытия;
- успешный запуск команды открытия не означает, что преподаватель проверил обновление;
- статус проверки меняется только вручную по правилам `Ф003`.

Приложение не должно передавать в команду открытия непроверенные shell-строки. Путь должен быть отдельным аргументом дочернего процесса.

## Защита основной копии `main`

Основная локальная копия защищается организационно и технически.

Технические правила:

- все операции исторического открытия выполняются только с настроенной базовой папкой review-копий;
- перед созданием worktree приложение не делает checkout в основной копии;
- после создания worktree приложение может проверить, что основная копия все еще находится на `main`;
- если основная копия оказалась не на `main`, это ошибка состояния основной копии для `Т005`;
- ошибки review-копии не должны приводить к `reset`, `checkout` или удалению основной копии;
- очистка review-копий не должна удалять `data/repositories/`.

Если преподаватель вручную изменил основную копию вне приложения, следующая операция обновления должна обработать это как грязную или некорректную локальную копию по правилам `Т005`.

## Отношение к ИИ-анализу

`Ф004` допускает использование review-копии или безопасной папки анализа для запуска Codex CLI.

Правила `Т006` для таких копий:

- Codex CLI должен работать с review-копией или другой безопасной папкой, а не с основной копией для обновления `main`;
- если ИИ-анализ использует ту же review-копию, она должна соответствовать `new commit` выбранного обновления;
- ИИ не должен изменять файлы review-копии;
- ошибка ИИ-анализа не должна приводить к изменению основной копии проекта;
- технический запуск Codex CLI подробно описывается в `Т009`.

## Очистка review-копий

Review-копии являются служебными данными, но их нельзя удалять без проверки состояния.

Разрешено автоматически удалять или пересоздавать review-копию, если:

- она находится внутри настроенной базовой папки review-копий;
- она соответствует известному обновлению;
- она является worktree или ожидаемой локальной копией;
- она не используется текущим действием открытия или анализа.

Если review-копия является ожидаемым Git-репозиторием, но содержит локальные изменения, приложение может не удалять папку, а восстановить ее на сохраненный `new commit` через Git-команды внутри этой review-копии.

Нельзя автоматически удалять review-копию, если:

- она не распознана как review-копия Pilot Tracker;
- ее путь выходит за пределы настроенной базовой папки review-копий;
- Git не может прочитать ее состояние;
- есть сомнение, что удаление затронет основную копию.

Если копия нераспознанная или Git не может восстановить ее до нужного коммита, приложение должно сохранить ее и показать путь для ручного решения.

## Политика сохранения

Для первой версии безопаснее сохранять review-копии между запусками приложения, если они уже были созданы.

Правила:

- повторное открытие обновления может использовать существующую чистую копию;
- повторное открытие обновления может принудительно восстановить существующую грязную копию до сохраненного `new commit`;
- очистка может выполняться вручную или как безопасная техническая операция только для распознанных review-копий;
- приложение не обязано очищать review-копии автоматически при каждом запуске;
- удаление студента или истории обновления, если оно реализовано, должно предупреждать о связанных review-копиях;
- потеря review-копии не должна уничтожать историю проверки, потому что история хранится в `data/app/`.

Review-копия является воспроизводимой технической папкой. Поэтому при действии открытия кода приложение может отбросить локальные изменения внутри review-копии, чтобы преподаватель всегда видел именно сохраненную версию выбранного обновления.

## Ошибки review-копий и VS Code

Минимальные категории ошибок:

- `update_commit_missing` - у обновления нет известного нового коммита;
- `main_repository_missing` - основная локальная копия отсутствует;
- `main_repository_invalid` - основная локальная копия не является ожидаемым Git-репозиторием;
- `commit_not_available` - сохраненный коммит не найден локально;
- `review_path_invalid` - путь review-копии выходит за пределы настроенной базовой папки review-копий;
- `review_copy_restore_failed` - Git не смог восстановить существующую review-копию до нужного коммита;
- `review_copy_mismatch` - существующая review-копия указывает на другой коммит;
- `worktree_create_failed` - Git не смог создать worktree;
- `worktree_remove_failed` - Git не смог удалить чистый worktree;
- `vscode_not_available` - команда открытия VS Code не найдена;
- `vscode_open_failed` - команда открытия VS Code завершилась ошибкой;
- `git_command_failed` - Git-команда завершилась ошибкой без более точной категории.

Для ошибки нужно сохранять:

- категорию;
- человекочитаемое сообщение;
- техническое сообщение;
- путь основной копии, если он применим;
- путь review-копии, если он применим;
- целевой коммит;
- команду, которая завершилась ошибкой;
- код завершения, если он есть;
- дату и время.

Сообщения об ошибках не должны содержать секреты из окружения или вывода Git.

## Пользовательское поведение при ошибках

Если открыть код нельзя, страница выбранного обновления должна объяснить причину.

Минимальные реакции:

- нет `new commit` - действие открытия недоступно;
- основная копия отсутствует - предложить обновить проект по правилам `Т005`;
- коммит не найден локально - показать, что локальная копия не содержит нужную версию;
- восстановление review-копии не удалось - не менять основную копию и показать путь;
- VS Code недоступен - показать путь review-копии, если она создана, и диагностическое сообщение;
- создание worktree завершилось ошибкой - не менять основную копию и сохранить ошибку.

Ошибка открытия VS Code не должна удалять review-копию и не должна менять статус проверки.

## Безопасность запуска команд

Git и VS Code запускаются как дочерние процессы без shell-интерполяции пользовательских данных.

Правила:

- команда и аргументы передаются отдельными значениями;
- путь review-копии проверяется перед передачей в команду;
- приложение не выполняет shell-команды, собранные из имени студента, URL или пути;
- stdout и stderr сохраняются только после очистки от секретов;
- команда открытия VS Code не должна получать дополнительные аргументы из данных студента или репозитория.

## Что не входит в первую версию

- выбор редактора кроме VS Code;
- встроенный просмотр кода в интерфейсе Pilot Tracker;
- отображение всех review-копий как отдельного раздела;
- автоматическая синхронизация изменений из review-копии;
- создание коммитов из review-копии;
- push в репозитории студентов;
- публикация комментариев в GitHub;
- переключение основной копии на исторические коммиты;
- поддержка нескольких review-копий одного обновления как пользовательской функции.

## Критерии приемки

Техническая реализация `Т006` считается готовой, если:

- код выбранного обновления открывается из настроенной базовой папки review-копий, а не из основной копии;
- review-копия соответствует `new commit` выбранного обновления;
- основная копия остается на ветке `main`;
- приложение не делает checkout исторического коммита в основной копии;
- повторное открытие использует существующую чистую review-копию, принудительно восстанавливает грязную review-копию или безопасно пересоздает несоответствующую чистую копию;
- грязная review-копия после успешного открытия снова соответствует `new commit` выбранного обновления;
- отсутствие `new commit` делает действие открытия недоступным;
- отсутствие нужного коммита сохраняется как ошибка;
- недоступность VS Code сохраняется как понятная ошибка;
- приложение не выполняет `git push`, `git commit`, `git merge` или `git rebase` из review-копии;
- очистка и восстановление затрагивают только распознанные review-копии внутри настроенной базовой папки review-копий;
- ошибка review-копии не повреждает и не удаляет основную копию проекта.

## Открытые вопросы

На текущий момент открытых вопросов по `Т006` нет.

## Решения первой версии

- Исторические версии по умолчанию открываются через review-копии в `data/review-copies/`.
- Предпочтительный механизм review-копии - Git worktree в detached `HEAD`.
- Открываемый коммит всегда берется из `new commit` выбранного обновления.
- Основная локальная копия в `data/repositories/` остается копией для обновления `main`.
- Приложение открывает в VS Code только review-копию.
- В первой версии поддерживается только VS Code.
- Грязные review-копии при открытии кода принудительно восстанавливаются до `new commit` выбранного обновления.
- Review-копии можно безопасно очищать или восстанавливать только после проверки, что они распознаны и находятся внутри настроенной базовой папки review-копий.
- Приложение не создает коммиты и не делает push из review-копий.
