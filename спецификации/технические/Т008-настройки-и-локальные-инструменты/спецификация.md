# Настройки и локальные инструменты

## Цель

Зафиксировать технические требования к настройкам первой версии Pilot Tracker: путям локальных артефактов, командам локальных инструментов, проверке доступности Git, GitHub CLI, Codex CLI и VS Code, состоянию подключения к Supabase, а также диагностике проблем окружения.

Эта спецификация уточняет интерфейс и поведение настроек из `Ф005`, не добавляя публичную регистрацию, управление пользователями, несколько ролей, LMS, студенческий интерфейс, несколько курсов или экспорт данных.

## Изменение после согласования `Т011`

После согласования `Т011` экран настроек должен также показывать состояние подключения к онлайн-проекту Supabase. Supabase-настройки относятся только к подключению и диагностике базы данных; локальные пути, Git, GitHub CLI, Codex CLI и VS Code остаются настройками конкретного компьютера. Локальный Supabase через Docker не диагностируется как обязательная часть окружения.

Секреты подключения к Supabase не должны попадать в клиентские компоненты, журналы диагностики или Git.

## Изменение после согласования `Т012`

После согласования `Т012` экран настроек может показывать состояние администраторской сессии и подключения к Supabase без пароля, cookie, JWT и секретных ключей. Настройки не становятся экраном регистрации, смены пароля или управления пользователями.

## Роль настроек в первой версии

Настройки нужны, чтобы преподаватель мог понять и исправить локальное окружение, от которого зависят:

- подключение к онлайн-проекту Supabase для прикладных данных;
- размещение основных локальных репозиториев;
- размещение review-копий;
- локальные настройки и миграционные архивы;
- запуск Git-команд;
- получение контекста pull request через GitHub CLI, если он доступен;
- запуск локального Codex CLI;
- открытие выбранного обновления в VS Code.

Настройки не должны превращаться в отдельный административный раздел. В первой версии это рабочий экран для одного преподавателя и одной активной когорты.

## Экран настроек

Экран настроек должен быть доступен из основной навигации, описанной в `Т007`.

Минимальные блоки экрана:

- пути данных;
- локальные инструменты;
- настраиваемые команды;
- диагностика окружения;
- сведения о последней проверке.

Экран должен показывать текущее состояние настроек, позволять изменить допустимые значения и запускать повторную диагностику. Если настройки еще не сохранялись, должны использоваться значения по умолчанию из `Ф005`.

## Пути данных

По умолчанию все данные Pilot Tracker находятся внутри папки `data/` в папке приложения.

Минимальные пути:

- базовая папка данных приложения: `data/`;
- папка локальных настроек и миграционных архивов: `data/app/`;
- базовая папка локальных студенческих репозиториев: `data/repositories/`;
- базовая папка review-копий: `data/review-copies/`;
- папка будущих локальных резервных копий: `data/backups/`.

Настройки должны хранить эти пути как служебные значения. Интерфейс может показывать их преподавателю и давать возможность изменить базовые папки, если это нужно для переноса данных или исправления локального окружения.

Если путь изменен, приложение должно:

- проверить, что путь не пустой;
- показать итоговый абсолютный путь;
- проверить возможность чтения;
- проверить возможность записи для папок, куда приложение должно сохранять данные;
- не переносить данные автоматически без отдельного явного сценария;
- предупредить, если изменение пути может привести к тому, что ранее сохраненные локальные репозитории или review-копии не будут найдены.

Экспорт данных в отдельный формат не входит в первую версию. Настройки путей нужны только для локальной работы, переносимости и резервного копирования папки данных.

## Локальные настройки и старые JSON-данные

После согласования `Т011` основное хранилище прикладных данных - онлайн-проект Supabase. `data/app/` используется для локальных настроек, диагностических данных и миграционного архива старых JSON-файлов.

Диагностика путей должна проверять:

- существует ли базовая папка данных;
- существует ли папка `data/app/`;
- можно ли создать отсутствующие папки, если это разрешено текущим сценарием;
- можно ли читать локальный файл настроек и старые JSON-файлы при миграции;
- можно ли записывать настройки в `settings.json`;
- не указывает ли путь данных на файл вместо папки.

Если старые JSON-файлы отсутствуют при первом запуске после перехода на Supabase, это не считается ошибкой данных. Приложение должно использовать Supabase как источник прикладных данных и создать только локальные настройки, если они отсутствуют и текущее окружение позволяет запись на диск.

Если приложение запущено в hosted-окружении без доступной для записи локальной папки проекта, например в serverless runtime, отсутствие `data/app/settings.json` не должно блокировать чтение прикладных данных из Supabase. В таком окружении приложение использует начальные локальные настройки в памяти, а действия, зависящие от локальных папок, Git, VS Code, GitHub CLI или Codex CLI, остаются ограниченными.

Если локальный `settings.json` поврежден или не читается, это ошибка локального хранения. Если поврежден старый JSON-файл миграции, экран настроек или сценарий миграции должен показать понятное сообщение и технические детали для диагностики, но не должен перезаписывать файл пустой структурой.

## Локальные инструменты

Первая версия зависит от локально установленных инструментов:

- Git для клонирования, обновления и работы с коммитами;
- GitHub CLI для получения контекста pull request, если он используется и доступен;
- Codex CLI для локального ИИ-анализа;
- VS Code для открытия кода выбранного обновления.

Приложение должно уметь определить доступность каждого инструмента и сохранить результат последней проверки в настройках или диагностическом состоянии.

Недоступность одного инструмента не должна полностью блокировать приложение. Она должна блокировать только те действия, которым этот инструмент нужен.

## Git

Git обязателен для обновления проектов, создания локальных копий и подготовки review-копий.

Настройки Git должны включать:

- команду или путь к Git;
- признак доступности;
- версию Git, если ее удалось определить;
- текст ошибки последней проверки, если Git недоступен;
- время последней проверки.

Базовая проверка:

```bash
git --version
```

Если Git недоступен:

- список студентов и сохраненная история должны открываться, если данные читаются;
- обновление проектов должно быть недоступно;
- открытие исторической версии через review-копию должно быть недоступно;
- ИИ-анализ выбранного обновления должен быть недоступен, если для него нужны Git-команды или review-копия;
- интерфейс должен показать, что нужно исправить команду Git или установить Git.

Приложение не должно выполнять `commit` или `push` через настройки и не должно менять Git-аутентификацию преподавателя.

## GitHub CLI

GitHub CLI нужен только для контекста pull request в ИИ-анализе, как описано в `Ф004`.

Настройки GitHub CLI должны включать:

- команду или путь к GitHub CLI;
- признак доступности;
- версию, если ее удалось определить;
- признак локальной аутентификации, если его удалось проверить без хранения секретов;
- текст ошибки последней проверки;
- время последней проверки.

Базовые проверки:

```bash
gh --version
gh auth status
```

Если GitHub CLI недоступен:

- обновление проектов через Git не должно блокироваться только из-за этого;
- ИИ-анализ может выполняться по локальному Git-контексту выбранного режима без контекста pull request;
- ИИ-рапорт должен содержать диагностическую отметку, что контекст pull request недоступен, если анализ пытался его получить;
- интерфейс настроек должен показать, что контекст pull request ограничен.

Приложение не должно хранить GitHub-пароль или собственный GitHub-токен. Аутентификация остается локальной настройкой преподавателя вне Pilot Tracker.

## Codex CLI

Codex CLI обязателен для ИИ-анализа первой версии.

Настройки Codex CLI должны включать:

- команду или путь к Codex CLI;
- признак доступности;
- версию, если ее удалось определить;
- признак готовности к запуску, если его можно проверить безопасной командой;
- текст ошибки последней проверки;
- время последней проверки.

Базовая проверка:

```bash
codex --version
```

Проверка не должна запускать анализ проекта и не должна отправлять данные студенческого репозитория. Она нужна только для понимания, доступна ли команда.

Если Codex CLI недоступен:

- список студентов, обновление проектов, комментарии и статусы проверки должны продолжать работать;
- действие ИИ-анализа в деталях обновления должно быть недоступно;
- интерфейс должен объяснить, что требуется локальный Codex CLI;
- новое событие ИИ-рапорта не создается, если запуск был заблокирован заранее известной недоступностью Codex CLI;
- ошибка не должна удалять старые ИИ-рапорты.

Приложение не вызывает OpenAI API напрямую, не хранит API-ключи OpenAI и не поддерживает альтернативные ИИ-провайдеры.

## VS Code

VS Code нужен для открытия проекта на версии выбранного обновления.

Настройки VS Code должны включать:

- команду или путь для открытия VS Code;
- признак доступности;
- версию, если ее удалось определить;
- текст ошибки последней проверки;
- время последней проверки.

Базовая проверка:

```bash
code --version
```

Если команда `code` недоступна, интерфейс должен позволить указать путь или команду открытия VS Code, если техническая платформа приложения это поддерживает.

Если VS Code недоступен:

- список студентов и история проверки должны работать;
- обновление проектов должно работать, если Git доступен;
- ИИ-анализ должен работать, если Git и Codex CLI доступны;
- действие открытия кода в VS Code должно быть недоступно или завершаться понятной ошибкой;
- ошибка должна показывать путь, который приложение пыталось открыть.

В первой версии не нужен выбор другого редактора.

## Настраиваемые команды

Для каждого локального инструмента настройки должны хранить команду или путь:

- `git`;
- `gh`;
- `codex`;
- `code`.

Правила:

- значение по умолчанию равно имени команды;
- значение можно изменить на абсолютный путь или другую команду запуска;
- пустое значение недопустимо для обязательных инструментов;
- изменение команды должно требовать повторной диагностики;
- команда должна использоваться только через безопасный запуск процесса без конкатенации пользовательского ввода в shell-строку;
- аргументы рабочих операций должны формироваться приложением отдельно от сохраненной команды.

Настройки команд не должны давать преподавателю произвольно редактировать Git-команды обновления проектов, команды `codex exec` или внутренние команды review-копий. В первой версии настраивается путь к инструменту, а не сценарий работы продукта.

## Диагностика окружения

Экран настроек должен давать действие `Проверить окружение`.

Диагностика должна проверять:

- доступность папки данных;
- доступность папки локальных настроек и миграционных архивов;
- возможность чтения и записи настроек;
- доступность папки локальных репозиториев;
- доступность папки review-копий;
- доступность Git;
- доступность GitHub CLI;
- доступность Codex CLI;
- доступность VS Code;
- состояние локальной GitHub-аутентификации, если GitHub CLI доступен.

Для каждой проверки нужно сохранять:

- статус: не проверялось, выполняется, успешно, предупреждение, ошибка;
- краткое пользовательское сообщение;
- технические детали;
- версию инструмента, если ее удалось определить;
- состояние локальной GitHub-аутентификации для GitHub CLI, если его удалось проверить;
- время проверки.

В локальных настройках эти статусы сохраняются техническими кодами:

- `unknown` - не проверялось;
- `running` - выполняется;
- `success` - успешно;
- `warning` - предупреждение;
- `error` - ошибка.

Диагностика должна быть повторяемой и не должна менять данные студентов, историю обновлений, комментарии, ИИ-рапорты или локальные репозитории, кроме создания отсутствующих служебных папок в разрешенном сценарии.

## Уровни результата диагностики

Статусы диагностики:

- `не проверялось` - проверка еще не запускалась;
- `выполняется` - проверка сейчас идет;
- `успешно` - инструмент или путь доступен;
- `предупреждение` - базовая работа возможна, но часть функций ограничена;
- `ошибка` - связанные действия должны быть недоступны до исправления.

Примеры предупреждений:

- GitHub CLI недоступен, но Git работает;
- GitHub CLI доступен, но пользователь не авторизован;
- Codex CLI недоступен, но базовая проверка проектов без ИИ возможна;
- VS Code недоступен, но история проверки и ИИ-рапорты доступны.

Примеры ошибок:

- папка данных недоступна для записи;
- Git недоступен при попытке обновлять проекты;
- сохраненный путь локального репозитория указывает на отсутствующую папку;
- команда инструмента пустая или не запускается.

## Хранение настроек

Настройки должны сохраняться локально, как описано в `Ф005`, в файле `data/app/settings.json` или совместимой структуре, определенной технической спецификацией данных. Прикладные данные студентов и истории проверки после `Т011` сохраняются в Supabase, а не в `settings.json`.

Минимальные данные настроек:

- базовая папка данных;
- папка локальных настроек и миграционных архивов;
- папка локальных репозиториев;
- папка review-копий;
- команда Git;
- команда GitHub CLI;
- команда Codex CLI;
- команда VS Code;
- результат последней диагностики по каждому пути и инструменту;
- версия инструмента, если она известна;
- пользовательское сообщение последней диагностики;
- технические детали последней диагностики без секретов;
- состояние локальной GitHub-аутентификации для GitHub CLI;
- дата последнего сохранения настроек.

Настройки не должны хранить:

- GitHub-пароль;
- GitHub-токен;
- OpenAI API-ключ;
- секреты Codex CLI;
- данные авторизации пользователя;
- настройки нескольких курсов или пользователей.

## Связь настроек с действиями приложения

Действия приложения должны учитывать диагностику:

- обновление всех проектов требует доступного Git и доступной папки локальных репозиториев;
- одиночное обновление проекта требует доступного Git и доступной папки локальных репозиториев;
- открытие VS Code требует доступного Git для review-копии, доступной папки review-копий и доступной команды VS Code;
- ИИ-анализ требует доступного Codex CLI, локального пути проекта, сохраненного нового коммита и рабочей папки анализа;
- получение контекста pull request требует доступного GitHub CLI и локальной GitHub-аутентификации, но его отсутствие не блокирует основной локальный анализ;
- сохранение комментария и статуса требует доступного Supabase-хранилища и действующей администраторской сессии.

Если требуемая настройка отсутствует или проверка завершилась ошибкой, соответствующее действие должно быть отключено или завершаться понятной ошибкой без потери данных.

## Ошибки настроек

Ошибки настроек должны показываться на экране настроек и в месте действия, которое не может быть выполнено.

Минимальные сообщения должны объяснять:

- какой инструмент или путь недоступен;
- какое действие из-за этого ограничено;
- какое значение сейчас используется;
- какие технические детали вернула проверка.

Технические детали нужны для диагностики, но пользовательское сообщение должно оставаться коротким и понятным.

Ошибки диагностики не должны автоматически удалять или менять сохраненные пути проектов, историю обновлений, комментарии, статусы и ИИ-рапорты.

## Ограничения первой версии

Настройки первой версии не должны добавлять:

- публичную регистрацию;
- смену пароля администратора;
- управление пользователями;
- несколько ролей;
- хранение секретов GitHub или OpenAI;
- управление несколькими курсами, семестрами или архивами;
- подключение LMS;
- студенческий интерфейс;
- экспорт данных;
- офлайн-синхронизацию с конфликтами между устройствами;
- настройку альтернативных ИИ-провайдеров;
- выбор редактора кроме VS Code;
- произвольное редактирование внутренних Git-команд обновления.

## Критерии приемки

Техническая реализация настроек соответствует этой спецификации, если:

- экран настроек доступен из основной навигации;
- приложение показывает текущие пути `data/`, `data/app/`, `data/repositories/` и `data/review-copies/`;
- приложение показывает команды Git, GitHub CLI, Codex CLI и VS Code;
- преподаватель может изменить путь к инструменту или команду запуска;
- преподаватель может запустить повторную диагностику;
- диагностика показывает состояние путей данных;
- диагностика показывает доступность Git;
