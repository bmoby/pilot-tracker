# Модель данных и JSON-схемы

## Цель

Зафиксировать модель данных первой локальной версии Pilot Tracker: сущности, связи, идентификаторы, временные метки, статусы и контракты JSON-файлов в `data/app/`.

Эта спецификация переводит требования `Ф001`-`Ф005` в технический контракт хранения. Она не описывает операции чтения, записи, блокировок и восстановления после ошибок. Эти правила описывает `Т004`.

## Границы первой версии

В первой версии модель данных описывает один активный курс или одну активную рабочую группу студентов.

В первую версию входят только локальные JSON-файлы:

- `students.json`;
- `projects.json`;
- `update-runs.json`;
- `update-events.json`;
- `comments.json`;
- `review-statuses.json`;
- `ai-reports.json`;
- `settings.json`.

В первой версии не используются SQLite, внешняя база данных, сервер, облачная синхронизация и экспорт истории курса.

## Общие правила данных

Все JSON-файлы хранятся в `data/app/` и имеют верхнеуровневое поле `schemaVersion`.

Минимальная версия схемы первой версии:

```json
{
  "schemaVersion": 1
}
```

Правила:

- каждый файл содержит данные только одной коллекции или одного служебного объекта;
- имена JSON-полей стабильны и записываются в `camelCase`;
- пользовательские тексты хранятся как строки UTF-8;
- отсутствие необязательного значения хранится как `null`, а не как пустая строка;
- пустая строка не считается валидным значением для обязательных текстовых полей;
- старые события обновлений, комментарии и ИИ-рапорты не перезаписываются новыми событиями;
- связи между сущностями хранятся через идентификаторы, а не через вложенные копии объектов;
- денормализованные снимки допускаются только там, где нужно сохранить состояние на момент события.

## Идентификаторы

Все идентификаторы являются строками, которые создает приложение.

Формат первой версии:

```text
<префикс>_<uuid-v4>
```

Префиксы:

- `student` для студентов;
- `project` для проектов;
- `run` для запусков обновления;
- `update` для событий обновления;
- `comment` для комментариев;
- `review_status` для статусов проверки;
- `ai_report` для ИИ-рапортов.

Пример:

```text
student_018f4b3d-3b78-4a2e-bf0b-5c6c1d4b7e21
```

Правила:

- идентификатор неизменен после создания записи;
- идентификатор не должен зависеть от имени студента, URL репозитория или локального пути;
- приложение не должно переиспользовать идентификатор удаленной записи;
- человекочитаемый порядок записей определяется датами, а не идентификаторами.

## Временные метки

Все временные метки хранятся в формате ISO 8601 в UTC.

Пример:

```text
2026-05-07T10:15:30.000Z
```

Правила:

- `createdAt` фиксирует момент создания записи;
- `updatedAt` фиксирует последнее изменение изменяемой записи;
- исторические события имеют отдельные поля начала и завершения операции;
- временные метки не должны храниться в локальном часовом поясе;
- сортировка ленты обновлений выполняется по `occurredAt` или `startedAt`, если `occurredAt` отсутствует.

## Связи сущностей

Основные связи первой версии:

- один студент имеет один основной проект;
- проект всегда принадлежит одному студенту;
- запуск обновления группирует одно или несколько событий обновления;
- событие обновления принадлежит одному студенту и одному проекту;
- комментарий принадлежит одному событию обновления;
- статус проверки принадлежит одному событию обновления;
- ИИ-рапорт принадлежит одному событию обновления;
- настройки относятся ко всему локальному приложению, а не к отдельному студенту.

Схема связей:

```text
student 1 -> 1 project
update-run 1 -> many update-events
update-event 1 -> many comments
update-event 1 -> 1 review-status
update-event 1 -> many ai-reports
```

## `students.json`

Файл хранит текущую рабочую когорту преподавателя.

Форма файла:

```json
{
  "schemaVersion": 1,
  "students": []
}
```

Схема записи студента:

```json
{
  "id": "student_<uuid-v4>",
  "displayName": "Имя студента",
  "notes": null,
  "projectId": "project_<uuid-v4>",
  "createdAt": "2026-05-07T10:15:30.000Z",
  "updatedAt": "2026-05-07T10:15:30.000Z"
}
```

Поля:

- `id` - обязательный стабильный идентификатор студента;
- `displayName` - обязательное имя студента;
- `notes` - дополнительная информация преподавателя в свободной форме;
- `projectId` - идентификатор основного проекта студента;
- `createdAt` - дата создания записи;
- `updatedAt` - дата последнего изменения карточки студента.

Правила:

- `displayName` обязателен и не может быть пустой строкой;
- `notes` может быть `null`;
- студент первой версии должен иметь ровно один связанный проект;
- изменение карточки студента не удаляет историю обновлений, комментарии, статусы или ИИ-рапорты;
- группы, теги, email, аватары и несколько проектов на студента в первой версии не входят.

## `projects.json`

Файл хранит проекты студентов и текущий технический снимок проекта для списка студентов.

Форма файла:

```json
{
  "schemaVersion": 1,
  "projects": []
}
```

Схема проекта:

```json
{
  "id": "project_<uuid-v4>",
  "studentId": "student_<uuid-v4>",
  "repositoryUrl": null,
  "localPath": null,
  "defaultBranch": "main",
  "currentBranch": null,
  "lastKnownCommit": null,
  "lastUpdatedAt": null,
  "lastUpdateEventId": null,
  "status": "not_connected",
  "lastError": null,
  "aiDescription": {
    "status": "missing",
    "summary": null,
    "idea": null,
    "keyParts": [],
    "sourceAiReportId": null,
    "sourceCommit": null,
    "updatedAt": null,
    "error": null
  },
  "createdAt": "2026-05-07T10:15:30.000Z",
  "updatedAt": "2026-05-07T10:15:30.000Z"
}
```

Поля:

- `id` - обязательный идентификатор проекта;
- `studentId` - обязательная связь со студентом;
- `repositoryUrl` - ссылка на GitHub-репозиторий или `null`;
- `localPath` - относительный путь основной локальной копии от корня `data/`, например `repositories/<student-id>/<project-id>`;
- `defaultBranch` - основная ветка первой версии, по умолчанию `main`;
- `currentBranch` - ветка, которую приложение последний раз успешно определило локально;
- `lastKnownCommit` - последний известный коммит ветки `main`;
- `lastUpdatedAt` - дата последнего события обновления проекта;
- `lastUpdateEventId` - ссылка на последнее событие обновления;
- `status` - текущий статус проекта для списка студентов;
- `lastError` - последний текст ошибки или `null`;
- `aiDescription` - краткое описание проекта, созданное или обновленное ИИ;
- `createdAt` - дата создания проекта;
- `updatedAt` - дата последнего изменения записи проекта.

Статусы проекта:

- `not_connected` - ссылка GitHub не указана;
- `never_updated` - ссылка указана, но обновление еще не выполнялось;
- `updating` - обновление сейчас выполняется;
- `first_loaded` - проект впервые успешно загружен и требует первой проверки;
- `no_changes` - последнее успешное обновление не нашло новых коммитов;
- `has_changes` - последнее успешное обновление нашло новые коммиты;
- `update_error` - последнее обновление завершилось ошибкой;
- `local_copy_missing` - сохраненный локальный путь отсутствует;
- `local_copy_unavailable` - локальная копия повреждена или недоступна;
- `repository_unavailable` - ссылка на репозиторий недоступна или требует исправления.

Статусы краткого описания проекта:

- `missing` - описание отсутствует;
- `running` - описание создается во время ИИ-анализа;
- `ready` - описание готово;
- `error` - описание не удалось создать или обновить.

Правила:

- `repositoryUrl` не обязателен;
- если `repositoryUrl` указан, он должен выглядеть как ссылка на GitHub-репозиторий;
- `localPath` хранится как относительный путь внутри `data/`, а не как абсолютный путь;
- основная ветка первой версии - `main`;
- краткое описание проекта хранится в проекте, потому что в первой версии отдельная сущность описания не нужна;
- приложение не должно изменять файлы студенческого репозитория ради обновления этой записи.

## `update-runs.json`

Файл хранит запуски обновления проектов.

Форма файла:

```json
{
  "schemaVersion": 1,
  "updateRuns": []
}
```

Схема запуска обновления:

```json
{
  "id": "run_<uuid-v4>",
  "scope": "all_projects",
  "studentId": null,
  "projectId": null,
  "status": "running",
  "startedAt": "2026-05-07T10:15:30.000Z",
  "finishedAt": null,
  "summary": {
    "studentsTotal": 0,
    "projectsAttempted": 0,
    "projectsFirstLoaded": 0,
    "projectsWithChanges": 0,
    "newCommitsTotal": 0,
    "projectsWithoutChanges": 0,
    "errorsTotal": 0,
    "studentsWithoutRepository": 0
  },
  "error": null
}
```

Поля:

- `id` - идентификатор запуска;
- `scope` - область запуска;
- `studentId` - заполнен только для одиночного обновления проекта;
- `projectId` - заполнен только для одиночного обновления проекта;
- `status` - статус запуска;
- `startedAt` - дата и время начала;
- `finishedAt` - дата и время завершения или `null`;
- `summary` - итоговая сводка;
- `error` - ошибка запуска верхнего уровня или `null`.

Поля `summary`:

- `studentsTotal` - количество студентов в списке на момент запуска;
- `projectsAttempted` - количество проектов, для которых запускались Git-операции;
- `projectsFirstLoaded` - количество проектов, впервые успешно загруженных локально;
- `projectsWithChanges` - количество повторно обновленных проектов с новыми коммитами;
- `newCommitsTotal` - сумма новых коммитов по повторным обновлениям, где число коммитов вычислено;
- `projectsWithoutChanges` - количество повторных успешных обновлений без новых коммитов;
- `errorsTotal` - количество проектов с ошибкой обновления;
- `studentsWithoutRepository` - количество студентов без GitHub-ссылки.

Значения `scope`:

- `all_projects` - общий запуск обновления всех проектов;
- `single_project` - одиночный запуск обновления одного проекта со страницы студента.

Статусы запуска:

- `running` - запуск выполняется;
- `completed` - запуск завершился без ошибок проектов;
- `completed_with_errors` - запуск завершился, но у части проектов есть ошибки;
- `failed` - запуск не смог выполниться как операция верхнего уровня.

Правила:

- один запуск должен иметь одно или несколько связанных событий обновления, если загрузка состояния студентов была успешной;
- ошибка одного проекта не переводит весь запуск в `failed`;
- `failed` используется для ошибок уровня запуска, например невозможности прочитать список студентов или записать историю;
- итоговая сводка считается по событиям обновления этого запуска.

## `update-events.json`

Файл хранит историю обновлений конкретных проектов.

Форма файла:

```json
{
  "schemaVersion": 1,
  "updateEvents": []
}
```

Схема события обновления:

```json
{
  "id": "update_<uuid-v4>",
  "runId": "run_<uuid-v4>",
  "studentId": "student_<uuid-v4>",
  "projectId": "project_<uuid-v4>",
  "status": "completed",
  "startedAt": "2026-05-07T10:15:30.000Z",
  "finishedAt": "2026-05-07T10:15:31.000Z",
  "occurredAt": "2026-05-07T10:15:30.000Z",
  "repositoryUrlSnapshot": null,
  "projectLocalPathSnapshot": null,
  "branch": "main",
  "previousCommit": null,
  "newCommit": null,
  "newCommitsCount": null,
  "hasNewChanges": false,
  "result": "skipped_no_repository",
  "error": null,
  "analysisBoundaryRecordedAt": null,
  "createdAt": "2026-05-07T10:15:30.000Z",
  "updatedAt": "2026-05-07T10:15:31.000Z"
}
```

Поля:

- `id` - идентификатор события обновления;
- `runId` - запуск, внутри которого создано событие;
- `studentId` - студент на момент обновления;
- `projectId` - проект на момент обновления;
- `status` - технический статус жизненного цикла события обновления;
- `startedAt` - дата и время начала обработки проекта;
- `finishedAt` - дата и время завершения обработки проекта или `null`;
- `occurredAt` - дата и время фиксации результата события или `null`, если результат еще не зафиксирован;
- `repositoryUrlSnapshot` - ссылка GitHub на момент обновления или `null`;
- `projectLocalPathSnapshot` - относительный локальный путь на момент обновления или `null`;
- `branch` - ветка, которая обновлялась;
- `previousCommit` - старый известный коммит или `null`;
- `newCommit` - новый коммит или `null`;
- `newCommitsCount` - количество новых коммитов или `null`, если число нельзя корректно вычислить;
- `hasNewChanges` - были ли найдены новые изменения;
- `result` - результат обновления или `null`, если событие еще не завершено;
- `error` - текст ошибки или причина недоступности;
- `analysisBoundaryRecordedAt` - дата фиксации анализируемого состояния или границы анализа для ИИ;
- `createdAt` - дата записи события в историю;
- `updatedAt` - дата последнего изменения технического состояния события.

Статусы события обновления:

- `running` - обработка проекта началась и еще не завершилась;
- `completed` - событие завершилось штатным результатом;
- `failed` - событие завершилось ошибкой проекта;
- `interrupted` - приложение было остановлено до фиксации результата события.

Результаты обновления:

- `skipped_no_repository` - у студента нет ссылки GitHub;
- `cloned` - проект впервые успешно склонирован;
- `updated_with_changes` - проект обновлен и найдены новые коммиты;
- `updated_no_changes` - проект обновлен, новых коммитов нет;
- `error` - обновление проекта завершилось ошибкой.

Правила:

- каждое обновление проекта создает событие истории;
- событие создается и для студента без GitHub-ссылки;
- успешное обновление без новых изменений тоже сохраняется;
- `previousCommit` может быть `null` при первой загрузке или недоступном проекте;
- `newCommitsCount` равен `null` при первой загрузке, недоступном проекте, ошибке или прерванном событии;
- для `updated_no_changes` значение `newCommitsCount` равно `0`;
- для `updated_with_changes` значение `newCommitsCount` больше `0`;
- `newCommit` обязателен для успешных результатов `cloned`, `updated_with_changes` и `updated_no_changes`;
- `result` может быть `null` только в статусах `running` и `interrupted`;
- `hasNewChanges` равен `true` только для `updated_with_changes`;
- событие `cloned` не считается повторным изменением, но отображается отдельно как первая успешная загрузка, требующая первой проверки;
- `analysisBoundaryRecordedAt` заполняется, если событие имеет достаточное состояние для ИИ-анализа;
- при запуске приложения события, оставшиеся в статусе `running`, переводятся в `interrupted` с понятной ошибкой прерывания;
- завершенное событие обновления является историческим фактом и не редактируется после записи результата, кроме технического исправления ошибки записи по явному решению разработчика.

## `comments.json`

Файл хранит ручные комментарии преподавателя.

Форма файла:

```json
{
  "schemaVersion": 1,
  "comments": []
}
```

Схема комментария:

```json
{
  "id": "comment_<uuid-v4>",
  "studentId": "student_<uuid-v4>",
  "projectId": "project_<uuid-v4>",
  "updateEventId": "update_<uuid-v4>",
  "text": "Текст комментария преподавателя",
  "basedOnAiReportId": null,
  "createdAt": "2026-05-07T10:15:30.000Z",
  "updatedAt": "2026-05-07T10:15:30.000Z"
}
```

Поля:

- `id` - идентификатор комментария;
- `studentId` - студент, к которому относится комментарий;
- `projectId` - проект, к которому относится комментарий;
- `updateEventId` - обновление, к которому привязан комментарий;
- `text` - текст комментария преподавателя;
- `basedOnAiReportId` - ИИ-рапорт, если комментарий создан на основе черновика;
- `createdAt` - дата создания;
- `updatedAt` - дата последнего редактирования.

Правила:

- комментарий всегда привязан к событию обновления целиком;
- комментарии к файлам, строкам, коммитам и фрагментам diff в первой версии не хранятся;
- `text` обязателен и не может быть пустой строкой;
- удаление комментария удаляет только соответствующую запись из массива `comments`;
- удаление комментария не удаляет событие обновления, статус проверки, ИИ-рапорты или другие комментарии;
- идентификатор удаленного комментария не должен переиспользоваться;
- историю редакций комментариев первая версия не хранит;
- историю удаления комментариев первая версия не хранит;
- комментарии не отправляются студентам и не публикуются в GitHub автоматически.

## `review-statuses.json`

Файл хранит текущий ручной статус проверки каждого обновления.

Форма файла:

```json
{
  "schemaVersion": 1,
  "reviewStatuses": []
}
```

Схема статуса проверки:

```json
{
  "id": "review_status_<uuid-v4>",
  "studentId": "student_<uuid-v4>",
  "projectId": "project_<uuid-v4>",
  "updateEventId": "update_<uuid-v4>",
  "status": "not_reviewed",
  "createdAt": "2026-05-07T10:15:30.000Z",
  "updatedAt": "2026-05-07T10:15:30.000Z"
}
```

Статусы проверки:

- `not_reviewed` - не проверено;
- `in_review` - проверяется;
- `reviewed` - проверено;
- `needs_work` - требует доработки;
- `needs_recheck` - требует повторной проверки;
- `skipped` - пропущено.

Правила:

- для каждого события обновления должен существовать один текущий статус проверки;
- новый статус для события создается со значением `not_reviewed`;
- изменение статуса обновляет существующую запись и поле `updatedAt`;
- история изменения статусов в первой версии не хранится;
- ИИ-рапорт не может автоматически выставить статус проверки.

## `ai-reports.json`

Файл хранит ИИ-рапорты выбранных обновлений.

Форма файла:

```json
{
  "schemaVersion": 1,
  "aiReports": []
}
```

Схема ИИ-рапорта:

```json
{
  "id": "ai_report_<uuid-v4>",
  "studentId": "student_<uuid-v4>",
  "projectId": "project_<uuid-v4>",
  "updateEventId": "update_<uuid-v4>",
  "status": "running",
  "analysisMode": "current_state",
  "startedAt": "2026-05-07T10:15:30.000Z",
  "finishedAt": null,
  "repositoryUrlSnapshot": null,
  "projectLocalPathSnapshot": null,
  "analysisPath": null,
  "branch": "main",
  "previousCommit": null,
  "newCommit": null,
  "summary": null,
  "importantFiles": [],
  "changes": null,
  "risks": [],
  "manualReviewQuestions": [],
  "teacherCommentDraft": null,
  "fullText": null,
  "pullRequestContext": {
    "status": "not_requested",
    "number": null,
    "title": null,
    "url": null,
    "state": null,
    "error": null
  },
  "technicalDetails": null,
  "error": null,
  "createdAt": "2026-05-07T10:15:30.000Z",
  "updatedAt": "2026-05-07T10:15:30.000Z"
}
```

Статусы ИИ-рапорта:

- `running` - анализ выполняется;
- `ready` - рапорт готов;
- `error` - анализ завершился ошибкой.

Режимы анализа:

- `current_state` - ИИ анализирует текущее состояние проекта на `newCommit`;
- `changes_between_commits` - ИИ анализирует изменения между `previousCommit` и `newCommit`.

Статусы контекста pull request:

- `not_requested` - контекст не запрашивался;
- `found` - контекст найден и сохранен;
- `not_found` - pull request не найден;
- `unavailable` - контекст недоступен из-за ошибки GitHub CLI, доступа или сети.

Поля:

- `analysisMode` - режим анализа, выбранный автоматически по сохраненным коммитам обновления;
- `analysisPath` - относительный путь review-копии или безопасной папки анализа;
- `summary` - краткое резюме обновления человеческим языком;
- `importantFiles` - важные файлы для технической справки и ручного углубления, если преподаватель решит открыть код;
- `changes` - описание того, что добавлено как функциональность или видимое поведение проекта, либо краткое описание текущего состояния при первой загрузке;
- `risks` - пункты, которые преподавателю стоит быстро проверить руками;
- `manualReviewQuestions` - вопросы, которые стоит задать студенту;
- `teacherCommentDraft` - черновик комментария преподавателя;
- `fullText` - полный текст рапорта в человекочитаемой форме;
- `technicalDetails` - техническая диагностика запуска;
- `error` - ошибка ИИ-анализа или `null`.

Правила:

- ИИ-рапорт всегда привязан к одному событию обновления;
- если `analysisMode` равен `current_state`, поле `previousCommit` может быть `null`, а `newCommit` обязателен;
- если `analysisMode` равен `current_state`, рапорт создается только для первой загрузки проекта или другого явно разрешенного первого снимка, а не для повторного обновления без новых коммитов;
- если `analysisMode` равен `changes_between_commits`, поля `previousCommit` и `newCommit` обязательны, коммиты должны различаться, а количество новых коммитов должно быть больше нуля;
- повторное обновление без новых коммитов не создает новый ИИ-рапорт;
- у одного обновления может быть несколько ИИ-рапортов;
- повторный запуск ИИ не перезаписывает старые рапорты;
- неуспешный ИИ-запуск тоже сохраняется как рапорт со статусом `error`;
- черновик комментария не становится ручным комментарием автоматически;
- ИИ-рапорт не меняет статус проверки обновления;
- приложение не хранит API-ключи OpenAI в ИИ-рапортах или настройках.

- `success` - инструмент доступен;
- `warning` - базовая работа возможна, но часть возможностей ограничена;

Статусы локальной GitHub-аутентификации для `githubCli.authStatus`:

- `unknown` - состояние аутентификации еще не проверялось;
- `authenticated` - GitHub CLI локально авторизован;
- `not_authenticated` - GitHub CLI доступен, но локальная авторизация отсутствует или недействительна;
- `not_applicable` - статус аутентификации не относится к этому инструменту.

Правила:

- `dataRoot` по умолчанию указывает на папку `data/` внутри папки приложения;
- `appDataPath`, `repositoriesPath` и `reviewCopiesPath` хранятся как пути относительно `dataRoot`;
- команды локальных инструментов можно переопределить в настройках;
- диагностика инструмента хранит пользовательское сообщение, технические детали, версию инструмента и время последней проверки;
- приложение не хранит GitHub-пароли, GitHub-токены, OpenAI API-ключи или секреты Codex;
- GitHub CLI используется только через локально настроенную авторизацию преподавателя;
- Codex CLI используется только как локальный инструмент.

## Валидация JSON

При чтении и перед записью приложение должно проверять:

- наличие `schemaVersion`;
- соответствие `schemaVersion` значению `1`;
- наличие обязательной коллекции или объекта настроек;
- уникальность идентификаторов внутри каждой коллекции;
- формат идентификаторов и префиксов;
- формат временных меток;
- допустимые значения статусов и режимов анализа;
- обязательные связи между сущностями;
- отсутствие пустых строк в обязательных пользовательских текстах;
- корректность относительных путей внутри `data/`.

Проверка связей:

- каждый `project.studentId` должен ссылаться на существующего студента;
- каждый `student.projectId` должен ссылаться на существующий проект;
- каждый `updateEvent.runId` должен ссылаться на существующий запуск обновления;
- каждый комментарий, статус проверки и ИИ-рапорт должен ссылаться на существующее событие обновления;
- `studentId` и `projectId` в комментариях, статусах и ИИ-рапортах должны совпадать со связями события обновления.

## Удаление данных

Удаление студента в первой версии является явным действием преподавателя после подтверждения из `Ф001`.

Правила модели данных:

- удаление студента удаляет запись студента;
- связанный проект студента удаляется из `projects.json`;
- события обновлений этого проекта удаляются из `update-events.json`;
- запуски обновления удаляются только если после удаления связанных событий у запуска не остается ни одного события;
- сводки оставшихся запусков обновления пересчитываются по событиям, которые остаются связанными с запуском после удаления студента;
- комментарии, статусы проверки и ИИ-рапорты удаленных событий обновления удаляются вместе с ними;
- физическое удаление папок `data/repositories/` и `data/review-copies/` не является частью модели данных и требует отдельного правила файловых операций;
- перед удалением приложение должно предупредить преподавателя, если вместе со студентом будут удалены история, комментарии, статусы или ИИ-рапорты.

Мягкое удаление, архивирование студентов и история удаления в первой версии не требуются.

## Что не входит сюда

- файловые операции чтения и безопасной записи;
- Git-команды клонирования и обновления;
- создание и очистка review-копий;
- запуск VS Code;

## Критерии приемки

Первая версия модели данных считается готовой, если:

- все обязательные сущности первой версии имеют JSON-контракт;
- каждый студент связан с одним основным проектом;
- история обновлений хранится через `update-runs` и `update-events`;
- комментарии, статусы проверки и ИИ-рапорты связаны с конкретным событием обновления;
- статусы проектов, обновлений, проверки и ИИ-рапортов имеют фиксированные значения;
- локальные пути внутри `data/` хранятся переносимо;
- настройки не содержат секретов;
- контракт не требует SQLite, сервера, облачной базы данных или экспорта.
