# Supabase Auth и доступ администратора

## Роль этой спецификации

Эта спецификация фиксирует защищенный вход администратора в Pilot Tracker через Supabase Auth.

Решение меняет старую рамку первой версии: раньше интерфейс не требовал учетной записи внутри приложения, теперь рабочие данные должны быть закрыты от любого пользователя без действующей администраторской сессии.

При этом Pilot Tracker не становится командным продуктом, студенческим кабинетом или системой ролей. Приложение остается инструментом одного преподавателя-администратора.

## Цель

Дать преподавателю возможность открыть Pilot Tracker с другого компьютера, войти по email и паролю администратора и работать с теми же прикладными данными Supabase без раскрытия секретных ключей в браузере.

## Главный принцип

Доступ делится на три уровня:

```text
Браузер
  страница входа
  publishable key
  cookie-сессия Supabase Auth

Next.js runtime
  проверка сессии
  request-scoped Supabase-клиент
  вызов сценариев приложения

Supabase
  Auth
  RLS-политики
  прикладные таблицы Pilot Tracker
```

Рабочие данные доступны только после успешной проверки сессии администратора. Без сессии интерфейс не должен читать список студентов, обновления, комментарии, статусы или ИИ-рапорты.

## Модель администратора

Первая версия поддерживает одну учетную запись администратора-преподавателя.

Правила:

- администратор создается вручную в Supabase Auth, вне интерфейса Pilot Tracker;
- публичная регистрация из интерфейса не реализуется;
- публичная регистрация в настройках Supabase должна быть отключена, если проект открыт в интернете;
- восстановление пароля через интерфейс Pilot Tracker не входит в первый этап;
- смена пароля через интерфейс Pilot Tracker не входит в первый этап;
- выбор роли не показывается;
- студенческие аккаунты не создаются;
- несколько преподавателей не поддерживаются.

Разрешение администратора фиксируется через `app_metadata` пользователя Supabase Auth, например значением `pilot_tracker_role = "admin"`.

Запрещено использовать `user_metadata` или данные формы входа для решения, является ли пользователь администратором, потому что пользовательские metadata могут изменяться самим пользователем.

Если `app_metadata` администратора изменяется, действующие сессии могут увидеть новое право не мгновенно. Для строгой смены доступа нужно принудительно завершить сессии или потребовать повторный вход.

## Переменные окружения

Минимальные переменные:

- `NEXT_PUBLIC_SUPABASE_URL` - URL проекта Supabase, безопасный для браузера;
- `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY` - publishable key или другой публичный ключ Supabase, безопасный для браузера;
- `PILOT_SUPABASE_SERVICE_ROLE_KEY` - секретный ключ только для миграций, служебных скриптов и контролируемых серверных операций вне пользовательского потока.

Правила:

- `PILOT_SUPABASE_SERVICE_ROLE_KEY` не должен иметь префикс `NEXT_PUBLIC`;
- секретный ключ не попадает в клиентские React-компоненты;
- секретный ключ не попадает в пользовательские сообщения, диагностику и логи;
- `.env.example` должен показывать только имена переменных без реальных значений;
- пользовательский поток чтения и записи данных не должен зависеть от `service_role`.

## Интеграция Next.js

Для Next.js нужно использовать серверную интеграцию Supabase с cookie-сессиями.

Ожидаемая структура:

- утилита browser-клиента Supabase для страницы входа;
- утилита server-клиента Supabase для серверных компонентов, server actions и route handlers;
- `proxy.ts` или эквивалентный серверный слой Next.js для обновления сессии и защиты рабочих маршрутов;
- серверная проверка пользователя через `auth.getUser()`, а не только доверие локально прочитанной сессии;
- маршруты входа и выхода;
- защищенный layout или серверная проверка перед загрузкой рабочих данных.

Страница входа может быть доступна без сессии. Все рабочие страницы приложения должны требовать действующую администраторскую сессию.

## RLS

Все прикладные таблицы Pilot Tracker в схеме `public` должны иметь включенный Row Level Security.

Минимальные политики:

- неавторизованный пользователь не может читать данные;
- неавторизованный пользователь не может создавать, изменять или удалять данные;
- авторизованный пользователь без признака администратора не может читать или изменять данные;
- администратор может читать и изменять прикладные данные Pilot Tracker;
- политики используют безопасный признак из `app_metadata` или другой защищенный allow-list, но не `user_metadata`.

Если таблицы доступны через Data API, RLS является обязательной защитой. Отказ RLS должен обрабатываться как ошибка доступа или сохранения.

## Пользовательский поток

### Без сессии

1. Пользователь открывает рабочий маршрут.
2. Приложение проверяет сессию на сервере.
3. Сессии нет или она недействительна.
4. Приложение показывает страницу входа.
5. Прикладные данные не загружаются.

### Успешный вход

1. Пользователь вводит email и пароль администратора.
2. Supabase Auth подтверждает учетные данные.
3. Серверная проверка убеждается, что пользователь является разрешенным администратором.
4. Приложение создает cookie-сессию.
5. Пользователь попадает на список студентов.

### Ошибка входа

1. Пользователь вводит неверные данные или входит под неразрешенной учетной записью.
2. Приложение показывает короткую ошибку входа.
3. Сообщение не уточняет, существует ли email.
4. Прикладные данные остаются недоступными.

### Выход

1. Администратор нажимает действие выхода.
2. Приложение завершает сессию через Supabase Auth.
3. Пользователь возвращается на страницу входа.
4. Рабочие данные больше не показываются.

## Локальные инструменты на другом компьютере

Администраторский вход дает доступ к прикладным данным Supabase, но не переносит локальные инструменты и репозитории.

Если пользователь вошел с нового компьютера:

- список студентов, GitHub-ссылки, история обновлений, комментарии, статусы и ИИ-рапорты доступны из Supabase;
- локальные репозитории могут отсутствовать;
- Git, GitHub CLI, Codex CLI и VS Code могут быть не настроены;
- действия обновления, открытия кода и локального ИИ-анализа должны быть отключены или показывать понятную диагностику, пока локальная среда не готова.

## Ошибки и диагностика

Диагностика может сохранять:

- факт отказа входа без email в открытом виде, если это не нужно для диагностики;
- тип ошибки Supabase Auth;
- факт истечения сессии;
- факт отказа RLS;
- отсутствие обязательных переменных окружения;
- состояние подключения к Supabase.

Диагностика не должна сохранять:

- пароль;
- cookie;
- access token;
- refresh token;
- JWT целиком;
- `service_role` ключ;
- publishable key, если он не нужен для диагностики;
- полные значения переменных окружения.

## Тестирование

Минимальные автоматические проверки:

- рабочий маршрут без сессии возвращает вход или отказ доступа;
- страница входа не загружает прикладные данные;
- успешный вход через подмененный Supabase Auth-клиент открывает рабочий экран;
- ошибка входа показывает русское сообщение без раскрытия существования email;
- выход завершает сессию и закрывает доступ к рабочим данным;
- серверный пользовательский Supabase-клиент не использует `service_role`;
- миграция RLS включает защиту для всех прикладных таблиц;
- RLS отклоняет anon-доступ к чтению и записи;
- RLS разрешает доступ администратору.

Автоматические тесты не должны требовать реальные секреты Supabase. Проверка реального проекта выполняется отдельной ручной диагностикой.

## Что не входит

- публичная регистрация;
- восстановление пароля из интерфейса Pilot Tracker;
- смена пароля из интерфейса Pilot Tracker;
- несколько ролей;
- несколько преподавателей;
- студенческие аккаунты;
- управление пользователями;
- приглашения в команду;
- аудит действий нескольких пользователей;
- удаленный запуск Git, VS Code, GitHub CLI или Codex CLI на сервере;
- хранение кода студентов в Supabase;
- автоматическая публикация комментариев студентам.

## Критерии приемки

Техническая реализация соответствует этой спецификации, если:

- без сессии рабочие страницы не показывают прикладные данные;
- страница входа принимает email и пароль администратора;
- успешный вход открывает список студентов из Supabase;
- пользователь без признака администратора не получает доступ к данным;
- выход закрывает доступ к данным;
- публичная регистрация отсутствует;
- `service_role` не используется в пользовательском потоке и не попадает в браузер;
- RLS включен для прикладных таблиц и не открывает данные anon-пользователю;
- ошибки входа, истечения сессии и отказа RLS показаны понятным русским текстом;
- проверки проекта проходят без реальных Supabase-секретов.

## Открытые вопросы

- Нужно подтвердить, будет ли защищенный интерфейс развернут как внешний сайт или пока будет запускаться только через `localhost`.
- Нужно вручную создать администратора в Supabase Auth и назначить ему `app_metadata.pilot_tracker_role = "admin"`.
- Нужно проверить настройки Supabase Auth для публичной регистрации и redirect URL после выбора способа запуска.
