# План технических спецификаций

## Роль этого документа

Этот документ фиксирует порядок технических спецификаций, которые нужны перед началом разработки Pilot Tracker.

Функциональные спецификации описывают, что должен уметь продукт. Технические спецификации описывают, как это поведение должно быть реализовано без выхода за продуктовые ограничения.

## Принцип чтения

Перед разработкой любой части приложения нужно читать:

1. `спецификации/бизнес-контекст.md`;
2. функциональные спецификации `Ф001`-`Ф005`, которые относятся к задаче;
3. технические спецификации `Т001`-`Т012`, которые относятся к реализации этой задачи;
4. согласованные изменения технической рамки, если задача прямо касается области, которую они описывают.

Если функциональная и техническая спецификация расходятся, функциональная спецификация и бизнес-контекст имеют приоритет до явного обновления спецификаций владельцем проекта.

## Решения перед первой разработкой

Для начала первой разработки зафиксированы следующие технические решения:

- тип интерфейса первой рабочей версии - локальный веб-интерфейс на `localhost`;
- стек интерфейса - Next.js, React и TypeScript;
- менеджер пакетов - `pnpm`;
- runtime для сценариев, хранилища и интеграций - Node.js;
- точка входа локального интерфейса - маршруты Next.js в `app/`;
- сценарии приложения размещаются в `src/application/`, чтобы не смешивать их с маршрутизацией Next.js;
- отдельный командный интерфейс для преподавателя не входит в первый рабочий контур;
- отдельный общий пакет типов на старте не нужен, потому что первая разработка ведется как один локальный пакет приложения.

Эти решения не отменяют уже принятые продуктовые ограничения: первая версия остается инструментом одного преподавателя без многопользовательского режима, студенческого кабинета и экспорта истории курса. После согласования `Т011` прикладные данные хранятся в онлайн-проекте Supabase, а после согласования `Т012` доступ к рабочим данным защищается входом администратора через Supabase Auth. Студенческие репозитории, review-копии, локальный ИИ-анализ и машинные настройки остаются на компьютере преподавателя. Локальный Supabase через Docker не является целевым окружением разработки этой ветки.

## Границы первой версии

Технические спецификации не должны добавлять в первую версию:

- облачное хранение кода студентов;
- многопользовательский режим;
- студенческий кабинет;
- интеграцию с LMS;
- автоматическую публикацию комментариев;
- автоматическую оценку студентов;
- экспорт данных и истории курса;
- прямой вызов OpenAI API из приложения.

Серверный runtime Next.js и авторизация администратора разрешены только в границах `Т012`: один администратор-преподаватель, Supabase Auth, RLS и отсутствие публичной регистрации.

`Т011` согласован владельцем проекта и становится основанием для разработки Supabase-хранилища.

`Т012` согласован владельцем проекта как основание для разработки защищенного администраторского доступа.

Связанные спецификации `Ф005`, `Т001`, `Т003`, `Т004`, `Т008`, `Т009` и `Т010` должны читаться с учетом `Т011`. Если старый текст говорит, что основным хранилищем являются локальные JSON-файлы, приоритет имеет согласованная Supabase-рамка из `Т011`.

Связанные спецификации `Ф005`, `Т001`, `Т002`, `Т007`, `Т010` и `Т011` должны читаться с учетом `Т012`. Если старый текст говорит, что авторизация внутри приложения не входит в первую версию, приоритет имеет согласованная рамка защищенного администраторского доступа из `Т012`.
