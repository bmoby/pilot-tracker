# План технических спецификаций

## Роль этого документа

Этот документ фиксирует порядок технических спецификаций, которые нужны перед началом разработки Pilot Tracker.

Функциональные спецификации описывают, что должен уметь продукт. Технические спецификации описывают, как это поведение должно быть реализовано в первой локальной версии без выхода за продуктовые ограничения.

## Принцип чтения

Перед разработкой любой части приложения нужно читать:

1. `спецификации/бизнес-контекст.md`;
2. функциональные спецификации `Ф001`-`Ф005`, которые относятся к задаче;
3. технические спецификации `Т001`-`Т010`, которые относятся к реализации этой задачи.

Если функциональная и техническая спецификация расходятся, функциональная спецификация и бизнес-контекст имеют приоритет до явного обновления спецификаций владельцем проекта.

3. `Т005` и `Т006` - зафиксировать работу с Git и review-копиями.

## Решения перед первой разработкой

Для начала первой разработки зафиксированы следующие технические решения:

- тип интерфейса первой рабочей версии - локальный веб-интерфейс на `localhost`;
- стек интерфейса - Next.js, React и TypeScript;
- менеджер пакетов - `pnpm`;
- runtime для сценариев, хранилища и интеграций - Node.js;
- точка входа локального интерфейса - маршруты Next.js в `app/`;
- сценарии приложения размещаются в `src/application/`, чтобы не смешивать их с маршрутизацией Next.js;
- отдельный командный интерфейс для преподавателя не входит в первый рабочий контур;
- отдельный общий пакет типов на старте не нужен, потому что первая разработка ведется как один локальный пакет приложения.

Эти решения не отменяют уже принятые продуктовые ограничения: первая версия остается локальной, без облачной базы данных, серверного развертывания, авторизации внутри приложения, многопользовательского режима и экспорта истории курса.

## Границы первой версии

Технические спецификации не должны добавлять в первую версию:

- серверную версию продукта;
- облачную базу данных;
- авторизацию внутри приложения;
- многопользовательский режим;
- студенческий кабинет;
- интеграцию с LMS;
- автоматическую публикацию комментариев;
- автоматическую оценку студентов;
- экспорт данных и истории курса;
- прямой вызов OpenAI API из приложения.

Если такая возможность нужна позже, она должна появиться как отдельное решение после первой локальной версии.
