# Платформа и архитектура приложения

## Цель

Зафиксировать платформенную рамку первой версии Pilot Tracker и архитектурные границы, внутри которых должны реализовываться функциональные спецификации.

Эта спецификация фиксирует выбранную платформенную рамку первой разработки: локальный веб-интерфейс на Next.js, который запускается на компьютере преподавателя через `localhost`. Ее задача - определить, что приложение остается локальным рабочим инструментом преподавателя, не превращается в серверный продукт и не добавляет возможности, которых нет в бизнес-контексте и функциональных спецификациях.

## Контекст

Pilot Tracker первой версии работает на компьютере одного преподавателя и помогает вести одну активную рабочую группу студентов.

Приложение должно:

- хранить данные локально;
- работать с локальными JSON-файлами в `data/app/`;
- клонировать и обновлять студенческие репозитории в `data/repositories/`;
- создавать или переиспользовать review-копии в `data/review-copies/`;
- запускать локальные инструменты Git, VS Code, Codex CLI и при необходимости GitHub CLI;
- сохранять собственную историю обновлений, комментариев, статусов проверки и ИИ-рапортов;
- не отправлять изменения в репозитории студентов;
- не публиковать комментарии в GitHub автоматически;
- не принимать итоговые решения вместо преподавателя.

Первая версия не требует собственного сервера, облачной базы данных, авторизации внутри приложения, многопользовательского режима или студенческого интерфейса.

## Границы

Архитектура первой версии должна быть локальной.

Разрешенные внешние зависимости по назначению:

- GitHub как источник кода студенческих репозиториев;
- локально настроенная Git-аутентификация преподавателя для публичных и приватных репозиториев;
- GitHub CLI для получения контекста pull request, если он используется в ИИ-анализе;
- локальный Codex CLI для ручного ИИ-анализа выбранного обновления;
- VS Code для открытия кода выбранного обновления;
- файловая система компьютера преподавателя для данных приложения и локальных копий.

Архитектура не должна требовать:

- развернутого серверного приложения;
- постоянного фонового сервиса;
- облачной базы данных;
- сетевой синхронизации данных Pilot Tracker;
- учетных записей внутри приложения;
- ролей пользователей;
- прав доступа между пользователями;
- отдельного студенческого кабинета;
- прямого вызова OpenAI API из приложения.

Если для выбранного варианта интерфейса понадобится локальный процесс на `localhost`, он должен рассматриваться только как способ показать локальный интерфейс на компьютере преподавателя. Такой процесс не является серверной версией продукта, не должен требовать развертывания и не должен добавлять авторизацию или многопользовательские сценарии.

Для первой разработки выбран именно такой вариант: локальный веб-интерфейс на Next.js, который запускается как процесс разработки или локальная сборка на компьютере преподавателя.

## Решения первой версии

- Pilot Tracker является локальным приложением для одного преподавателя.
- Приложение работает с одной активной рабочей группой студентов.
- Основной источник истины для данных проверки - локальное хранилище приложения, а не GitHub.
- GitHub является источником кода студентов и контекста pull request, если этот контекст доступен.
- Локальные копии студенческих репозиториев используются для чтения, обновления и анализа, но не для внесения изменений в проекты студентов.
- Основная локальная копия проекта предназначена для обновления ветки `main`.
- Исторические версии открываются через отдельные review-копии или другой безопасный локальный механизм, чтобы не переключать основную копию между коммитами.
- ИИ-анализ запускается только вручную для конкретного обновления.
- ИИ-интеграция выполняется через локальный Codex CLI.
- Приложение не хранит API-ключи OpenAI и не управляет авторизацией Codex.
- Приложение не хранит GitHub-пароли и не обходит права доступа GitHub.
- Формат хранения первой версии - локальные JSON-файлы в `data/app/`.
- SQLite, внешняя база данных и облачное хранилище не входят в первую версию.
- Интерфейс первой версии - локальный веб-интерфейс на Next.js, доступный через `localhost`.
- Процесс Next.js используется только как локальный runtime интерфейса и сценариев приложения на компьютере преподавателя.
- Next.js не делает Pilot Tracker серверным продуктом и не добавляет развертывание, авторизацию, роли, многопользовательский доступ или облачное хранение.
- Отдельный командный интерфейс не входит в первый рабочий контур. Диагностика локальной среды должна быть доступна через интерфейс настроек.

## Требования

### Архитектурные слои

Реализация должна разделять ответственность минимум на следующие слои:

- интерфейс преподавателя;
- сценарии приложения;
- доменная модель и правила;
- локальное хранилище;
- интеграции с локальными инструментами;
- диагностика и обработка ошибок.

Интерфейс не должен напрямую изменять JSON-файлы, запускать Git-команды или управлять Codex CLI. Такие действия должны проходить через сценарии приложения и специализированные сервисы.

### Сценарии приложения

Сценарии приложения должны выражать пользовательские действия из функциональных спецификаций:

- создать, изменить и удалить студента;
- обновить все проекты;
- обновить один проект;
- открыть страницу студента и ленту обновлений;
- открыть выбранное обновление;
- добавить или изменить комментарий преподавателя;
- изменить статус проверки обновления;
- открыть код выбранного обновления в VS Code;
- запустить ИИ-анализ выбранного обновления;
- сохранить ИИ-рапорт или ошибку анализа.

Сценарии должны возвращать понятный результат для интерфейса: успешное состояние, частичный успех, ожидаемую ошибку или диагностическую ошибку.

### Локальное хранилище

Хранилище должно сохранять данные, описанные в `Ф005`, в локальной папке `data/`.

Минимальные правила:

- данные приложения сохраняются в `data/app/`;
- основные локальные репозитории находятся в `data/repositories/`;
- review-копии находятся в `data/review-copies/`;
- старые события обновления не перезаписываются новыми событиями;
- ИИ-рапорты сохраняются внутри соответствующих обновлений по связи данных;
- удаление данных с историей требует явного подтверждения на уровне пользовательского сценария.

Подробные JSON-схемы и правила безопасной записи должны быть описаны в `Т003` и `Т004`.

### Локальные интеграции

Интеграции должны быть изолированы от интерфейса и доменной модели.

Минимальные интеграции первой версии:

- Git для клонирования, получения ветки `main`, сравнения коммитов и обновления основной локальной копии;
- VS Code для открытия review-копии выбранного обновления;
- Codex CLI для локального ИИ-анализа;
- GitHub CLI для контекста pull request, если он используется.

Интеграции должны фиксировать ошибки так, чтобы приложение могло сохранить их в истории обновления или ИИ-рапорте, а не терять после завершения процесса.

### Безопасность локальных репозиториев

Приложение не должно:

- создавать коммиты в репозиториях студентов;
- выполнять `push`;
- публиковать комментарии;
- автоматически исправлять грязное состояние локальной копии;
- переключать основную локальную копию проекта на исторические коммиты для проверки.

Если локальная копия содержит незакоммиченные изменения или повреждена, это должно сохраняться как ошибка обновления конкретного проекта.

### Интерфейсная независимость

Несмотря на выбор Next.js для первой разработки, доменная модель, локальное хранилище, сценарии приложения и интеграции не должны зависеть от компонентов интерфейса.

Интерфейс Next.js является адаптером над сценариями приложения. Он не должен напрямую читать или записывать JSON-файлы, запускать Git, открывать VS Code или управлять Codex CLI.

Выбор Next.js не должен менять модель данных, правила обновления проектов, способ хранения истории или правила запуска ИИ-анализа. Если позже понадобится настольная оболочка или служебный командный интерфейс, они должны переиспользовать те же сценарии приложения без переписывания доменной модели.

### Наблюдаемость и ошибки

Ошибки должны быть частью пользовательских сценариев, а не только техническими логами.

Приложение должно сохранять:

- ошибки обновления проектов;
- ошибки отсутствующей GitHub-ссылки;
- ошибки доступа к приватным репозиториям;
- ошибки недоступной или поврежденной локальной копии;
- ошибки открытия VS Code;
- ошибки запуска Codex CLI;
- ошибки получения контекста pull request, если он запрашивался;
- ошибки сохранения данных.

Подробная стратегия диагностики и тестирования должна быть описана в `Т010`.

## Что не входит

В эту спецификацию не входят:

- детальная структура экранов;
- точные JSON-схемы;
- правила безопасной записи JSON-файлов;
- точные команды Git сверх уже заданных в `Ф002`;
- реализация review-копий;
- схема запуска `codex exec`;
- дизайн настроек;
- тестовая стратегия на уровне отдельных модулей;
- альтернативная настольная оболочка;
- отдельный продуктовый командный интерфейс.

В первую версию продукта не входят:

- серверная версия;
- облачная база данных;
- авторизация внутри приложения;
- многопользовательская работа;
- студенческий кабинет;
- интеграция с LMS;
- экспорт истории курса;
- автоматическая публикация комментариев;
- автоматическая оценка студентов;
- прямой вызов OpenAI API из приложения.

## Критерии приемки

Техническая платформа первой версии считается согласованной, если:

- реализация может работать локально на компьютере преподавателя;
- приложение не требует собственного сервера;
- приложение не требует облачной базы данных;
- приложение не требует авторизации внутри приложения;
- данные проверки сохраняются локально в `data/`;
- архитектура отделяет интерфейс от сценариев приложения, хранилища и интеграций;
- Git, VS Code, Codex CLI и GitHub CLI вызываются через изолированные интеграционные модули;
- сценарии обновления не отправляют изменения в репозитории студентов;
- ИИ-анализ запускается только вручную через локальный Codex CLI;
- ошибки внешних инструментов сохраняются как часть истории или диагностики;
- локальный веб-интерфейс на Next.js запускается на `localhost` и не требует серверного развертывания;
- доменная модель и хранилище остаются отделены от Next.js-компонентов;
- ни одно техническое решение не добавляет серверную, облачную, авторизационную или многопользовательскую область.

## Открытые вопросы

На момент начала первой разработки открытых платформенных вопросов, которые блокируют старт, нет.

Закрытые решения:

- интерфейс первой рабочей версии - локальный веб-интерфейс на Next.js через `localhost`;
- отдельный командный интерфейс для преподавателя в первый рабочий контур не входит;
- диагностика локальной среды реализуется через экран настроек;
- первая разработка ориентируется на локальную среду, где доступны Node.js, `pnpm`, Git, VS Code и Codex CLI по правилам связанных спецификаций.

Отдельная матрица поддерживаемых операционных систем не фиксируется как продуктовая возможность первой версии. Приложение остается локальным инструментом преподавателя и не должно требовать серверного развертывания, облака, авторизации или многопользовательского режима.
