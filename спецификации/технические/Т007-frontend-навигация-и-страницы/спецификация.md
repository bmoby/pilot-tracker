# Фронтенд, навигация и страницы

## Цель

Зафиксировать набор экранов первой версии Pilot Tracker, правила навигации между ними, обязательные состояния интерфейса и принципы показа ошибок.

Эта спецификация описывает только пользовательский интерфейс локального приложения преподавателя. Она не меняет функциональные правила из `Ф001`-`Ф005` и не добавляет новые продуктовые возможности.

Интерфейс первой разработки реализуется на Next.js как локальный веб-интерфейс через `localhost`. Компоненты и маршруты Next.js должны оставаться адаптером над сценариями приложения и не должны напрямую работать с JSON-файлами, Git, VS Code, GitHub CLI или Codex CLI.

## Контекст первой версии

Интерфейс первой версии должен поддерживать главный рабочий поток преподавателя:

1. Открыть список студентов.
2. Понять, у кого появились новые изменения, где есть ошибки и какие проекты не подключены.
3. Запустить обновление всех проектов или открыть конкретного студента.
4. На странице студента увидеть ленту обновлений.
5. Открыть конкретное обновление.
6. Запустить ИИ-анализ выбранного обновления, открыть код в VS Code, оставить комментарий и изменить статус проверки.
7. При необходимости перейти в настройки, чтобы проверить пути и локальные инструменты.

Первый экран приложения должен быть рабочим списком студентов. Маркетинговая посадочная страница, приветственный промо-экран, экран выбора курса или экран авторизации в первой версии не нужны.

## Общая структура интерфейса

Минимальная структура приложения:

- постоянная верхняя или боковая навигация;
- главный рабочий экран списка студентов;
- страница конкретного студента;
- рабочая область выбранного обновления внутри страницы студента;
- экран настроек;
- модальные окна для коротких форм и подтверждений;
- системные уведомления о завершении действий и ошибках.

Навигация должна оставаться простой, потому что первая версия работает с одной активной когортой и одним преподавателем.

Основные разделы навигации:

- `Студенты` - главный экран со списком студентов и статусами проектов;
- `Настройки` - экран путей, команд локальных инструментов и диагностики;
- `Диагностика` может быть отдельной вкладкой внутри настроек, но не отдельным продуктовым разделом первой версии.

В навигации не должно быть разделов для курсов, семестров, LMS, экспорта, студенческого кабинета, ролей или пользователей.

## Экран списка студентов

Экран списка студентов является главным экраном первой версии и соответствует `Ф001`.

На экране нужно показывать:

- заголовок рабочей области;
- действие добавления студента;
- действие обновления всех проектов;
- состояние общего обновления, если оно выполняется;
- компактную итоговую сводку последнего запуска обновления, если она есть;
- список студентов, отсортированный по правилам `Ф001`;
- пустое состояние, если студентов еще нет;
- ошибки хранения или загрузки данных, если список нельзя прочитать.

Для каждого студента в списке нужно показывать:

- имя студента;
- краткую дополнительную информацию, если она есть;
- ссылку GitHub как кликабельное действие или признак отсутствующей ссылки;
- последний статус проекта;
- дату последнего обновления;
- признак первой успешной загрузки, если последний результат проекта - первая загрузка;
- признак новых изменений;
- количество новых коммитов в последнем релевантном обновлении, если оно известно;
- ошибку последнего обновления, если она есть;
- действие открытия страницы студента;
- действия редактирования и удаления студента, если они доступны из списка.

Компактная карточка студента не должна показывать:

- внутренний идентификатор студента;
- локальный путь проекта;
- последний известный коммит или хеш коммита;
- служебные технические детали, которые не помогают выбрать следующую проверку.

Действия карточки должны быть собраны в одну понятную зону управления. Если для редактирования и удаления используется меню, на карточке не должно быть второго конкурирующего меню с похожим назначением.

GitHub-ссылка в карточке должна открывать репозиторий в новой вкладке или внешнем браузере. Если ссылки нет, вместо нее нужно показать короткий рабочий признак отсутствующего репозитория и оставить доступным редактирование студента.

Список должен помогать выбрать следующую проверку. Поэтому визуальный приоритет получают:

- студенты с первой успешной загрузкой проекта;
- студенты с новыми изменениями;
- студенты с ошибками обновления;
- студенты без подключенного репозитория;
- студенты без новых изменений.

Поиск, теги, группы, сложные фильтры, архивы и переключение курсов в первой версии не требуются.

## Форма добавления и редактирования студента

Форма студента должна быть короткой и соответствовать `Ф001`.

Поля формы:

- имя студента;
- дополнительная информация в свободной форме;
- ссылка на GitHub-репозиторий.

Валидация:

- имя обязательно;
- имя не может быть пустой строкой;
- ссылка GitHub не обязательна;
- если ссылка GitHub указана, интерфейс должен проверить, что она выглядит как ссылка на репозиторий;
- проверка фактического доступа к репозиторию выполняется во время обновления, а не при сохранении формы.

При изменении ссылки GitHub интерфейс должен показать предупреждение, что это повлияет на будущие обновления и связанную локальную копию. Интерфейс не должен удалять историю студента молча.

Удаление студента требует подтверждения. Если у студента есть история обновлений, комментарии, статусы или ИИ-рапорты, подтверждение должно явно предупредить, что эти данные будут удалены вместе со студентом.

## Обновление всех проектов

Действие обновления всех проектов запускается с экрана списка студентов и соответствует `Ф002`.

Во время обновления интерфейс должен:

- показать, что общий запуск выполняется;
- заблокировать повторный запуск общего обновления до завершения текущего;
- оставить доступным просмотр уже загруженных данных, если это технически возможно;
- показывать промежуточное состояние по студентам, если оно доступно;
- не закрывать экран и не терять контекст преподавателя.

После завершения нужно показать сводку:

- сколько проектов обновлено успешно;
- сколько проектов впервые загружено локально;
- сколько проектов имеют новые изменения;
- сколько новых коммитов найдено;
- сколько проектов без новых изменений;
- сколько проектов завершились ошибкой;
- сколько студентов имеют проект без ссылки GitHub.

Сводка должна быть компактной и не должна занимать крупную верхнюю область из нескольких самостоятельных блоков, если эти блоки не помогают преподавателю выбрать следующую проверку. Сводку последнего обновления можно объединять с информацией о времени последнего запуска и состоянием списка.

Отдельные крупные счетчики общего количества студентов, студентов с репозиторием и студентов без репозитория не являются обязательной частью главного экрана. Их не нужно показывать на постоянной основе, если те же состояния уже понятны из списка, сводки обновления или признаков на карточках студентов.

Ошибка одного проекта не должна превращать весь запуск в неуспешный интерфейсный сценарий. Сводка должна показывать частичный результат, а ошибки отдельных проектов должны быть видны в списке и в истории студента.

## Страница студента

Страница студента соответствует `Ф003` и строится вокруг ленты обновлений.

Верхний контекст страницы должен показывать:

- имя студента;
- дополнительную информацию;
- ссылку GitHub;
- основной локальный путь проекта, если он есть;
- текущий статус проекта;
- последний известный коммит;
- дату последнего обновления;
- краткое описание проекта, созданное ИИ, или состояние его отсутствия;
- действие обновления только этого проекта.

Лента обновлений должна показывать все события обновления проекта студента:

- первую загрузку проекта;
- успешные обновления с новыми изменениями;
- успешные обновления без новых изменений;
- ошибки обновления;
- события недоступности проекта из-за отсутствующей GitHub-ссылки.

Новые записи показываются выше старых. Порядок должен быть стабильным по дате и времени события.

Если у студента есть GitHub-ссылка, но локальной копии еще нет, страница должна показать это как рабочее состояние и предложить обновить проект. Если GitHub-ссылки нет, страница должна объяснить, что проект не подключен и данные студента нужно исправить через форму студента.

## Карточка обновления

Карточка обновления в ленте должна кратко показывать:

- дату и время обновления;
- результат обновления;
- статус проверки;
- количество новых коммитов;
- старый и новый коммит, если они известны;
- краткое описание результата;
- признак ИИ-анализа;
- количество сохраненных ИИ-рапортов;
- количество комментариев преподавателя;
- наличие ошибки обновления или ИИ-анализа;
- действие открытия деталей обновления.

Карточка не должна показывать подробный diff, список файлов, содержимое pull request или встроенный просмотр кода. Эти данные относятся к ИИ-анализу, VS Code или будущим решениям, но не к списку обновлений первой версии.

## Детали выбранного обновления

Детали обновления являются рабочей областью проверки конкретного события из `Ф003`.

В деталях нужно показывать:

- дату и время обновления;
- результат обновления;
- ветку;
- старый коммит;
- новый коммит;
- количество новых коммитов;
- текст ошибки обновления, если она была;
- текущий статус проверки;
- список комментариев преподавателя;
- форму добавления комментария;
- список ИИ-рапортов;
- состояние текущего ИИ-запуска;
- действие запуска ИИ-анализа;
- действие открытия кода в VS Code на версии выбранного обновления.

Если обновление завершилось ошибкой, его все равно можно открыть. В деталях нужно показать ошибку, разрешить комментарий и изменение статуса, но недоступные действия должны быть отключены с понятной причиной.

Если новых коммитов нет, интерфейс должен объяснить, что событие сохранено как успешное обновление без новых изменений. Такое обновление можно открыть и комментировать.

Если обновление является первой загрузкой проекта, интерфейс должен отличать его от повторного обновления без изменений: показать, что это первое известное состояние проекта, а количество новых коммитов не вычислялось из-за отсутствия предыдущей границы сравнения.

## Комментарии преподавателя

Комментарии привязываются к обновлению целиком.

Интерфейс должен позволять:

- добавить один или несколько комментариев к выбранному обновлению;
- отредактировать существующий комментарий;
- увидеть дату создания;
- увидеть дату последнего редактирования, если комментарий менялся;
- увидеть количество комментариев в карточке обновления.

В первой версии интерфейс не должен предлагать комментарии к файлам, строкам, коммитам или фрагментам diff. Комментарии не отправляются студентам, не публикуются в GitHub и не считаются официальной выдачей оценки.

## Статус проверки

Статус проверки относится к конкретному обновлению.

Интерфейс должен поддерживать статусы из `Ф003`:

- не проверено;
- проверяется;
- проверено;
- требует доработки;
- требует повторной проверки;
- пропущено.

Изменение статуса выполняется вручную преподавателем в деталях обновления. Запуск обновления, запуск ИИ-анализа или открытие VS Code не должны автоматически менять статус на `проверено`.

## ИИ-рапорты в интерфейсе

ИИ-анализ запускается только из деталей выбранного обновления и соответствует `Ф004`.

Интерфейс должен:

- показывать действие запуска ИИ-анализа только там, где есть контекст конкретного обновления;
- отключать запуск с объяснением, если нет локального пути, нового коммита или доступного Codex CLI;
- если действие отключено из-за заранее известной недоступности Codex CLI, не создавать ошибочный ИИ-рапорт;
- если старого коммита нет, показывать, что ИИ проанализирует текущее состояние проекта, а не изменения между двумя коммитами;
- показывать состояние выполнения анализа;
- сохранять видимость выбранного обновления во время анализа;
- показывать успешный ИИ-рапорт внутри выбранного обновления;
- показывать ошибочный ИИ-рапорт внутри выбранного обновления, если ошибка произошла после начатой попытки анализа;
- показывать несколько ИИ-рапортов одного обновления без перезаписи старых.

Черновик комментария внутри ИИ-рапорта не должен автоматически становиться ручным комментарием. Интерфейс может дать преподавателю использовать черновик как основу для нового комментария, но итоговый комментарий должен сохраняться как действие преподавателя.

## Открытие кода в VS Code

Действие открытия кода доступно в деталях выбранного обновления и соответствует `Ф003`, `Ф005` и будущей технической реализации review-копий.

Интерфейс должен:

- показывать действие открытия VS Code только для выбранного обновления;
- объяснять, что открывается версия кода этого обновления;
- отключать действие, если нет известного нового коммита;
- отключать действие, если локальный репозиторий недоступен;
- показывать ошибку, если VS Code или команда открытия недоступны;
- показывать путь, который приложение пыталось открыть, если открытие завершилось ошибкой.

Интерфейс не должен предлагать выбор другого редактора в первой версии. Review-копии не показываются отдельным пользовательским разделом.

## Экран настроек

Экран настроек описывается подробнее в `Т008`, но для навигации первой версии он должен быть доступен из основного интерфейса.

На экране настроек должны быть зоны:

- пути данных;
- локальные инструменты;
- команды запуска;
- диагностика доступности.

Настройки не должны становиться мастером авторизации, экраном управления курсами, экраном LMS или экспортом данных.

## Пустые состояния

Минимальные пустые состояния:

- нет студентов: показать, что список пуст, и дать действие добавления студента;
- у студента нет GitHub-ссылки: показать, что проект не подключен, и дать действие редактирования студента;
- проект еще не загружен локально: показать, что нужна первая загрузка через обновление;
- лента обновлений пуста: показать, что обновлений еще не было;
- ИИ-описание проекта отсутствует: показать состояние отсутствия без автоматического запуска ИИ;
- комментариев нет: показать место для добавления первого комментария;
- ИИ-рапортов нет: показать, что анализ еще не запускался;
- настройки инструментов не проверялись: показать нейтральное состояние диагностики.

Пустые состояния должны быть рабочими, без маркетинговых обещаний и без расширения продукта за пределы первой версии.

## Состояния загрузки и выполнения

Интерфейс должен различать:

- начальную загрузку данных приложения;
- сохранение формы студента;
- удаление студента;
- общий запуск обновления;
- одиночное обновление проекта;
- подготовку review-копии;
- открытие VS Code;
- запуск ИИ-анализа;
- сохранение комментария;
- изменение статуса проверки;
- проверку настроек и диагностику инструментов.

Для длительных операций нужно показывать выполняющееся состояние и не позволять запускать конфликтующее действие повторно. Если операция относится к конкретному студенту или обновлению, состояние должно быть видно рядом с этим объектом, а не только в общем уведомлении.

## Ошибки в интерфейсе

Ошибки должны быть видны в том месте, где преподаватель может с ними работать.

Минимальные правила:

- ошибка загрузки данных показывается на уровне соответствующего экрана;
- ошибка сохранения формы показывается рядом с формой;
- ошибка общего обновления показывается в сводке и по затронутым студентам;
- ошибка одиночного обновления сохраняется как событие ленты;
- ошибка начатого ИИ-анализа показывается внутри выбранного обновления как ИИ-рапорт со статусом ошибки;
- заранее известная недоступность ИИ-инструмента показывается как причина отключенного действия, а не как ИИ-рапорт;
- ошибка открытия VS Code показывается в деталях обновления;
- ошибка настройки инструмента показывается на экране настроек;
- технические детали могут быть раскрыты по запросу преподавателя, но короткое пользовательское сообщение должно быть понятным без чтения лога.

Интерфейс не должен скрывать частично успешный результат из-за одной ошибки. Если часть операции выполнена и сохранена, это должно быть видно.

## Ограничения первой версии

Фронтенд первой версии не должен добавлять:

- авторизацию внутри приложения;
- роли пользователей;
- студенческий интерфейс;
- LMS-интеграцию;
- управление несколькими курсами или семестрами;
- архивы когорт;
- экспорт данных или истории курса;
- публикацию комментариев в GitHub;
- отправку комментариев студентам;
- встроенный полноценный просмотр кода;
- выбор редактора кроме VS Code;
- автоматическую оценку проекта;
- автоматический запуск ИИ после обновления.

## Критерии приемки

Техническая реализация интерфейса соответствует этой спецификации, если:

- первым рабочим экраном является список студентов;
- преподаватель может открыть настройки из основной навигации;
- преподаватель может добавить, изменить и удалить студента с нужными подтверждениями;
- список студентов показывает статусы, первые загрузки, новые изменения, ошибки и локальные пути;
- общий запуск обновления имеет понятное состояние выполнения и итоговую сводку;
- преподаватель может открыть страницу студента из списка;
- страница студента показывает верхний контекст и ленту обновлений;
- преподаватель может запустить обновление одного проекта со страницы студента;
- каждое обновление можно открыть в деталях;
- детали обновления показывают коммиты, статус проверки, комментарии, ИИ-рапорты и ошибки;
- преподаватель может вручную изменить статус проверки обновления;
- преподаватель может добавить и отредактировать комментарий к обновлению;
- запуск ИИ доступен только из деталей выбранного обновления;
- заранее отключенный запуск ИИ не создает ошибочный ИИ-рапорт;
- ошибка начатого ИИ-анализа показывается как ИИ-рапорт внутри выбранного обновления;
- действие открытия VS Code относится к версии выбранного обновления;
- пустые, загрузочные и ошибочные состояния описаны и реализованы для основных экранов;
- интерфейс не содержит маркетинговой посадочной страницы и не добавляет запрещенные функции первой версии.
