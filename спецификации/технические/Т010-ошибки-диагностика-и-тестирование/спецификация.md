# Ошибки, диагностика и тестирование

## Цель

Зафиксировать единую техническую стратегию ошибок, локальной диагностики, журналов и минимального тестирования, достаточную для начала разработки первой версии Pilot Tracker.

Спецификация описывает:

- как классифицировать ошибки;
- какие ошибки сохраняются в истории продукта;
- какие события пишутся в диагностические журналы;
- какие данные нельзя записывать в диагностику;
- какие уровни тестов нужны на старте разработки;
- какие сценарии нужно проверить вручную перед первым рабочим использованием.

## Продуктовая рамка

Pilot Tracker - локальный инструмент одного преподавателя. Первая версия не должна требовать собственного сервера, облачной базы данных, авторизации внутри приложения или сетевой диагностики за пределами GitHub, Git и локальных инструментов преподавателя.

Ошибки не должны разрушать историю проверки. Если операция не удалась, приложение должно сохранить понятное состояние и дать преподавателю продолжить работу с остальными студентами, обновлениями и сохраненными данными.

## Общие принципы обработки ошибок

Базовые правила:

- ошибка одного студента или проекта не останавливает общий запуск обновления остальных проектов;
- ошибка ИИ-анализа не ломает страницу студента и не меняет статус проверки;
- заранее заблокированное действие ИИ-анализа не создает ошибочный ИИ-рапорт;
- начатая попытка ИИ-анализа всегда завершается успешным или ошибочным ИИ-рапортом;
- ошибка получения pull request не блокирует основной локальный анализ;
- ошибка открытия VS Code не меняет данные обновления;
- ошибка сохранения данных считается критичной для текущего действия и должна быть явно видна;
- история успешных и неуспешных действий не должна случайно перезаписываться;
- пользовательский текст ошибки должен быть понятен преподавателю;
- технические детали должны быть доступны для диагностики, но не перегружать основной интерфейс.

Приложение должно различать пользовательское сообщение и техническую диагностику. Пользовательское сообщение объясняет, что произошло и что можно сделать. Диагностика помогает разработчику или опытному пользователю понять причину.

## Классы ошибок

### Ошибки данных

Ошибки данных связаны с локальными JSON-файлами и связями между сущностями.

Примеры:

- файл данных отсутствует;
- JSON поврежден;
- запись с нужным идентификатором не найдена;
- обновление ссылается на отсутствующий проект;
- ИИ-рапорт ссылается на отсутствующее обновление;
- обязательное поле отсутствует;
- значение статуса неизвестно текущей версии приложения.

Поведение:

- приложение не должно молча удалять поврежденные данные;
- ошибка чтения должна быть видна в диагностике;
- если возможно, приложение показывает частично доступные данные;
- запись новых данных не должна выполняться поверх поврежденного файла без безопасного решения;
- для критичных ошибок хранения действие останавливается.

### Ошибки локального хранилища

Ошибки хранилища связаны с папкой `data/`, безопасной записью файлов и правами доступа.

Примеры:

- папку `data/` нельзя создать;
- файл нельзя прочитать;
- файл нельзя записать;
- временный файл безопасной записи нельзя переименовать;
- недостаточно места на диске;
- нет прав на запись;
- путь находится вне ожидаемой структуры данных.

Поведение:

- пользовательское действие считается неуспешным, если его результат нельзя надежно сохранить;
- приложение должно показать путь, с которым возникла проблема;
- диагностика должна сохранить системную причину ошибки, если она доступна;
- приложение не должно считать действие успешным до подтвержденной записи данных.

### Ошибки Git

Ошибки Git возникают при клонировании, обновлении, чтении истории и подготовке review-копий.

Примеры:

- Git недоступен;
- ссылка GitHub некорректная;
- репозиторий недоступен;
- нет доступа к приватному репозиторию;
- отсутствует интернет;
- ветка `main` отсутствует;
- локальная папка проекта отсутствует;
- локальная папка не является ожидаемым репозиторием;
- локальная копия содержит изменения;
- команда Git завершилась ошибкой;
- нужный коммит отсутствует;
- review-копию нельзя создать.

Поведение:

- ошибка обновления проекта сохраняется в событии обновления;
- общий запуск обновления продолжается для остальных проектов;
- ошибка подготовки review-копии блокирует открытие исторической версии и ИИ-анализ для выбранного обновления;
- приложение не должно автоматически исправлять грязное состояние локальной копии;
- приложение не должно делать commit или push.

### Ошибки локальных инструментов

Ошибки локальных инструментов связаны с Git, GitHub CLI, Codex CLI и VS Code.

Примеры:

- команда не найдена;
- команда недоступна по настроенному пути;
- версия инструмента несовместима с первой версией приложения;
- GitHub CLI не авторизован;
- Codex CLI не авторизован локально;
- VS Code нельзя открыть;
- процесс завершился с ненулевым кодом.

Поведение:

- настройки должны показывать доступность инструмента;
- действие, зависящее от инструмента, не должно начинаться без базовой проверки доступности;
- ошибка инструмента сохраняется в диагностике действия;
- ошибка GitHub CLI при получении pull request не блокирует основной локальный ИИ-анализ;
- известная заранее недоступность Codex CLI блокирует запуск ИИ без создания ИИ-рапорта;
- ошибка Codex CLI после начатой попытки сохраняется как неуспешный ИИ-рапорт.

### Ошибки ИИ-анализа

Ошибки ИИ-анализа связаны с подготовкой контекста, запуском Codex CLI, схемой результата и сохранением ИИ-рапорта.

Примеры:

- отсутствует новое состояние проекта для анализа во время начатой попытки;
- review-копия недоступна;
- Codex CLI не найден во время начатой попытки;
- Codex CLI завершился ошибкой;
- финальный ответ не создан;
- финальный ответ не соответствует схеме;
- результат нельзя сохранить;
- входные данные были усечены сильнее допустимого.

Поведение:

- ошибка сохраняется как неуспешный ИИ-рапорт внутри выбранного обновления, если попытка анализа была начата;
- если действие было заранее недоступно, ошибка показывается в интерфейсе и диагностике без создания ИИ-рапорта;
- старые успешные ИИ-рапорты не удаляются;
- статус проверки обновления не меняется;
- преподаватель может повторить запуск после исправления причины.

### Ошибки интерфейса и сценариев

Ошибки интерфейса связаны с невозможностью выполнить действие в текущем состоянии.

Примеры:

- запуск ИИ недоступен, потому что нет локального проекта;
- открытие VS Code недоступно, потому что нет нового коммита;
- обновление недоступно, потому что у студента нет GitHub-ссылки;
- сохранение комментария недоступно, потому что обновление не найдено.

Поведение:

- недоступное действие должно объяснять причину;
- интерфейс не должен показывать успешное состояние до завершения операции;
- повторное нажатие во время выполнения не должно создавать дублирующие операции без явного решения;
- после ошибки преподаватель должен оставаться в том же рабочем контексте.

## Уровни серьезности

Для диагностики используется минимальный набор уровней:

- `информация` - штатное событие, полезное для понимания последовательности действий;
- `предупреждение` - действие завершилось, но есть ограничение или частичная потеря контекста;
- `ошибка` - конкретное действие не выполнено, но приложение продолжает работу;
- `критическая ошибка` - приложение не может надежно читать или сохранять ключевые данные.

Пользовательский интерфейс не обязан показывать эти названия уровней дословно, но диагностика должна хранить уровень события.

## Что сохраняется в истории продукта

История продукта хранит события, важные для преподавателя и курса.

Нужно сохранять:

- успешные обновления проектов;
- обновления без новых изменений;
- события отсутствующей GitHub-ссылки во время запуска обновления;
- ошибки обновления проекта;
- комментарии преподавателя;
- статусы проверки обновлений;
- успешные ИИ-рапорты;
- неуспешные ИИ-рапорты;
- краткое описание проекта, созданное или обновленное ИИ.

Не нужно сохранять в историю продукта:

- каждую техническую команду;
- полный stdout и stderr процессов;
- временные пути, если они не нужны для понимания результата;
- внутренние повторы операций;
- проверки доступности инструментов без пользовательского действия.

Такие детали относятся к диагностическим журналам.

## Диагностические журналы

Диагностический журнал нужен для локального разбора проблем во время разработки и ежедневной работы преподавателя.

Рекомендуемое расположение первой версии:

```text
data/logs/diagnostics.jsonl
```

Диагностический журнал хранится отдельно от JSON-коллекций приложения в `data/app/`, чтобы не смешивать служебные журналы с основными данными курса. Если реализация выберет другой путь внутри `data/`, он должен оставаться локальным и понятным для резервного копирования.

Минимальные поля события журнала:

- дата и время;
- уровень;
- область: данные, хранилище, Git, review-копия, VS Code, GitHub CLI, Codex CLI, ИИ-анализ, интерфейс;
- идентификатор связанного действия, если он есть;
- идентификатор студента, если применимо;
- идентификатор проекта, если применимо;
- идентификатор обновления, если применимо;
- короткое пользовательское описание;
- техническое описание;
- имя команды без секретов, если применимо;
- код выхода процесса, если применимо;
- путь рабочей папки, если применимо;
- признак, что данные были усечены.

Журнал может быть обычным JSONL-файлом или другим локальным структурированным форматом, если он читается без внешнего сервера.

## Что нельзя писать в диагностику

Диагностика не должна сохранять:

- пароли;
- токены GitHub;
- API-ключи;
- cookie;
- приватные значения переменных окружения;
- полные домашние пути, если для диагностики достаточно относительного пути внутри проекта;
- полный diff, если он может быть большим или содержать лишние данные;
- секреты из stdout или stderr внешних инструментов.

Если внешний инструмент вернул сообщение, приложение должно по возможности сохранить короткую очищенную версию. Полный вывод можно оставлять только во временной отладке разработчика, не как обязательную часть данных первой версии.

## Пользовательские сообщения об ошибках

Сообщение преподавателю должно отвечать на три вопроса:

- что не получилось;
- к какому студенту, проекту или обновлению это относится;
- что можно сделать дальше.

Примеры допустимого смысла сообщений:

- репозиторий недоступен, проверьте ссылку или доступ GitHub;
- локальная копия проекта содержит изменения, автоматическое обновление остановлено;
- Codex CLI не найден, проверьте путь в настройках;
- ИИ-анализ не запущен, потому что Codex CLI заранее отмечен как недоступный;
- ИИ-рапорт не сохранен, потому что файл данных недоступен;
- pull request не найден, анализ выполнен только по локальному Git-контексту.

Сообщения не должны обещать автоматическое исправление, публикацию комментариев или оценку студента.

## Тестовая стратегия

Перед началом разработки нужна минимальная тестовая пирамида, которая покрывает рискованные части первой версии.

Уровни тестов:

- модульные тесты для чистой логики;
- интеграционные тесты для локального хранилища и внешних команд через управляемые заглушки;
- сценарные тесты для ключевых пользовательских потоков;
- ручная локальная проверка перед использованием на реальных студенческих репозиториях.

Тесты не должны требовать реального OpenAI API, облачной базы данных или публикации данных в GitHub.

## Модульные тесты

Модульными тестами нужно покрыть:

- валидацию студента и GitHub-ссылки;
- вычисление статусов проекта для списка студентов;
- сортировку студентов по новым изменениям, ошибкам и имени;
- создание события обновления;
- определение режима ИИ-анализа по сохраненным коммитам;
- привязку комментария к обновлению;
- удаление комментария только после подтверждения;
- пересчет количества комментариев после удаления комментария;
- привязку ИИ-рапорта к обновлению;
- переходы статусов ИИ-запуска;
- нормализацию ошибки в пользовательское сообщение и диагностическое событие;
- проверку структуры результата Codex по схеме;
- запрет автоматического изменения статуса проверки после ИИ-запуска.

## Интеграционные тесты

Интеграционные тесты должны использовать временную папку данных и контролируемые локальные репозитории или заглушки команд.

Минимальные проверки:

- создание структуры `data/`;
- чтение и запись JSON-файлов;
- безопасная запись без потери существующего файла при ошибке;
- восстановление данных после перезапуска тестового приложения;
- клонирование или имитация клонирования репозитория;
- повторное обновление репозитория с новым коммитом;
- обработка репозитория без новых коммитов;
- сохранение ошибки Git без остановки остальных проектов;
- создание review-копии для выбранного обновления;
- ошибка создания review-копии;
- запуск Codex CLI через заглушку процесса;
- передача промпта через stdin в заглушку Codex;
- чтение файла `--output-last-message`;
- проверка результата по схеме;
- сохранение успешного ИИ-рапорта;
- сохранение неуспешного ИИ-рапорта;
- блокировка ИИ-анализа без создания ИИ-рапорта, если Codex CLI заранее известен как недоступный;
- получение pull request через заглушку GitHub CLI;
- продолжение ИИ-анализа при ошибке GitHub CLI.

Интеграционные тесты не должны выполнять реальные push, merge, публикацию комментариев или прямые сетевые вызовы к OpenAI API.

## Сценарные тесты

Сценарные тесты проверяют рабочие потоки преподавателя на уровне приложения.

Минимальные сценарии:

- создать студента без GitHub-ссылки и увидеть статус неподключенного проекта;
- добавить студенту GitHub-ссылку;
- запустить общее обновление и получить событие для каждого студента;
- увидеть ошибку одного проекта без остановки общего обновления;
- открыть страницу студента и увидеть ленту обновлений;
- открыть обновление с новыми коммитами;
- добавить комментарий к обновлению;
- удалить комментарий к обновлению после подтверждения;
- изменить статус проверки обновления вручную;
- открыть код выбранного обновления в VS Code через заглушку команды открытия;
- запустить ИИ-анализ выбранного обновления через заглушку Codex;
- увидеть сохраненный ИИ-рапорт внутри обновления;
- повторно запустить ИИ-анализ и увидеть два рапорта;
- получить ошибку ИИ-анализа и увидеть ее внутри обновления;
- увидеть отключенный запуск ИИ без нового ИИ-рапорта, если Codex CLI заранее недоступен;
- перезапустить приложение и увидеть сохраненную историю.

## Ручная локальная проверка

Перед первой работой с реальными студенческими проектами нужно выполнить ручную проверку на тестовых репозиториях.

Минимальный чек-лист:

- приложение стартует без существующей папки `data/`;
- приложение сохраняет созданного студента после перезапуска;
- общий запуск обновления обрабатывает студента без GitHub-ссылки;
- тестовый публичный репозиторий клонируется в `data/repositories/`;
- повторное обновление без новых коммитов сохраняет событие истории;
- тестовый новый коммит определяется как новое изменение;
- ошибка доступа к репозиторию видна в списке и истории;
- review-копия создается в `data/review-copies/`;
- открытие VS Code вызывает настроенную команду или показывает понятную ошибку;
- Codex CLI определяется настройками;
- тестовый ИИ-запуск создает ИИ-рапорт;
- заранее известная недоступность Codex CLI блокирует запуск без создания ИИ-рапорта;
- ошибка Codex CLI после начатой попытки сохраняется как неуспешный ИИ-рапорт;
- GitHub CLI без доступного pull request не блокирует ИИ-анализ;
- после перезапуска видны студенты, обновления, комментарии, статусы и ИИ-рапорты.

## Стартовые критерии готовности к разработке

Разработку первой рабочей версии можно начинать, когда техническая основа позволяет:

- создать локальную папку данных;
- записать и прочитать основные JSON-файлы;
- выполнить или заглушить внешнюю команду;
- сохранить диагностическое событие;
- создать событие обновления с ошибкой;
- создать успешный и неуспешный ИИ-рапорт;
- запустить базовый набор тестов локально без внешней облачной инфраструктуры.

## Что не входит в первую версию

В эту техническую спецификацию не входит:

- отправка диагностических данных во внешний сервис;
- облачный сбор логов;
- телеметрия;
- автоматическое исправление поврежденных репозиториев;
- автоматическое восстановление поврежденных JSON-файлов без решения преподавателя;
- прямой вызов OpenAI API;
- тесты против реального OpenAI API;
- поддержка альтернативных ИИ-провайдеров;
- автоматическая публикация комментариев в GitHub;
- автоматическая оценка студента;
- полноценная проверка CI/CD студенческих проектов.

## Критерии приемки

Техническая реализация соответствует этой спецификации, если:

- ошибки обновления сохраняются в истории и не останавливают остальные проекты;
- ошибки начатого ИИ-анализа сохраняются как неуспешные ИИ-рапорты;
- заранее заблокированные действия ИИ-анализа не создают ИИ-рапорты;
- ошибки GitHub CLI при pull request не блокируют основной локальный анализ;
- ошибки хранения не считаются успешными действиями;
- диагностический журнал сохраняет уровень, область, связанный объект и очищенное техническое описание;
- диагностика не сохраняет секреты;
- пользовательские сообщения объясняют причину и следующий шаг;
- тесты покрывают данные, Git-операции, review-копии, Codex-запуск и основные пользовательские сценарии;
- ручной чек-лист можно пройти на тестовых локальных данных без облачной базы данных и без прямого OpenAI API.
