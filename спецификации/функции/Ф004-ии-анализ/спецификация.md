# ИИ-анализ и черновики комментариев

## Цель

Описать, как преподаватель вручную запускает локальный ИИ-анализ выбранного обновления проекта, получает проверяемый ИИ-рапорт и при необходимости использует его как основу для собственного комментария.

## Почему это одна спецификация

ИИ-анализ и сохранение ИИ-артефактов не стоит разделять: ценность анализа появляется тогда, когда преподаватель может привязать результат к конкретному обновлению или оставить его как черновик.

ИИ не является отдельным проверяющим. Он помогает преподавателю понять изменения, найти важные файлы и подготовить черновик, но итоговый статус проверки и финальный комментарий остаются за преподавателем.

## Пользовательская задача

Преподаватель открывает конкретное обновление в ленте студента и хочет быстро понять:

- что изменилось в проекте после прошлого обновления;
- какие файлы стоит посмотреть в первую очередь;
- какие места выглядят слабыми, подозрительными или требующими ручной проверки;
- какие повторяющиеся проблемы или улучшения видны по сравнению с предыдущими обновлениями;
- какой черновик комментария можно использовать как основу для своей проверки.

После анализа преподаватель должен получить сохраненный ИИ-рапорт, который можно открыть позже вместе с тем обновлением, для которого он был создан.

## Основные объекты

### ИИ-запуск

ИИ-запуск - это попытка выполнить локальный анализ конкретного обновления.

ИИ-запуск должен хранить:

- дату и время начала;
- дату и время завершения, если запуск завершился;
- студента;
- проект;
- обновление, для которого запущен анализ;
- локальный путь проекта;
- ветку;
- старый коммит как начало диапазона анализа;
- новый коммит как конец диапазона анализа;
- статус запуска: выполняется, готов, ошибка;
- текст ошибки, если запуск не удался.

ИИ-запуск не должен существовать отдельно от обновления. В интерфейсе он отображается как ИИ-рапорт или ошибка ИИ-анализа внутри выбранного обновления.

### ИИ-рапорт

ИИ-рапорт - это результат успешного или неуспешного ИИ-запуска, сохраненный внутри выбранного обновления.

ИИ-рапорт должен хранить:

- дату и время запуска;
- дату и время завершения;
- статус: готов или ошибка;
- студента;
- проект;
- обновление;
- старый коммит;
- новый коммит;
- краткое резюме;
- список важных файлов, если ИИ смог их выделить;
- описание изменений;
- возможные проблемы или зоны риска;
- черновик комментария преподавателя;
- полный текст рапорта;
- технические детали запуска, если они нужны для диагностики;
- текст ошибки, если анализ не завершился успешно.

У одного обновления может быть несколько ИИ-рапортов. Повторный запуск не должен перезаписывать старый рапорт без явного действия преподавателя.

### Краткое описание проекта

Краткое описание проекта - это отдельный ИИ-артефакт, который показывается в верхнем контексте страницы студента в `Ф003`.

Описание относится к проекту в целом, а не к одному обновлению.

Описание должно хранить:

- краткое резюме проекта;
- основную пользовательскую или техническую идею;
- ключевые части приложения, если они известны;
- дату создания или обновления описания;
- коммит, на котором описание было создано;
- статус: отсутствует, создается, готово, ошибка.

В первой версии описание проекта создается или обновляется только после ручного запуска ИИ-анализа выбранного обновления. Автоматический запуск после клонирования или обновления проекта не нужен.

## Откуда запускается ИИ-анализ

ИИ-анализ запускается только из деталей конкретного обновления на странице студента, описанной в `Ф003`.

Минимальное поведение:

- действие запуска доступно, если у обновления есть локальный путь проекта и известная граница анализа из `Ф002`;
- если границы анализа нет, интерфейс объясняет, почему анализ нельзя запустить;
- во время запуска в выбранном обновлении показывается состояние выполнения;
- после успешного запуска внутри обновления появляется новый ИИ-рапорт;
- после ошибки внутри обновления сохраняется неуспешный ИИ-рапорт с причиной ошибки.

ИИ-анализ запускается только вручную. Общее обновление проектов из `Ф002` и одиночное обновление проекта из `Ф003` не должны автоматически запускать ИИ.

## Запуск локального Codex

В первой версии приложение запускает только локально установленный Codex CLI. Приложение не вызывает OpenAI API напрямую и не делает адаптеры для других ИИ-провайдеров.

Рекомендуемая модель запуска для первой версии:

- приложение создает review-копию или использует безопасную папку выбранного обновления;
- приложение формирует промпт анализа на основе данных обновления, Git-диапазона и контекста pull request;
- приложение запускает Codex как дочерний процесс через `codex exec`;
- рабочей папкой Codex должна быть папка review-копии или другая безопасная папка анализа, а не основная локальная копия для обновления `main`;
- Codex запускается в режиме чтения;
- результат сохраняется как ИИ-рапорт внутри выбранного обновления.

Базовая команда:

```bash
codex exec -C <путь-к-review-копии> -s read-only --output-schema <путь-к-json-schema> --output-last-message <путь-к-рапорту>
```

Правила:

- промпт лучше передавать через stdin, чтобы не зависеть от ограничений командной строки;
- `--output-schema` нужно использовать для стабильной структуры ИИ-рапорта, если техническая реализация поддерживает схему;
- `--output-last-message` можно использовать для сохранения финального ответа Codex в файл перед разбором и записью в хранилище приложения;
- `--json` можно использовать для технической диагностики или прогресса выполнения, но пользовательский результат должен сохраняться как ИИ-рапорт;
- приложение должно проверять доступность команды `codex` в настройках из `Ф005`;
- приложение не должно хранить API-ключи OpenAI и не должно управлять авторизацией Codex напрямую.

Если локальный Codex не установлен, не авторизован или завершился ошибкой, приложение сохраняет ошибку ИИ-анализа внутри выбранного обновления.

## Входные данные для ИИ

ИИ должен получать только данные, которые относятся к выбранному обновлению и нужны для анализа.

Минимальные входные данные:

- имя студента;
- дополнительная информация о студенте, если она есть;
- ссылка GitHub на момент обновления;
- локальный путь проекта;
- ветка;
- старый коммит;
- новый коммит;
- количество новых коммитов;
- дата и время обновления;
- предыдущие комментарии преподавателя по этому обновлению, если они уже есть;
- предыдущие ИИ-рапорты этого обновления, если преподаватель запускает анализ повторно;
- краткое описание проекта, если оно уже есть;
- контекст pull request, если он доступен.

Для чтения изменений ИИ может использовать локальную копию проекта и команды Git по сохраненной границе:

```bash
git log <старый-коммит>..<новый-коммит>
git diff <старый-коммит>..<новый-коммит>
git show <коммит>
```

## Контекст pull request

Так как студенты работают через pull request, контекст pull request полезен для ИИ-анализа и должен передаваться в Codex, если он доступен.

Контекст pull request помогает понять:

- как студент сам описывает изменения;
- какие файлы и коммиты входят в работу;
- какие вопросы уже обсуждались в комментариях;
- какие замечания были оставлены в review;
- что было изменено после обратной связи.

Минимальный контекст pull request:

- номер pull request;
- заголовок;
- описание;
- автор;
- исходная ветка;
- целевая ветка;
- состояние pull request;
- список коммитов pull request;
- список измененных файлов;
- комментарии обсуждения, если они доступны;
- review-комментарии, если они доступны.

Правила:

- контекст pull request получает `Ф004`, а не `Ф002`;
- если pull request найден, ИИ-рапорт должен учитывать его описание и обсуждение;
- если pull request недоступен, анализ продолжается по локальному Git-диапазону;
- если доступ к GitHub или pull request завершился ошибкой, ошибка контекста pull request сохраняется как диагностическая часть ИИ-рапорта, но не должна блокировать анализ diff;
- ИИ должен отличать факты из кода от утверждений в описании pull request;
- приложение не публикует комментарии в pull request автоматически.

Для первой версии достаточно получать контекст pull request через локально настроенный доступ к GitHub, например через GitHub CLI. Приложение не должно хранить GitHub-пароль или собственный GitHub-токен.

## Режим чтения

ИИ-интеграция должна быть осторожной.

Правила:

- ИИ может читать локальную копию проекта;
- ИИ может читать историю Git и diff между сохраненными коммитами;
- ИИ не должен изменять файлы проекта;
- ИИ не должен создавать коммиты;
- ИИ не должен делать push;
- ИИ не должен менять локальную рабочую копию основного репозитория;
- ИИ не должен отправлять комментарии студенту или публиковать их в GitHub;
- ИИ не должен выставлять итоговый статус проверки.

Если техническая реализация требует отдельной временной копии или worktree для безопасного чтения, это относится к `Ф005`.

## Содержание стандартного ИИ-рапорта

В первой версии перед запуском анализа не нужно выбирать режим или тип рапорта. Запуск создает стандартный ИИ-рапорт для выбранного обновления.

Стандартный ИИ-рапорт должен содержать:

- краткое резюме обновления;
- список новых коммитов или краткое описание диапазона;
- основные изменения в проекте;
- важные файлы для ручного просмотра;
- возможные проблемы, слабые места или подозрительные решения;
- вопросы, которые преподавателю стоит проверить вручную;
- черновик комментария преподавателя;
- отметку, что выводы ИИ требуют проверки преподавателем.

ИИ может показать файлы, на которые стоит обратить внимание в текущем анализе, но проверка не обязана хранить отдельный список фактически просмотренных файлов.

## Черновик комментария

Черновик комментария - это часть ИИ-рапорта, а не финальный комментарий преподавателя.

Правила:

- черновик сохраняется внутри ИИ-рапорта;
- черновик не публикуется студенту автоматически;
- черновик не становится ручным комментарием автоматически;
- преподаватель может использовать текст черновика как основу для собственного комментария в `Ф003`;
- если преподаватель создает ручной комментарий на основе черновика, итоговый текст комментария хранится как комментарий преподавателя внутри выбранного обновления.

## Связь с обновлением и коммитами

Каждый сохраненный ИИ-анализ должен быть привязан к конкретному состоянию проекта.

Минимальная связь:

- студент;
- проект;
- обновление;
- старый коммит;
- новый коммит;
- дата и время фиксации границы анализа;
- локальный путь проекта на момент анализа.

Это нужно, чтобы позже было понятно, какую именно версию проекта анализировал ИИ и почему рапорт относится именно к этому обновлению.

## Ошибки ИИ-анализа

Ошибка ИИ-анализа не должна ломать страницу студента и не должна менять статус проверки обновления.

Ошибка должна сохраняться внутри выбранного обновления как неуспешный ИИ-рапорт.

Нужно сохранять ошибку, если:

- локальный путь проекта отсутствует;
- локальный репозиторий недоступен или поврежден;
- старый или новый коммит отсутствует;
- Git-команда для анализа завершилась ошибкой;
- локальный Codex CLI недоступен;
- запуск ИИ завершился ошибкой;
- результат ИИ не удалось сохранить.

Ошибки должны быть видны:

- в деталях выбранного обновления;
- в карточке обновления в ленте через признак ошибки ИИ;
- в диагностике, если преподаватель хочет понять, почему анализ не выполнен.

## Что не входит сюда

- ручной комментарий преподавателя как основной источник решения;
- изменение файлов проекта ИИ-помощником;
- автоматическая отправка комментариев студенту;
- автоматическая публикация комментариев в GitHub;
- автоматическое выставление оценки или статуса проверки;
- сложный выбор режима анализа перед запуском;
- полноценная проверка CI/CD студенческого проекта;
- поддержка альтернативных ИИ-провайдеров или адаптеров;
- формат локального хранения ИИ-рапортов.

## Критерии приемки

Первая версия этой функции считается готовой, если преподаватель может:

- открыть конкретное обновление и вручную запустить ИИ-анализ;
- получить новый ИИ-рапорт, сохраненный внутри выбранного обновления;
- увидеть, на каких коммитах был создан ИИ-рапорт;
- увидеть краткое резюме изменений;
- увидеть важные файлы для ручного просмотра;
- увидеть возможные проблемы или зоны риска;
- получить черновик комментария;
- использовать черновик как основу для ручного комментария без автоматической публикации;
- увидеть контекст pull request в ИИ-рапорте, если он доступен;
- повторно запустить ИИ-анализ и сохранить новый рапорт без потери старого;
- увидеть ошибку ИИ-анализа внутри выбранного обновления;
- вернуться к сохраненному ИИ-рапорту после перезапуска приложения.

## Открытые вопросы

На текущий момент открытых вопросов для первой версии `Ф004` нет.

## Решения для первой версии

- ИИ-анализ запускается только вручную преподавателем.
- ИИ-анализ запускается только для конкретного обновления.
- Приложение запускает локальный Codex CLI через `codex exec`.
- Приложение не вызывает OpenAI API напрямую.
- Приложение не делает адаптеры для других ИИ-провайдеров.
- ИИ работает в режиме чтения.
- ИИ не изменяет файлы проекта, не делает коммиты и не делает push.
- ИИ-рапорт сохраняется внутри выбранного обновления.
- У одного обновления может быть несколько ИИ-рапортов.
- Стандартный рапорт создается без выбора режима анализа.
- Контекст pull request передается в ИИ-анализ, если он доступен.
- Черновик комментария является частью ИИ-рапорта и не становится финальным комментарием автоматически.
- Итоговый статус проверки остается ручным решением преподавателя.
