# Обновление и история изменений

## Цель

Описать, как преподаватель одной кнопкой обновляет проекты студентов и как приложение сохраняет результат каждого обновления.

## Почему это одна спецификация

Обновление, локальный репозиторий, сравнение коммитов, ошибки Git и история обновлений являются частями одного процесса. Их лучше описывать вместе, потому что каждое обновление сразу создает запись в истории.

## Пользовательская задача

Преподаватель должен нажать одну кнопку и получить понятный результат:

- какие проекты обновились;
- у кого появились новые изменения;
- сколько новых коммитов появилось;
- у кого изменений нет;
- какие проекты нельзя было обновить;
- какие студенты пока не имеют GitHub-ссылки.

После обновления преподаватель должен сразу понимать, кого проверять первым.

## Основные объекты

### Запуск обновления

Запуск обновления - это одно действие преподавателя, которое проходит по списку студентов и пытается обновить проекты, у которых есть ссылка GitHub.

Запуск обновления является сводным объектом. Он группирует отдельные события обновления проектов, но не заменяет их.

Запуск должен хранить:

- дату и время начала;
- дату и время завершения;
- количество студентов в списке;
- количество проектов, которые пытались обновить;
- количество проектов с новыми изменениями;
- общее количество новых коммитов;
- количество проектов без новых изменений;
- количество ошибок;
- количество студентов без подключенного репозитория.

### Событие обновления проекта

Событие обновления проекта - это отдельный объект обновления, привязанный к конкретному студенту и проекту. Он хранит результат обновления одного проекта или то, что проект был недоступен на момент обновления.

Именно эти объекты обновления показываются в ленте обновлений студента в `Ф003`.

Для каждого события нужно хранить:

- дату и время;
- студента;
- проект;
- ссылку GitHub на момент обновления, если она была указана;
- локальный путь проекта, если он есть;
- ветку, которая обновлялась;
- старый коммит как начало диапазона, если он был известен;
- новый коммит как конец диапазона, если он был получен;
- количество новых коммитов между старым и новым коммитом;
- были ли новые изменения;
- результат обновления;
- текст ошибки или причина недоступности, если обновление не удалось.

Даже если новых изменений нет, событие обновления полезно и должно сохраняться.

Если у студента нет ссылки GitHub, это тоже фиксируется как событие истории: на момент обновления проект был недоступен, потому что репозиторий не указан.

## Главный сценарий: обновить все проекты

1. Преподаватель открывает список студентов.
2. Нажимает действие обновления всех проектов.
3. Приложение проходит по всем студентам.
4. Для каждого студента приложение создает отдельный объект обновления, который будет привязан к его проекту и показан в ленте студента.
5. Для студентов без ссылки GitHub приложение не запускает Git-команды, но сохраняет объект обновления о недоступном проекте.
6. Для студентов со ссылкой GitHub приложение проверяет локальную копию проекта.
7. Если локальной копии нет, приложение клонирует репозиторий.
8. Если локальная копия есть, приложение обновляет ее через Git.
9. Приложение получает последнюю версию ветки `main`.
10. Приложение сравнивает предыдущий известный коммит с последним коммитом ветки `main`.
11. Приложение считает количество новых коммитов.
12. Приложение сохраняет границы анализа: локальный путь проекта, ветку `main`, старый коммит и новый коммит.
13. Приложение сохраняет результат в объекте обновления каждого студента.
14. После завершения показывает итоговую сводку.

Ожидаемый результат:

- преподаватель видит, у кого появились новые изменения;
- преподаватель видит, сколько новых коммитов появилось;
- преподаватель видит, у кого изменений нет;
- преподаватель видит ошибки обновления;
- преподаватель видит студентов без GitHub-ссылки;
- статусы в списке студентов обновлены;
- история обновлений сохранена;
- на странице каждого студента в `Ф003` появляется новая запись ленты, созданная этим общим запуском.

## Сценарий: первая загрузка проекта

Если у студента есть ссылка GitHub, но локальной копии проекта еще нет:

- приложение клонирует репозиторий в локальную папку;
- приложение настраивает локальную копию на ветку `main`;
- сохраняет локальный путь проекта;
- определяет последний коммит ветки `main`;
- сохраняет текущий коммит `main` как новый известный коммит;
- создает событие обновления.

Для первой загрузки проект считается обновленным, потому что приложение впервые получило состояние проекта.

## Сценарий: повторное обновление проекта

Если локальная копия уже существует:

- приложение получает последние изменения из удаленного репозитория;
- приложение обновляет локальную копию до актуального состояния ветки `main`;
- определяет новый коммит ветки `main`;
- сравнивает его с предыдущим известным коммитом;
- считает количество новых коммитов;
- сохраняет границы анализа для `Ф004`;
- сохраняет результат.

Если коммит изменился, у проекта появляются новые изменения.

Если коммит не изменился, событие все равно сохраняется как успешное обновление без новых изменений.

## Git-стратегия первой версии

Цель обновления - получить последнюю версию ветки `main`.

Для нового проекта приложение клонирует репозиторий и переходит на `main`.

Для существующего проекта приложение использует безопасную стратегию:

- получить актуальное состояние удаленного репозитория;
- проверить состояние `origin/main`;
- обновить локальную рабочую копию до `origin/main`;
- не создавать коммиты;
- не делать push;
- не изменять файлы проекта вручную.

Так как приложение никогда не изменяет файлы в локальных копиях студенческих репозиториев, в нормальном сценарии локальных незакоммиченных изменений быть не должно.

Если локальные изменения все-таки обнаружены, это считается ошибочным состоянием локальной копии. Приложение не должно пытаться автоматически исправлять это без явного решения преподавателя.

## Команды Git первой версии

Для нового проекта:

```bash
git clone --branch main --single-branch <github-url> <local-path>
```

После клонирования:

```bash
git rev-parse HEAD
```

Для уже склонированного проекта:

```bash
git status --porcelain
git fetch origin main --prune
git rev-parse origin/main
git rev-list --count <last-known-commit>..origin/main
git reset --hard origin/main
```

Правила:

- `git status --porcelain` используется перед обновлением, чтобы убедиться, что локальная копия чистая;
- если локальная копия не чистая, обновление этого проекта останавливается и сохраняется ошибка;
- `git fetch origin main --prune` получает последние данные ветки `main`;
- `git rev-parse origin/main` дает новый коммит ветки `main`;
- `git rev-list --count` дает количество новых коммитов;
- `git reset --hard origin/main` приводит локальную копию к последней версии `main`, но только после проверки, что локальных изменений нет.

Приложение не использует `commit` и `push`.

`Ф002` не обязан получать подробный список измененных файлов, содержимое diff, описание pull request или комментарии pull request. Эти данные может получить `Ф004`, когда ИИ будет анализировать зафиксированную границу `старый коммит .. новый коммит`.

## Студенты без GitHub-ссылки

Студента можно создать без ссылки GitHub. Такие студенты не должны ломать общий запуск обновления.

Поведение:

- студент без ссылки GitHub не обновляется через Git;
- приложение не пытается клонировать или обновлять проект;
- в списке студентов показывается статус "проект не подключен";
- в итоговой сводке такие студенты считаются отдельно;
- в истории сохраняется событие о том, что на момент обновления проект был недоступен, потому что ссылка GitHub не указана.

Это событие не считается ошибкой Git.

## Приватные репозитории

Приватные GitHub-репозитории поддерживаются, если у преподавателя есть доступ.

Правила:

- если преподаватель приглашен в приватный репозиторий и локальная Git-аутентификация настроена, проект обновляется как обычный репозиторий;
- если преподаватель не приглашен, потерял доступ или аутентификация не работает, приложение сохраняет ошибку доступа;
- отсутствие доступа к одному приватному репозиторию не должно останавливать обновление остальных проектов.

Приложение не должно пытаться обходить права доступа GitHub.

## Граница анализа для ИИ

`Ф002` не выполняет подробный анализ изменений. Его задача - зафиксировать надежную и воспроизводимую границу, которую потом использует `Ф004`.

Для `Ф004` нужно сохранить:

- локальный путь проекта;
- ветку `main`;
- старый коммит как начало диапазона;
- новый коммит как конец диапазона;
- количество новых коммитов;
- дату и время фиксации диапазона;
- ссылку GitHub на репозиторий.

Этого достаточно, чтобы ИИ в `Ф004` самостоятельно выполнил команды вроде:

```bash
git log <старый-коммит>..<новый-коммит>
git diff <старый-коммит>..<новый-коммит>
git show <коммит>
```

Если в `Ф004` понадобится контекст pull request, его нужно получать там, а не в `Ф002`.

## Статусы после обновления

`Ф002` вычисляет статусы проекта, а `Ф001` показывает их в списке.

Минимальные статусы:

- проект не подключен;
- проект еще не обновлялся;
- обновление выполняется;
- новых изменений нет;
- есть новые изменения;
- обновление завершилось ошибкой;
- локальная копия проекта отсутствует;
- локальная копия проекта повреждена или недоступна;
- ссылка GitHub недоступна или требует исправления.

## Ошибки

Ошибка одного проекта не должна останавливать обновление остальных проектов.

Для проекта нужно сохранить ошибку, если:

- ссылка GitHub некорректная;
- репозиторий недоступен;
- репозиторий приватный и у преподавателя нет доступа;
- отсутствует интернет;
- команда Git завершилась ошибкой;
- локальная папка проекта отсутствует, хотя путь уже был сохранен;
- локальная папка существует, но не является ожидаемым репозиторием;
- локальная копия содержит состояние, которое мешает обновлению.

Ошибки должны быть видны:

- в итоговой сводке запуска обновления;
- в списке студентов;
- в истории обновлений конкретного студента.

## История и неизменность

История обновлений нужна, чтобы преподаватель мог понять, что происходило с проектом в течение курса.

Правила:

- успешное обновление сохраняется в истории;
- обновление без новых изменений сохраняется в истории;
- событие недоступности проекта без GitHub-ссылки сохраняется в истории;
- ошибка обновления сохраняется в истории;
- старые события обновлений не перезаписываются новым обновлением;
- последний статус проекта может обновляться, но история событий должна сохраняться.

## Итоговая сводка

После общего обновления приложение показывает сводку:

- сколько проектов обновлено успешно;
- сколько проектов имеют новые изменения;
- сколько новых коммитов найдено;
- сколько проектов без новых изменений;
- сколько проектов завершились ошибкой;
- сколько студентов имеют проект без ссылки GitHub.

Сводка должна помогать преподавателю сразу перейти к студентам с новыми изменениями или ошибками.

## Что не входит сюда

- создание и редактирование карточки студента;
- дизайн списка студентов;
- ручная проверка проекта преподавателем;
- комментарии и заметки;
- ИИ-анализ изменений;
- подробное отображение списка измененных файлов сразу после обновления;
- получение описаний и комментариев pull request;
- окончательный выбор формата локального хранения.

## Критерии приемки

Первая версия этой функции считается готовой, если преподаватель может:

- запустить обновление всех проектов одной кнопкой;
- получить событие истории для студента без GitHub-ссылки;
- обновить студентов, у которых есть ссылка GitHub;
- не блокировать весь запуск из-за одной ошибки;
- впервые склонировать проект студента;
- повторно обновить уже склонированный проект;
- получить последнюю версию ветки `main`;
- увидеть, у кого появились новые изменения;
- увидеть количество новых коммитов;
- сохранить локальный путь, ветку, старый коммит и новый коммит как границу анализа для `Ф004`;
- увидеть, у кого изменений нет;
- увидеть ошибки обновления;
- увидеть итоговую сводку;
- открыть историю обновлений студента после перезапуска приложения.

## Открытые вопросы

На текущий момент открытых вопросов по `Ф002` нет.

## Решения для первой версии

- Обновление всех проектов запускается вручную преподавателем.
- Студенты без ссылки GitHub не обновляются через Git, но получают событие истории о недоступном проекте.
- Ошибка одного проекта не останавливает обновление остальных.
- Каждая попытка обновить проект с GitHub-ссылкой должна оставить запись в истории.
- Сравнение изменений выполняется между последним известным коммитом и последним коммитом ветки `main`.
- Основная ветка первой версии - `main`.
- Для каждого успешного обновления с изменениями сохраняется количество новых коммитов.
- Для `Ф004` сохраняется только надежная граница анализа: локальный путь, ветка, старый коммит и новый коммит.
- Подробные diff, файлы, описания pull request и комментарии pull request получает `Ф004`, если они нужны для ИИ-анализа.
- Подробный список изменений не показывается сразу после обновления.
- Приватные репозитории поддерживаются только при наличии доступа преподавателя.
- Если доступа к приватному репозиторию нет, это сохраняется как ошибка.
- Локальные копии студенческих репозиториев считаются read-only для приложения.
- Приложение не отправляет ничего в репозитории студентов.
