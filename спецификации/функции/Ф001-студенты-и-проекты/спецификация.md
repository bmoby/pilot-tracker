# Студенты и проекты

## Цель

Описать, как преподаватель ведет список студентов, связывает каждого студента с проектом и видит текущее состояние этих проектов.

## Почему это одна спецификация

Студент, проект и обзор статусов не стоит разделять: преподаватель работает с ними как с одним списком. Отдельный "обзор студентов" без правил студента и проекта не имеет самостоятельного смысла.

## Пользовательская задача

Преподаватель должен иметь одно место, где он видит всех студентов курса и понимает:

- кто есть в списке;
- у какого студента какой GitHub-проект;
- какой проект уже был загружен локально;
- у кого есть новые изменения после последнего обновления;
- кого нужно открыть для проверки.

В первой версии этот список является текущей рабочей когортой преподавателя. Приложение не должно поддерживать несколько отдельных курсов, семестров или архивов групп.

## Основные объекты

### Студент

Студент - это запись, которую создает и ведет преподаватель.

Минимальные поля первой версии:

- имя студента;
- дополнительная информация в свободной форме;
- ссылка на основной GitHub-репозиторий, если она уже известна;
- связанный проект;
- последний известный статус проекта;
- дата последнего обновления проекта, если обновление уже выполнялось.

Поля, которые можно добавить позже:

- группа;
- email;
- уровень;
- теги;
- аватар;
- ссылка на демо;
- несколько проектов на одного студента.

### Проект

В первой версии у одного студента есть один основной проект.

Проект хранит:

- ссылку на GitHub-репозиторий;
- локальный путь к копии проекта, если проект уже был склонирован;
- текущую ветку, если она известна;
- последний известный коммит, если проект уже обновлялся;
- дату последнего обновления;
- результат последнего обновления.

Проект не является отдельной независимой сущностью для преподавателя в первой версии. Он всегда показывается через студента.

## Главный сценарий: добавить студента

1. Преподаватель открывает список студентов.
2. Нажимает действие добавления студента.
3. Вводит имя студента.
4. Вводит дополнительную информацию, если она нужна.
5. Вводит ссылку на GitHub-репозиторий проекта, если она уже есть.
6. Сохраняет запись.

Ожидаемый результат:

- студент появляется в общем списке;
- если ссылка GitHub указана, приложение показывает, что проект еще не обновлялся;
- если ссылка GitHub не указана, приложение показывает, что проект пока не подключен;
- проект участвует в общем обновлении только после добавления ссылки на репозиторий.

## Сценарий: изменить студента

Преподаватель может изменить:

- имя студента;
- дополнительную информацию;
- ссылку на GitHub-репозиторий.

Если ссылка на репозиторий изменилась, приложение должно явно показать, что это влияет на связанный проект и будущие обновления. Подробное поведение локальной копии описывается в `Ф002` и `Ф005`.

Правило первой версии для изменения ссылки:

- если у проекта еще нет успешной локальной загрузки, сохраненного локального пути и последнего известного коммита, ссылку GitHub можно изменить через карточку студента;
- старые события ошибок или события отсутствующей ссылки сохраняются в истории как факты прошлого состояния;
- если у проекта уже есть успешная локальная загрузка, локальный путь или последний известный коммит, замена GitHub-ссылки на другой репозиторий в первой версии не выполняется;
- в таком случае приложение должно объяснить, что замена уже загруженного репозитория требует отдельного решения спецификации, потому что может нарушить историю обновлений, review-копии и ИИ-рапорты;
- приложение не должно молча очищать локальный путь, последний известный коммит, историю обновлений или review-копии при изменении карточки студента.

## Сценарий: открыть студента для проверки

Из списка студентов преподаватель может открыть конкретного студента.

Ожидаемый результат:

- открывается пространство проверки проекта;
- туда передается выбранный студент и его связанный проект;
- подробная проверка, комментарии и заметки описываются в `Ф003`.

## Список студентов

Список студентов должен быть главным рабочим экраном первой версии.

Список студентов не является только справочником. Он должен работать как экран выбора следующей проверки: преподаватель должен быстро увидеть, где появились новые изменения, где есть ошибки обновления и какие проекты требуют внимания.

В списке нужно показывать:

- имя студента;
- краткую дополнительную информацию, если она есть;
- ссылку на GitHub как кликабельное действие или признак отсутствующей ссылки;
- последний статус проекта;
- дату последнего обновления;
- признак первой успешной загрузки, если проект только впервые появился в истории;
- признак новых изменений, если они были найдены;
- ошибку последнего обновления, если обновление не удалось.

Карточка студента в общем списке должна оставаться компактной и показывать только те сведения, которые помогают выбрать следующую проверку или исправить очевидную проблему.

В компактной карточке общего списка не нужно показывать:

- внутренний идентификатор студента;
- локальный путь проекта;
- последний известный коммит или хеш коммита;
- другие технические детали, которые преподаватель не использует при выборе следующей проверки.

Локальный путь проекта и последний известный коммит остаются частью данных проекта и могут показываться на странице студента, в деталях обновления или в диагностическом контексте, где они действительно помогают понять состояние проекта.

Базовая сортировка:

- сначала студенты с первой успешной загрузкой или новыми изменениями;
- затем студенты с ошибками обновления;
- затем остальные студенты.

Внутри каждой группы можно сортировать по имени.

Поиск, теги, группы и сложные фильтры в первой версии не нужны.

## Статусы проекта в списке

Статусы в `Ф001` только отображаются. Правила их вычисления описываются в `Ф002`.

Минимальные статусы:

- проект не подключен, потому что ссылка GitHub еще не указана;
- проект еще не обновлялся;
- обновление выполняется;
- проект впервые загружен и требует первой проверки;
- новых изменений нет;
- есть новые изменения;
- обновление завершилось ошибкой;
- локальная копия проекта отсутствует;
- локальная копия проекта повреждена или недоступна;
- ссылка на репозиторий отсутствует или требует исправления.

## Валидация

При создании или редактировании студента:

- имя студента обязательно;
- имя не должно быть пустой строкой;
- ссылка GitHub не обязательна;
- если ссылка GitHub указана, она должна выглядеть как ссылка на репозиторий;
- приложение не должно терять уже сохраненную историю студента при изменении карточки.

Проверка доступности репозитория не относится к этой спецификации. Она выполняется во время обновления проектов и описывается в `Ф002`.

## Удаление студента

Преподаватель может удалить студента, если запись больше не нужна.

Базовое правило для первой версии:

- удаление студента доступно из карточки или списка студентов;
- перед удалением приложение должно показать подтверждение;
- после подтверждения студент удаляется из списка;
- если у студента уже есть история обновлений, проверок или заметок, приложение должно предупредить, что эти данные тоже будут удалены.

В первой версии не нужен отдельный режим архивирования.

## Что не входит сюда

- фактическое клонирование и обновление репозитория;
- сравнение коммитов;
- история событий обновления;
- ручные комментарии и заметки преподавателя;
- ИИ-анализ проекта;
- выбор формата локального хранения.

## Критерии приемки

Первая версия этой функции считается готовой, если преподаватель может:

- создать студента с именем и дополнительной информацией без ссылки GitHub;
- добавить ссылку GitHub к студенту позже;
- увидеть студента в общем списке;
- изменить данные студента;
- увидеть кликабельную ссылку GitHub в списке, если она указана;
- увидеть у студента последний известный статус проекта;
- понять из списка, у кого проект впервые загружен и требует первой проверки;
- понять из списка, у кого есть новые изменения;
- использовать общий список без перегрузки техническими деталями вроде локального пути, внутреннего идентификатора или хеша коммита;
- открыть студента для дальнейшей проверки;
- удалить студента после явного подтверждения.

## Открытые вопросы

На текущий момент открытых вопросов по `Ф001` нет.

## Связанные спецификации

- Ф002: обновление и история изменений;
- Ф003: проверка проекта и заметки преподавателя;
- Ф005: локальные данные и настройки.

## Решения для первой версии

- Один студент связан с одним основным проектом.
- Проект показывается как часть карточки студента, а не как отдельный раздел приложения.
- Студента можно создать без ссылки GitHub и добавить ссылку позже.
- Список студентов является главным экраном для выбора следующей проверки.
- Список студентов представляет одну активную рабочую когорту преподавателя.
- Статусы проекта в списке берутся из последнего результата обновления.
- Первая успешная загрузка проекта показывается отдельно от повторного обновления без новых изменений.
- Локальный путь проекта не показывается в компактной карточке общего списка; он доступен только в более подробном контексте студента, обновления или диагностики.
- Последний известный коммит не показывается в компактной карточке общего списка; для выбора следующей проверки достаточно статуса и признака новых изменений.
- Несколько курсов, группы, теги, поиск, архивы когорт и сложные фильтры в первой версии не требуются.
- Студента можно удалить после явного подтверждения.
- Архивирование студентов в первой версии не требуется.
