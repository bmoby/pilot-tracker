# Проверка проекта и заметки преподавателя

## Цель

Описать страницу конкретного студента как ленту обновлений проекта, где каждое обновление имеет дату, статус, коммиты, ИИ-рапорты, комментарии преподавателя и возможность открыть код именно на версии этого обновления.

## Почему это одна спецификация

Проверка проекта строится вокруг обновлений.

Каждый раз, когда преподаватель запускает обновление проекта студента, приложение создает новое событие в ленте. Это событие становится контейнером для всего, что относится к этой проверке:

- граница коммитов;
- краткое описание результата обновления;
- один или несколько ИИ-рапортов;
- комментарии преподавателя;
- статус проверки;
- возможность открыть код на версии этого обновления.

Так преподаватель видит не разрозненные заметки, а историю проекта по датам.

## Пользовательская задача

Преподаватель открывает страницу студента и должен сразу увидеть ленту обновлений:

- когда проект обновлялся;
- сколько новых коммитов было найдено;
- был ли запущен ИИ-анализ;
- сколько ИИ-рапортов сохранено;
- сколько комментариев оставлено;
- какой статус проверки у каждого обновления;
- какие обновления еще требуют внимания.

Когда преподаватель открывает конкретное обновление, он должен иметь возможность:

- запустить ИИ-анализ именно для этого обновления;
- увидеть сохраненные ИИ-рапорты этого обновления;
- открыть проект в VS Code именно на версии кода этого обновления;
- оставить комментарий именно к этому обновлению;
- изменить статус проверки этого обновления.

## Основная модель страницы

Страница студента состоит из двух основных зон:

- верхний контекст студента и проекта;
- лента обновлений по датам.

Лента обновлений является главным рабочим объектом страницы.

Одно обновление - это событие истории из `Ф002`, расширенное данными проверки из `Ф003` и ИИ-результатами из `Ф004`.

Обновление должно оставаться в истории неизменным как факт: когда оно произошло, какой был результат, какие коммиты были границей. Комментарии, ИИ-рапорты и статус проверки могут добавляться позже внутрь этого обновления.

## Что видно в верхнем контексте

В верхней части страницы нужно показывать:

- имя студента;
- дополнительную информацию о студенте из `Ф001`;
- ссылку на GitHub-репозиторий;
- основной локальный путь проекта, если проект уже загружен;
- текущий статус проекта из `Ф001` и `Ф002`;
- последний известный коммит;
- дату последнего обновления;
- краткое описание проекта, созданное ИИ;
- действие обновления только этого проекта.

Верхний контекст нужен для ориентации. Основная работа происходит ниже, в ленте обновлений.

## Краткое описание проекта

На странице студента должно быть краткое описание проекта.

Описание создается ИИ в `Ф004`. `Ф003` не определяет, как ИИ читает проект и формирует описание, но определяет, где это описание показывается.

Описание относится к проекту студента в целом, а не к отдельному обновлению.

Минимально описание должно показывать:

- краткое резюме проекта;
- основную пользовательскую или техническую идею;
- ключевые части приложения, если они известны;
- дату и коммит, на которых описание было создано;
- статус описания: отсутствует, создается, готово, ошибка.

Если проект впервые успешно загружен локально, страница должна показать состояние краткого описания проекта, но не должна автоматически запускать его создание.

В первой версии создание описания запускается только вручную через ИИ-анализ выбранного обновления. Если описание отсутствует, это не блокирует ленту обновлений.

## Лента обновлений

Страница должна показывать большой список всех обновлений проекта студента.

Каждый объект обновления, созданный для проекта студента, появляется новой записью в ленте, даже если новых коммитов нет или обновление завершилось ошибкой.

Источник записи может быть любым:

- общий запуск "обновить все проекты";
- одиночный запуск "обновить проект" со страницы этого студента.

В ленту должны попадать:

- первая загрузка проекта;
- успешные обновления с новыми изменениями;
- успешные обновления без новых изменений;
- неудачные попытки обновления;
- события, когда проект нельзя было обновить из-за отсутствия GitHub-ссылки.

По умолчанию самые новые обновления показываются сверху. При этом у каждого события должны быть понятные дата, время и порядок.

## Карточка обновления в ленте

Каждая запись в ленте должна кратко показывать:

- дату и время обновления;
- результат обновления;
- статус проверки;
- количество новых коммитов;
- старый и новый коммит, если они известны;
- краткое описание результата из `Ф002`;
- краткое ИИ-резюме, если оно уже есть;
- был ли запущен ИИ-анализ;
- количество сохраненных ИИ-рапортов;
- количество комментариев преподавателя;
- наличие ошибки обновления или ИИ-анализа.

Карточка обновления должна помогать преподавателю быстро решить, что открыть дальше.

Карточка обновления в закрытом состоянии должна оставаться компактной. Она не должна превращаться в узкое место для чтения длинных ИИ-рапортов, комментариев и технической справки.

Открытие обновления должно раскрывать рабочую область проверки этого обновления. Рабочая область может находиться внутри страницы студента сразу под краткой шапкой выбранной карточки, но должна использовать доступную ширину ленты, а не отображаться как маленькое боковое окно внутри карточки.

## Детали выбранного обновления

Преподаватель должен иметь возможность открыть конкретное обновление из ленты.

Детали обновления являются рабочим пространством этого обновления.

Это рабочее пространство должно быть организовано так, чтобы преподаватель быстро отделял:

- краткий контекст обновления;
- действия проверки и текущий статус;
- ИИ-рапорты;
- комментарии преподавателя;
- техническую справку для углубления.

На широком экране детали выбранного обновления должны лучше использовать доступное место: основной ИИ-рапорт и комментарии могут быть разнесены по отдельным колонкам или крупным блокам. На узком экране эти зоны могут идти друг за другом, но должны сохранять понятный порядок и не смешиваться в одну неразличимую длинную колонку.

В деталях выбранного обновления нужно показывать:

- дату и время обновления;
- результат обновления;
- ветку;
- старый коммит;
- новый коммит;
- количество новых коммитов;
- ошибку обновления, если она была;
- список ИИ-рапортов этого обновления;
- состояние текущего ИИ-запуска, если анализ выполняется;
- все комментарии преподавателя, привязанные к этому обновлению;
- текущий статус проверки этого обновления;
- действие запуска ИИ-анализа;
- действие открытия кода в VS Code на версии этого обновления.

Если обновление завершилось ошибкой, его все равно можно открыть. Внутри можно увидеть ошибку и добавить комментарий к этой попытке.

Если для обновления нет новых коммитов, его тоже можно открыть. Внутри должно быть понятно, что изменений между старым и новым состоянием не было.

ИИ-анализ для повторного обновления без новых коммитов недоступен, потому что в таком событии нечего анализировать как новую работу студента. Преподаватель может открыть старые ИИ-рапорты предыдущих обновлений или выбрать обновление, где была первая загрузка проекта либо появились новые коммиты.

## ИИ-рапорты внутри обновления

ИИ-анализ запускается в контексте конкретного обновления.

Результат ИИ-анализа сохраняется внутри этого обновления как ИИ-рапорт.

У одного обновления может быть один или несколько ИИ-рапортов. Это нужно, чтобы преподаватель мог повторно запустить анализ, если первый результат был неполным, устарел после изменения правил анализа или нужен другой фокус проверки.

Каждый ИИ-рапорт должен хранить:

- дату и время запуска;
- статус: выполняется, готов, ошибка;
- коммит или диапазон коммитов, на котором он был создан;
- краткое резюме человеческим языком;
- что добавлено как функциональность или видимое поведение проекта;
- что преподавателю можно быстро попробовать или проверить руками;
- вопросы студенту;
- черновик комментария преподавателя;
- полный текст рапорта или нормализованное представление результата;
- техническую справку, если она нужна для проверяемости результата;
- ошибку, если анализ не завершился успешно.

В деталях обновления нужно показывать список всех рапортов этого обновления. Последний успешный рапорт может быть показан первым или раскрыт по умолчанию, но старые рапорты не должны исчезать без явного решения.

Если у обновления есть несколько ИИ-рапортов, интерфейс должен визуально выделять последний релевантный рапорт как основной материал для текущей проверки. Более старые рапорты должны оставаться доступными как история, но не должны раскрытыми блоками удлинять основную рабочую область без необходимости.

В первой версии перед запуском ИИ-анализа не нужно выбирать режим или тип рапорта. Запуск создает стандартный ИИ-рапорт для выбранного обновления.

`Ф004` описывает, как именно ИИ читает проект, строит анализ и формирует рапорт. `Ф003` определяет, что запуск происходит из выбранного обновления и результат сохраняется внутри него.

## Запуск ИИ-анализа для обновления

Преподаватель может запустить ИИ-анализ только из деталей конкретного обновления.

Минимальное поведение:

- если у обновления есть локальный путь проекта, известный новый коммит и это первая загрузка проекта, действие запуска ИИ доступно;
- если у обновления есть локальный путь проекта, старый коммит, новый коммит и количество новых коммитов больше нуля, действие запуска ИИ доступно;
- если повторное обновление прошло без новых коммитов, действие запуска ИИ недоступно и новый ИИ-рапорт не создается;
- для первой загрузки ИИ-анализ фокусируется на понятном описании текущего состояния проекта: что это за проект, что уже можно увидеть или попробовать, какие основные части есть и что преподавателю стоит проверить руками;
- для обновления с новыми коммитами ИИ-анализ фокусируется на том, что студент добавил как функциональность, что именно было сделано в проекте и что преподавателю стоит быстро проверить вручную;
- если приложение заранее знает, что Codex CLI недоступен, действие запуска ИИ отключено и новый ИИ-рапорт не создается;
- если нет локального пути проекта или нового коммита, страница объясняет, почему ИИ-анализ нельзя запустить, и новый ИИ-рапорт не создается;
- во время анализа у выбранного обновления показывается состояние выполнения;
- после успешного анализа в этом обновлении появляется новый ИИ-рапорт;
- если анализ создал или обновил краткое описание проекта, оно обновляется в верхней части страницы;
- если начатый анализ завершился ошибкой, ошибка сохраняется как неуспешный ИИ-рапорт внутри этого обновления.

ИИ-анализ и краткое описание проекта не запускаются автоматически после обновления. Преподаватель сам решает, когда запускать анализ.

Запуск ИИ не должен автоматически менять статус проверки на "проверено". Решение о статусе остается за преподавателем.

## Комментарии внутри обновления

Преподаватель должен иметь возможность добавить один или несколько комментариев к любому обновлению проекта.

В первой версии комментарий привязывается к событию обновления целиком. Комментарии к конкретному файлу, коммиту, строке кода или отдельному фрагменту diff не нужны.

Комментарии преподавателя являются личными рабочими заметками и историей проверки. Они не отправляются студентам автоматически, не публикуются в GitHub и не считаются официальной выдачей оценки. Преподаватель может использовать их как основу для обратной связи вне приложения.

Комментарий хранит:

- текст комментария;
- студента;
- проект;
- обновление, к которому он относится;
- дату создания;
- дату последнего изменения как техническое поле данных, если оно уже есть в модели хранения.

Преподаватель должен иметь возможность:

- добавить несколько общих комментариев внутри выбранного обновления;
- удалить свой комментарий после явного подтверждения;
- увидеть все комментарии выбранного обновления;
- увидеть в ленте, сколько комментариев есть у обновления.

В интерфейсе первой версии сохраненный комментарий не должен выглядеть как открытая форма редактирования. Сохраненные комментарии показываются компактными заметками с датой, текстом и простым действием удаления. Внутри блока комментариев уже сохраненные заметки должны читаться выше формы добавления нового комментария, чтобы преподаватель сначала видел, что уже зафиксировано. Редактирование сохраненного комментария в интерфейсе первой версии не требуется, чтобы блок комментариев не превращался в набор вложенных форм.

Удаление комментария является удалением личной рабочей заметки преподавателя. Оно не должно удалять событие обновления, статус проверки, ИИ-рапорты или другие комментарии этого обновления.

История редакций и история удаления комментариев откладываются до второй версии.

В рабочей области выбранного обновления комментарии должны быть отдельным блоком рядом с проверкой или после нее, а не частью технической справки и не частью текста ИИ-рапорта. Это нужно, чтобы преподаватель ясно видел разницу между подсказками ИИ и собственными рабочими заметками.

## Статус проверки обновления

У каждого обновления должен быть статус проверки.

Минимальные статусы первой версии:

- не проверено;
- проверяется;
- проверено;
- требует доработки;
- требует повторной проверки;
- пропущено.

Статус относится к конкретному обновлению, а не ко всему студенту навсегда.

Преподаватель может изменить статус вручную в деталях выбранного обновления.

Лента должна показывать статус каждого обновления, чтобы преподаватель сразу видел, какие события еще не обработаны.

## Открыть код обновления в VS Code

В деталях выбранного обновления должна быть кнопка открытия проекта в VS Code именно на версии кода этого обновления.

Версией обновления считается новый коммит обновления из `Ф002`. Если обновление было первой загрузкой, это коммит, который был загружен первым. Если обновление не изменило проект, это тот же последний известный коммит.

Поведение:

- если у обновления есть новый коммит и локальный репозиторий доступен, приложение открывает VS Code на версии этого коммита;
- если обновление не имеет известного коммита, действие недоступно и страница показывает причину;
- если локальный репозиторий отсутствует или поврежден, приложение показывает ошибку;
- если VS Code или команда открытия недоступны, приложение показывает понятную ошибку и путь, который пыталось открыть.

Для первой версии приложение не должно переключать основную локальную копию проекта между историческими коммитами. Это может мешать будущим обновлениям.

Решение первой версии: для открытия конкретной версии приложение создает или переиспользует отдельную review-копию через Git worktree или другой безопасный локальный механизм и открывает в VS Code именно эту папку.

Такая review-копия должна соответствовать `new commit` выбранного обновления. Для преподавателя это должно выглядеть как "открыть проект на версии этого обновления".

Review-копии являются внутренним техническим механизмом. В первой версии их не нужно показывать отдельным списком или настройкой в интерфейсе.

Открытие VS Code не является обновлением проекта и не создает новое событие в ленте.

Приложение не должно автоматически менять файлы студенческого проекта. Комментарии преподавателя должны сохраняться в приложении, а не в файлах студенческого репозитория.

В первой версии поддерживается только открытие в VS Code. Выбор другого редактора не нужен.

## Обновить только этот проект

На странице студента должна быть возможность обновить репозиторий только этого студента.

Это действие использует те же правила Git, ошибки и историю, которые описаны в `Ф002`, но применяется только к одному проекту.

Сценарий:

1. Преподаватель открывает страницу студента.
2. Нажимает действие обновления проекта этого студента.
3. Приложение запускает обновление только для этого проекта.
4. На странице показывается состояние выполнения.
5. После завершения в ленте появляется новое обновление с датой.
6. В карточке обновления видны результат, количество коммитов и статус проверки.
7. Преподаватель открывает это обновление.
8. Преподаватель запускает ИИ-анализ, если он нужен.
9. ИИ-рапорт сохраняется внутри этого обновления.
10. Преподаватель открывает код этого обновления в VS Code, если нужно посмотреть файлы.
11. Преподаватель оставляет комментарии внутри этого обновления.
12. Преподаватель выставляет статус проверки этого обновления.

Ошибки одиночного обновления не должны ломать страницу студента. Ошибка сохраняется как событие ленты и показывается внутри него.

## Основной сценарий проверки студента

1. Преподаватель открывает список студентов.
2. Выбирает конкретного студента.
3. Открывается страница студента с лентой обновлений.
4. Преподаватель видит краткое описание проекта, если оно уже создано.
5. Преподаватель видит все обновления проекта по датам.
6. Преподаватель запускает обновление только этого проекта, если хочет проверить свежие изменения.
7. После завершения обновления в ленте появляется новая запись.
8. Преподаватель открывает новую запись.
9. Преподаватель запускает ИИ-анализ выбранного обновления.
10. После анализа внутри обновления появляется ИИ-рапорт.
11. Преподаватель открывает код обновления в VS Code на нужном коммите.
12. Преподаватель добавляет комментарии внутри обновления.
13. Преподаватель выставляет статус проверки обновления.

Ожидаемый результат:

- у студента есть единая лента обновлений;
- каждое обновление содержит собственные рапорты, комментарии и статус;
- преподаватель может вернуться к любому старому обновлению;
- преподаватель может открыть код старого или нового обновления на соответствующем коммите;
- история проверки не смешивается между разными обновлениями.

## Сценарий: первое открытие студента без локального проекта

Если у студента есть GitHub-ссылка, но проект еще не загружен:

- страница показывает, что локальной копии еще нет;
- лента может быть пустой;
- преподаватель может запустить обновление этого проекта;
- после успешной первой загрузки появляется первое обновление в ленте;
- после первой загрузки преподаватель может открыть это обновление и вручную запустить ИИ-анализ, если хочет создать краткое описание проекта.

Если у студента нет GitHub-ссылки:

- страница показывает, что проект не подключен;
- обновление через Git недоступно;
- преподаватель должен исправить данные студента в сценариях `Ф001`.

## Сценарий: открыть старое обновление

Преподаватель может открыть старое обновление из ленты.

Для старого обновления нужно показывать сохраненные данные так, как они были связаны с этим событием:

- дату;
- коммиты;
- результат обновления;
- ИИ-рапорты;
- комментарии преподавателя;
- статус проверки.

Новые комментарии можно добавлять и к старым обновлениям. Это нужно, чтобы преподаватель мог вернуться к истории проекта и добавить заметку задним числом.

Если у старого обновления есть новый коммит, преподаватель может открыть VS Code на версии этого обновления через review-копию.

## Что не входит сюда

- правила Git-обновления и команды Git основного обновления;
- подробная логика ИИ-анализа;
- формат хранения данных;
- редактирование карточки студента;
- отправка комментариев студенту;
- публикация комментариев в GitHub;
- комментарии к конкретным строкам diff;
- встроенный просмотр кода внутри приложения;
- выбор редактора кроме VS Code;
- отображение review-копий как отдельного пользовательского раздела;
- автоматическая проверка качества проекта без решения преподавателя.

## Важные моменты и серые зоны

### Лента как главный объект

Решение первой версии: страница студента строится вокруг ленты обновлений. Комментарии, ИИ-рапорты и статус проверки сохраняются внутри конкретного обновления, а не отдельно от него.

### Несколько ИИ-рапортов одного обновления

Требование: преподаватель может сделать один или несколько ИИ-рапортов для одного обновления.

Решение первой версии: несколько ИИ-рапортов разрешены. Лента показывает количество рапортов, а детали обновления показывают список рапортов.

### Открытие VS Code на исторической версии

Требование: VS Code должен открываться именно на версии кода выбранного обновления.

Серая зона: если переключать основную локальную копию на старые коммиты, это может мешать обновлениям и создавать грязное состояние.

Решение первой версии: открывать выбранный коммит через отдельную review-копию или Git worktree. Основная локальная копия проекта остается копией для обновления `main`. Review-копии не показываются отдельным списком или настройкой.

### Комментарий к обновлению или к конкретному изменению

Требование: преподаватель может добавлять комментарии на изменения.

Серая зона: "изменение" может означать все обновление, отдельный коммит, файл или строку.

Решение первой версии: комментарии привязываются только к обновлению целиком. К одному обновлению можно добавить несколько общих комментариев. Комментарии к файлам, коммитам, строкам и фрагментам diff не нужны.

### Обновление без новых коммитов

Серая зона: нужно ли разрешать ИИ-анализ, если новых изменений нет.

Решение первой версии: обновление без новых коммитов все равно попадает в ленту. ИИ-анализ можно запускать, если у события есть известный новый коммит. Если дополнительно известен старый коммит, ИИ анализирует изменения между состояниями.

### Анализ текущего состояния или изменений

Серая зона: что делать, если у обновления есть новый коммит, но нет старого коммита для сравнения.

Решение первой версии: режим выбирается автоматически. Если старого коммита нет, ИИ делает рапорт текущего состояния проекта на новом коммите и помогает создать краткое описание проекта. Если старый и новый коммит известны, ИИ фокусируется на изменениях между этими двумя состояниями.

### Автоматическое или ручное создание ИИ-описания

Требование: у проекта должно быть краткое ИИ-описание.

Серая зона: создавать его автоматически сразу после первой загрузки или только после ручного запуска ИИ.

Решение первой версии: запуск ИИ остается ручным. Приложение не запускает создание описания автоматически после первой загрузки.

### Режим ИИ-рапорта

Серая зона: нужен ли выбор режима или типа рапорта перед запуском анализа.

Решение первой версии: ручной выбор режима не нужен. Запуск ИИ создает стандартный рапорт для выбранного обновления, а внутренний фокус анализа определяется наличием старого коммита.

### Статус студента и статус обновления

Серая зона: статус "проверено" может относиться к студенту в целом или к конкретному обновлению.

Решение первой версии: статус проверки относится к конкретному обновлению. Сводный статус студента может вычисляться отдельно в `Ф001`.

## Критерии приемки

Первая версия этой функции считается готовой, если преподаватель может:

- открыть страницу конкретного студента;
- увидеть верхний контекст студента и проекта;
- увидеть краткое ИИ-описание проекта или понятное состояние его отсутствия;
- увидеть ленту всех обновлений проекта по датам;
- увидеть в карточке обновления дату, статус, количество коммитов, наличие ИИ-анализа и количество комментариев;
- запустить обновление только проекта текущего студента;
- увидеть новое обновление в ленте после одиночного обновления;
- открыть любое обновление из ленты;
- запустить ИИ-анализ именно для выбранного обновления;
- сохранить один или несколько ИИ-рапортов внутри выбранного обновления;
- увидеть список ИИ-рапортов выбранного обновления;
- добавить один или несколько комментариев внутри выбранного обновления;
- удалить комментарий после подтверждения;
- увидеть в ленте, сколько комментариев есть у обновления;
- вручную изменить статус проверки обновления;
- открыть локальный проект в VS Code на коммите выбранного обновления;
- открыть старое обновление и увидеть его сохраненные рапорты, комментарии и статус;
- увидеть понятные ошибки, если проект нельзя обновить, открыть или проанализировать.
