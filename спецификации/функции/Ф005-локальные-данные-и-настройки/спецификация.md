# Данные приложения и локальные настройки

## Цель

Описать, как Pilot Tracker хранит прикладные данные преподавателя в Supabase, защищает доступ к этим данным, где размещает локальные копии студенческих проектов и какие локальные настройки нужны для ежедневной работы.

## Почему это одна спецификация

После согласования `Т011` нужно явно разделять прикладные данные курса и данные конкретного компьютера. Студенты, проекты, история обновлений, комментарии, статусы проверки и ИИ-рапорты хранятся в Supabase. Локальные репозитории, review-копии, пути, инструменты и временные файлы остаются на компьютере преподавателя.

## Пользовательская задача

Преподаватель должен быть уверен, что данные курса не потеряются между запусками приложения, не будут доступны без входа администратора и что работу можно продолжить на другом компьютере после входа, настройки локальных инструментов и загрузки нужных студенческих репозиториев.

Приложение должно сохранять:

- список студентов;
- связи студентов с GitHub-репозиториями;
- состояние локальной загрузки проектов на конкретном компьютере;
- историю обновлений;
- комментарии преподавателя;
- статусы проверки;
- ИИ-рапорты;
- настройки GitHub, Git и локального Codex CLI;
- настройки размещения локальных репозиториев;
- подключение к Supabase;
- состояние входа администратора через Supabase Auth.

В первой версии эти данные относятся к одному активному курсу или одной активной рабочей группе студентов. Хранилище не должно требовать отдельной сущности курса, семестра или архива когорты.

## Принципы хранения данных после `Т011`

Первая версия использует онлайн-проект Supabase как основной источник истины для прикладных данных и локальное хранение для данных конкретного компьютера.

Правила:

- прикладные данные приложения сохраняются в онлайн-проекте Supabase;
- локальные репозитории студентов сохраняются на компьютере преподавателя;
- review-копии сохраняются на компьютере преподавателя;
- локальные настройки путей и инструментов сохраняются на компьютере преподавателя;
- история не должна теряться между запусками приложения;
- обновления, комментарии, статусы проверки и ИИ-рапорты должны быть записаны явно;
- данные курса должны быть доступны после открытия приложения на другом компьютере с тем же подключением к Supabase;
- данные курса должны быть доступны только после успешного входа администратора;
- приложение зависит от доступности Supabase для создания и изменения прикладных данных;
- приложение может работать через Next.js-runtime, который нужен для защищенного веб-доступа и серверной проверки сессии;
- приложение не требует локального Supabase через Docker;
- приложение рассчитано на одного преподавателя.

Формат хранения прикладных данных первой версии после `Т011` - таблицы Supabase. Старые JSON-файлы внутри `data/app/` могут использоваться только как источник миграции и локальный архив старого состояния.

Локальная файловая структура сохраняется для репозиториев, review-копий, временных файлов, диагностических журналов и машинных настроек.

## Базовая структура папок

Все локальные артефакты, которые создает или изменяет Pilot Tracker, должны по умолчанию находиться внутри папки `data/` в папке приложения. Прикладные данные курса находятся в Supabase.

Минимальная структура первой версии:

```text
data/
  app/
    settings.json
    migration-archive/
  repositories/
    <student-id>/
      <project-id>/
  review-copies/
    <student-id>/
      <update-id>/
  logs/
    diagnostics.jsonl
  backups/
```

Назначение папок:

- `data/app/` хранит локальные машинные настройки и архивы миграции, если они нужны;
- `data/repositories/` хранит основные локальные копии студенческих репозиториев;
- `data/review-copies/` хранит временные или переиспользуемые review-копии для открытия исторических версий;
- `data/logs/` хранит локальные диагностические журналы приложения;
- `data/backups/` зарезервирована для будущих локальных резервных копий.

Прикладные данные курса не должны записываться в локальные JSON-файлы как активное хранилище после перехода на Supabase.

## Данные, которые должны сохраняться

### Студенты и проекты

Для студентов и проектов нужно сохранять данные, описанные в `Ф001`:

- имя студента;
- дополнительную информацию;
- ссылку GitHub;
- связанный проект;
- локальный путь проекта;
- текущую ветку, если она известна;
- последний известный коммит;
- дату последнего обновления;
- последний статус проекта.

Изменение карточки студента не должно удалять уже сохраненную историю.

### Обновления и история

Для обновлений нужно сохранять данные, описанные в `Ф002`:

- запуск общего обновления;
- отдельные события обновления проектов;
- дату и время обновления;
- старый коммит;
- новый коммит;
- количество новых коммитов;
- результат обновления;
- ошибку или причину недоступности проекта;
- связь обновления со студентом и проектом.

Старые обновления не должны перезаписываться новым обновлением.

### Проверка и комментарии

Для проверки нужно сохранять данные, описанные в `Ф003`:

- комментарии преподавателя;
- дату создания комментария;
- дату последнего редактирования комментария;
- статус проверки конкретного обновления;
- связь комментариев и статусов с выбранным обновлением.

Ручные комментарии преподавателя можно редактировать, если он ошибся или передумал. История редакций комментариев не обязательна для первой версии.

### ИИ-рапорты и ИИ-артефакты

Для ИИ-анализа нужно сохранять данные, описанные в `Ф004`:

- ИИ-рапорты;
- задания локальной очереди ИИ-анализа;
- дату и время запуска анализа;
- статус анализа;
- коммит или диапазон коммитов;
- краткое резюме;
- полный текст рапорта;
- черновик комментария;
- ошибку ИИ-запуска, если анализ не удался;
- краткое описание проекта, если оно было создано или обновлено ИИ.

ИИ-рапорт должен сохраняться внутри выбранного обновления после ручного запуска анализа. Если анализ поставлен в локальную очередь, само задание очереди тоже сохраняется в Supabase, чтобы преподаватель мог перейти на другую страницу и позже увидеть состояние выполнения. Временные файлы запуска и диагностические артефакты остаются локальными.

## Локальные копии репозиториев

GitHub является источником кода студентов, но не единственным источником данных Pilot Tracker.

Приложение хранит собственную историю, потому что:

- GitHub не знает комментарии преподавателя;
- GitHub не показывает локальные ИИ-анализы;
- комментарии и ИИ-рапорты предназначены только для преподавателя;
- преподавателю нужна сводка по всем студентам, а не только отдельные репозитории.

Для локальных копий нужно хранить или вычислять локально:

- базовую папку студенческих репозиториев: `data/repositories/`;
- локальный путь каждого проекта на конкретном компьютере;
- локальные review-копии или worktree в `data/review-copies/`, если они используются для открытия исторических версий из `Ф003`;
- связь локального пути с GitHub-ссылкой и студентом.

На новом компьютере после входа администратора Supabase сразу дает список студентов, ссылки на репозитории и историю проверки. Если локальной копии проекта на этом компьютере еще нет, приложение должно показывать, что проект не загружен локально, и предлагать загрузить его по GitHub-ссылке перед обновлением, открытием в VS Code или новым ИИ-анализом.

Основная локальная копия проекта используется для обновления ветки `main`. Она не должна использоваться для переключения на старые коммиты при проверке исторических обновлений.

Review-копии являются внутренним техническим механизмом. В первой версии они хранятся отдельно от основных репозиториев, чтобы не путаться с локальными копиями для обновления `main`.

## Настройки первой версии

Минимальные настройки и служебные значения:

- базовая папка данных приложения: `data/`;
- локальная папка настроек и архивов миграции: `data/app/`;
- базовая папка локальных студенческих репозиториев: `data/repositories/`;
- базовая папка review-копий: `data/review-copies/`;
- команда или путь для открытия VS Code;
- команда или путь к локальному Codex CLI;
- параметры Git, если приложению нужно явно указывать окружение;
- команда или путь к GitHub CLI, если он используется для получения контекста pull request;
- результат последней диагностики Git;
- результат последней диагностики локального Codex CLI;
- результат последней диагностики GitHub CLI, если он используется;
- результат последней диагностики VS Code;
- состояние локальной GitHub-аутентификации, если его удалось проверить через GitHub CLI;
- адрес онлайн-проекта Supabase;
- состояние подключения к Supabase;
- время последней проверки подключения к Supabase;
- текст последней ошибки подключения к Supabase, если она есть;
- состояние текущей администраторской сессии без сохранения пароля в настройках.

Настройки GitHub для приватных репозиториев не должны хранить пароли внутри приложения. Первая версия должна опираться на локально настроенную Git-аутентификацию преподавателя.

Пароль администратора не хранится в настройках Pilot Tracker. Вход и хранение учетных данных выполняет Supabase Auth. Интерфейс не должен показывать пароль, токены сессии или секретные ключи Supabase.

Для ИИ-анализа первая версия использует только локальный Codex CLI. Приложение не вызывает OpenAI API напрямую, не хранит API-ключи OpenAI и не поддерживает альтернативные ИИ-провайдеры через адаптеры.

## Приватные репозитории

Приватные GitHub-репозитории поддерживаются, если у преподавателя есть доступ и локальная Git-аутентификация настроена.

Правила:

- приложение не должно пытаться обходить права доступа GitHub;
- приложение не должно хранить GitHub-пароль в собственных данных;
- ошибка доступа к приватному репозиторию сохраняется как ошибка обновления в `Ф002`;
- отсутствие доступа к одному репозиторию не должно блокировать работу с остальными проектами.

## Переносимость и резервное копирование

Данные должны быть переносимыми между компьютерами настолько, насколько это возможно для гибридного приложения.

Минимальные требования:

- преподаватель должен понимать, какие данные лежат в Supabase, а какие остаются локально;
- преподаватель должен понимать, где лежат локальные копии репозиториев;
- данные приложения должны быть пригодны для резервного копирования средствами Supabase и миграций;
- после переноса данных приложение должно уметь восстановить связи студентов, проектов, обновлений, комментариев, статусов и ИИ-рапортов;
- если локальные репозитории отсутствуют на новом компьютере, приложение должно предложить загрузить проект по GitHub-ссылке из Supabase;
- если администратор не вошел, приложение должно показывать страницу входа и не раскрывать рабочие данные.

Экспорт данных или истории курса в отдельный формат не входит в первую версию. Структура данных не должна специально усложняться ради будущего экспорта, но и не должна закрывать возможность добавить экспорт позже.

## История и защита от потери данных

Ключевое требование: не терять данные с момента создания проекта до конца курса.

Правила:

- каждое обновление должно оставлять запись в истории;
- старые обновления и связанные с ними данные проверки нельзя случайно перезаписывать или удалять;
- обновление без новых изменений тоже сохраняется;
- событие отсутствующей GitHub-ссылки тоже сохраняется;
- ошибка обновления тоже сохраняется;
- ИИ-рапорт сохраняется в Supabase после ручного запуска анализа;
- задание ИИ-анализа сохраняется в Supabase сразу после ручной постановки в очередь;
- явное удаление данных должно требовать подтверждения, если вместе с записью будут удалены история, комментарии или ИИ-рапорты;
- чтение и изменение прикладных данных должны требовать авторизованную администраторскую сессию.

## Что не входит сюда

- пользовательский сценарий обновления проектов;
- содержание проверки преподавателя;
- логика ИИ-анализа;
- дизайн интерфейса настроек;
- офлайн-синхронизация с конфликтами между устройствами;
- многопользовательский режим;
- публичная регистрация и многоролевая авторизация;
- студенческий кабинет;
- управление несколькими курсами, семестрами или архивами когорт;
- экспорт данных или истории курса;
- автоматическая публикация комментариев студентам.

## Критерии приемки

Первая версия этой функции считается готовой, если приложение:

- создает или использует папку `data/` внутри папки приложения;
- хранит прикладные данные в Supabase;
- защищает прикладные данные входом администратора;
- сохраняет список студентов между запусками;
- сохраняет связи студентов с GitHub-репозиториями;
- определяет локальное состояние загрузки проектов на конкретном компьютере;
- сохраняет историю обновлений;
- сохраняет комментарии преподавателя;
- сохраняет статусы проверки обновлений;
- сохраняет ИИ-рапорты внутри соответствующих обновлений;
- сохраняет ошибки обновления и ошибки ИИ-анализа;
- клонирует студенческие репозитории в `data/repositories/`;
- хранит review-копии или Git worktree в `data/review-copies/`;
- позволяет настроить или определить команду открытия VS Code;
- позволяет настроить или определить команду локального Codex CLI;
- позволяет настроить или определить GitHub CLI, если он используется для контекста pull request;
- не требует отдельного сервера вне выбранного Next.js-runtime;
- не теряет историю после перезапуска приложения.

## Открытые вопросы

На текущий момент открытых вопросов по `Ф005` нет.

## Решения для первой версии

- Прикладные данные хранятся в Supabase, а локальные артефакты и машинные настройки хранятся на компьютере преподавателя.
- Хранилище первой версии описывает один активный курс или одну активную рабочую группу студентов.
- В первой версии после `Т011` прикладные данные приложения хранятся в Supabase.
- Локальные репозитории, review-копии, временные файлы, журналы и машинные настройки по умолчанию находятся внутри папки `data/` в папке приложения.
- Локальные настройки, временные файлы и архивы миграции могут храниться в `data/app/`.
- Основные локальные копии студенческих репозиториев хранятся в `data/repositories/`.
- Review-копии и Git worktree для исторических версий хранятся в `data/review-copies/`.
- Отдельный сервер приложения вне Next.js-runtime не требуется.
- Приложение рассчитано на одного преподавателя.
- Для защищенного доступа используется Supabase Auth и одна учетная запись администратора-преподавателя.
- Приложение хранит собственную историю обновлений, комментариев, статусов и ИИ-рапортов.
- GitHub является источником кода, но не источником истории проверки.
- Приложение не хранит GitHub-пароль в собственных данных.
- Для ИИ-анализа приложение запускает только локальный Codex CLI.
- Приложение не вызывает OpenAI API напрямую.
- Приложение не поддерживает адаптеры для других ИИ-провайдеров.
- Локальные копии студенческих репозиториев считаются источником для чтения и анализа.
- Экспорт данных и истории курса не реализуется в первой версии, за исключением технических миграций Supabase.
- Данные должны быть пригодны для резервного копирования и переноса.
