# Общее обновление проектов перед проверкой

## Цель

Как преподаватель, я хочу вручную обновить все проекты перед проверкой, чтобы увидеть, у кого появились новые изменения, где изменений нет и где возникли ошибки.

## Связанные спецификации

- `Ф002`: обновление всех проектов запускается вручную и создает историю для каждого студента.
- `Ф001`: список студентов показывает статусы и помогает выбрать следующую проверку.
- `Ф005`: локальные репозитории хранятся в `data/repositories/`.
- `Т005`: Git-операции работают с веткой `main`, не делают commit и push.
- `Т010`: ошибка одного проекта не останавливает общий запуск.

## Лучший сценарий

1. Преподаватель открывает список студентов.
2. Нажимает действие обновления всех проектов.
3. Приложение проходит по всем студентам.
4. Для проектов без локальной копии приложение выполняет первый clone в `data/repositories/`.
5. Для существующих локальных копий приложение проверяет чистоту, получает `origin/main` и обновляет копию до `main`.
6. Для каждого проекта приложение создает событие обновления.
7. Для проектов с новыми коммитами приложение сохраняет старый коммит, новый коммит и количество новых коммитов.
8. Приложение показывает сводку: новые изменения, без изменений, ошибки, студенты без ссылки.
9. Список студентов сортируется так, чтобы проекты с новыми изменениями и ошибками были заметны.

Ожидаемый результат: преподаватель видит, с кого начать проверку, а история обновлений сохранена для каждого студента.

## Рабочие отклонения

### У студента нет GitHub-ссылки

Приложение не запускает Git-команды, но создает событие истории о недоступном проекте. В списке студент получает статус, что проект не подключен. Это не считается Git-ошибкой и не останавливает общий запуск.

### У проекта нет новых коммитов

Приложение все равно сохраняет событие успешного обновления без новых изменений. Это важно, чтобы преподаватель видел, что проверка актуальности была выполнена.

### Один приватный репозиторий недоступен

Приложение сохраняет ошибку доступа для конкретного проекта, показывает ее в списке и в истории студента, но продолжает обновлять остальные проекты.

### Локальная копия грязная

Приложение обнаруживает локальные изменения через Git и не делает `reset` автоматически. Для этого проекта сохраняется ошибка локальной копии, потому что приложение не должно молча исправлять состояние студенческого репозитория.

## Худший допустимый сценарий

Общий запуск не может прочитать список студентов или не может записать историю обновлений.

В этом случае операция верхнего уровня завершается ошибкой. Приложение показывает критичную проблему хранения или чтения данных и не делает вид, что обновление прошло успешно.

## Шаги и опора на спецификации

| Шаг | Действие | Почему это разрешено спецификациями |
| --- | --- | --- |
| 1 | Запустить обновление вручную | Бизнес-контекст и `Ф002` запрещают автоматический запуск без действия преподавателя. |
| 2 | Обработать каждого студента отдельно | `Ф002` требует отдельное событие обновления для каждого проекта. |
| 3 | Клонировать в `data/repositories/` | `Ф005` и `Т005` фиксируют папку основных локальных репозиториев. |
| 4 | Работать с веткой `main` | `Ф002` и `Т005` фиксируют `main` как основную ветку первой версии. |
| 5 | Сохранить границу анализа | `Ф002` требует старый коммит, новый коммит, ветку и локальный путь для `Ф004`. |
| 6 | Не делать push | Бизнес-контекст, `Ф002` и `Т005` запрещают отправлять изменения в репозитории студентов. |
| 7 | Продолжить после ошибки одного проекта | `Ф002` и `Т010` требуют не останавливать общий запуск из-за одной ошибки. |

## Границы

Эта история не получает diff, список файлов, pull request и комментарии GitHub. Эти данные получает ИИ-анализ в `Ф004`, если преподаватель запускает его вручную.
