# План историй пользователя

## Роль этого документа

Этот документ собирает пользовательские истории первой версии Pilot Tracker.

Истории не добавляют новые требования. Они помогают понять продукт перед разработкой через реальные рабочие ситуации преподавателя: лучший сценарий, обычные отклонения, ошибки и худший допустимый сценарий внутри уже принятых спецификаций.

## Правило источников

Каждая история должна опираться только на:

- `спецификации/бизнес-контекст.md`;
- функциональные спецификации `Ф001`-`Ф005`;
- технические спецификации `Т001`-`Т012`.

Если в истории появляется действие, которого нет в спецификациях, такую историю нужно остановить и сначала обновить спецификации по явному решению владельца проекта.

## Формат истории

Каждая история содержит:

- цель преподавателя;
- связанные спецификации;
- лучший сценарий;
- рабочие отклонения;
- худший допустимый сценарий;
- шаги и ссылку на спецификации, которые разрешают каждый шаг;
- границы, за которые история не выходит.

## Покрытие продукта

Эти истории покрывают основной поток первой версии:

1. преподаватель входит как администратор, если сессии еще нет;
2. готовит локальное окружение;
3. создает список студентов;
4. подключает GitHub-репозитории;
5. запускает обновление;
6. выбирает студента с новыми изменениями или ошибкой;
7. открывает конкретное обновление;
8. запускает ИИ-анализ вручную;
9. открывает код в VS Code через безопасную review-копию;
10. сохраняет комментарий и статус проверки;
11. возвращается к истории после перезапуска;
12. удаляет данные только после явного подтверждения;
13. выходит из администраторской сессии.

## Границы историй

Истории не описывают:

- студенческий кабинет;
- несколько курсов;
- публичную регистрацию;
- несколько ролей;
- LMS;
- публикацию комментариев в GitHub;
- автоматическую оценку;
- облачную синхронизацию;
- экспорт истории курса;
- прямой вызов OpenAI API из приложения.
