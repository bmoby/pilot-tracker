# Удаление студента с защитой от потери данных

## Цель

Как преподаватель, я хочу удалить студента только после понятного предупреждения, чтобы случайно не потерять историю обновлений, комментарии, статусы и ИИ-рапорты.

## Связанные спецификации

- Бизнес-контекст: явное удаление может быть разрешено только с понятным предупреждением о последствиях.
- `Ф001`: удаление студента требует подтверждения.
- `Ф005`: данные проверки не должны теряться случайно.
- `Т003`: удаление студента затрагивает проект, обновления, комментарии, статусы и ИИ-рапорты.
- `Т004`: физическое удаление папок требует отдельного осторожного правила.
- `Т007`: интерфейс должен показать подтверждение и последствия.

## Лучший сценарий

1. Преподаватель открывает список студентов.
2. Выбирает действие удаления у студента без истории.
3. Приложение показывает подтверждение.
4. Преподаватель подтверждает.
5. Студент исчезает из списка.
6. Связанный проект удаляется из локальных данных приложения.

Ожидаемый результат: удаление выполнено явно, без скрытых побочных действий и без влияния на других студентов.

## Рабочие отклонения

### У студента есть история обновлений

Приложение показывает предупреждение, что вместе со студентом будут удалены история обновлений, комментарии, статусы проверки и ИИ-рапорты. Без подтверждения удаление не выполняется.

### Преподаватель отменяет удаление

Приложение закрывает подтверждение и не изменяет данные. Студент остается в списке, история остается доступной.

### Есть локальный репозиторий или review-копии

Удаление записи студента не должно молча удалять физические папки репозиториев и review-копий, если для этого нет отдельного правила файловых операций и явного решения. История модели данных и файловые операции разделяются.

## Худший допустимый сценарий

Удаление началось, но запись изменений в JSON не удалась.

Приложение показывает ошибку хранения и не должно отображать удаление как успешное, пока данные не сохранены надежно. Если часть файлов уже была изменена, это становится диагностической проблемой, которую нельзя скрывать.

## Шаги и опора на спецификации

| Шаг | Действие | Почему это разрешено спецификациями |
| --- | --- | --- |
| 1 | Предложить удаление из списка или карточки | `Ф001` разрешает удаление студента после подтверждения. |
| 2 | Предупредить о потере истории | Бизнес-контекст, `Ф001` и `Ф005` требуют предупреждать о последствиях. |
| 3 | Удалить связанные данные модели | `Т003` описывает удаление студента, проекта, событий, комментариев, статусов и ИИ-рапортов. |
| 4 | Не удалять физические папки молча | `Т003` и `Т004` отделяют модель данных от файловых операций. |
| 5 | Остановиться при ошибке записи | `Т004` и `Т010` считают ошибку надежной записи критичной для действия. |

## Границы

Эта история не добавляет архивирование студентов, мягкое удаление, восстановление удаленных данных, экспорт перед удалением или отдельный экран управления корзиной.
