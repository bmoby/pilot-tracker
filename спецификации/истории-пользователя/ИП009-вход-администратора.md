# Вход администратора и защищенный доступ

## Цель

Как преподаватель-администратор, я хочу войти в Pilot Tracker по email и паролю, чтобы с любого компьютера видеть рабочие данные Supabase и чтобы посторонний пользователь не мог открыть список студентов, историю проверки, комментарии и ИИ-рапорты.

## Связанные спецификации

- Бизнес-контекст: Pilot Tracker остается инструментом одного преподавателя.
- `Ф005`: прикладные данные доступны через Supabase и защищены администраторским входом.
- `Т001`: приложение остается Next.js-инструментом одного преподавателя, а не командной платформой.
- `Т002`: рабочие маршруты проходят через серверные адаптеры и проверку сессии.
- `Т007`: без сессии показывается экран входа, после входа первым рабочим экраном остается список студентов.
- `Т010`: ошибки входа, истечения сессии и отказа доступа показываются понятно и без раскрытия секретов.
- `Т011`: Supabase является источником истины для прикладных данных.
- `Т012`: Supabase Auth, cookie-сессии, `app_metadata` администратора и RLS защищают данные.

## Лучший сценарий

1. Преподаватель открывает Pilot Tracker на компьютере без действующей сессии.
2. Приложение показывает экран входа администратора.
3. Преподаватель вводит email и пароль.
4. Supabase Auth подтверждает учетные данные.
5. Приложение проверяет, что пользователь является разрешенным администратором.
6. Приложение открывает список студентов из Supabase.
7. Преподаватель работает с данными курса.
8. Преподаватель нажимает выход.
9. Приложение завершает сессию и возвращает пользователя на вход.

Ожидаемый результат: рабочие данные доступны только после входа администратора и снова закрыты после выхода.

## Рабочие отклонения

### Неверный email или пароль

Приложение показывает короткую ошибку входа. Сообщение не уточняет, существует ли такой email в Supabase Auth.

### Пользователь не является администратором

Supabase Auth может подтвердить учетные данные, но приложение не открывает данные, если у пользователя нет разрешенного признака администратора.

### Сессия истекла

Приложение возвращает пользователя на вход. Рабочие данные не показываются до повторного входа.

### Supabase недоступен

Приложение показывает ошибку подключения. Создание, изменение и чтение прикладных данных не считаются успешными.

### Новый компьютер без локальных инструментов

После входа преподаватель видит прикладные данные Supabase, но действия Git, VS Code, GitHub CLI и Codex CLI могут быть недоступны до настройки локальной среды.

## Худший допустимый сценарий

RLS отклоняет доступ или Supabase возвращает ошибку авторизации.

Приложение не должно обходить это через `service_role` в пользовательском потоке. Оно показывает ошибку доступа, не раскрывает секреты и не показывает пустую историю как успешный результат.

## Шаги и опора на спецификации

| Шаг | Действие | Почему это разрешено спецификациями |
| --- | --- | --- |
| 1 | Показать вход без сессии | `Т007` и `Т012` требуют вход до показа рабочих данных. |
| 2 | Проверить email и пароль через Supabase Auth | `Т012` выбирает Supabase Auth как механизм входа администратора. |
| 3 | Проверить признак администратора | `Т012` требует безопасный признак из `app_metadata`, а не `user_metadata`. |
| 4 | Открыть список студентов | `Ф001`, `Т007` и `Т011` делают список студентов первым рабочим экраном после входа. |
| 5 | Работать с данными через RLS | `Т011` и `Т012` требуют Supabase как источник истины и RLS для таблиц. |
| 6 | Завершить сессию | `Т007` и `Т012` требуют действие выхода и закрытие доступа к данным. |
| 7 | Не раскрывать секреты в ошибках | `Т010` запрещает сохранять и показывать пароли, cookie, JWT и ключи Supabase. |

## Границы

Эта история не добавляет публичную регистрацию, восстановление пароля, смену пароля, управление пользователями, несколько ролей, студенческий кабинет, командную работу или удаленный запуск локальных инструментов.
