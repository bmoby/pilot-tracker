# Создание и уточнение рабочей когорты студентов

## Цель

Как преподаватель, я хочу создать список студентов и связать каждого студента с его GitHub-репозиторием, чтобы видеть текущую рабочую когорту и выбирать следующий проект для проверки.

## Связанные спецификации

- Бизнес-контекст: первая версия работает с одной активной рабочей группой.
- `Ф001`: студент создается с именем, дополнительной информацией и необязательной GitHub-ссылкой.
- `Ф005`: список студентов и связи с репозиториями сохраняются локально.
- `Т003`: студент связан с одним основным проектом.
- `Т007`: список студентов является главным экраном и показывает статусы проектов.

## Лучший сценарий

1. Преподаватель открывает список студентов.
2. Нажимает добавление студента.
3. Вводит имя, заметку и ссылку на GitHub-репозиторий.
4. Сохраняет запись.
5. Приложение показывает студента в списке со статусом, что проект еще не обновлялся.
6. Преподаватель добавляет остальных студентов.
7. Список остается одной рабочей когортой без курсов, групп и сложных фильтров.

Ожидаемый результат: у каждого студента есть карточка и связанный основной проект, а данные сохраняются в локальных JSON-файлах.

## Рабочие отклонения

### Студент пока без GitHub-ссылки

Преподаватель создает студента только с именем. Приложение сохраняет запись и показывает статус, что проект не подключен. Такой студент не ломает список и позже может получить ссылку GitHub через редактирование.

### Ссылка добавляется позже

Преподаватель редактирует карточку студента и добавляет GitHub-ссылку. Приложение сохраняет ссылку и показывает, что проект еще не обновлялся. Проверка доступности репозитория происходит не при сохранении формы, а во время обновления проектов.

### Ссылка изменяется после истории обновлений

Преподаватель меняет GitHub-ссылку у студента, у которого уже есть история. Приложение предупреждает, что это влияет на связанный проект и будущие обновления, но не удаляет старые обновления, комментарии, статусы и ИИ-рапорты молча.

## Худший допустимый сценарий

Преподаватель пытается сохранить студента без имени или с пустым именем.

Приложение не сохраняет такую запись, показывает ошибку валидации и не создает частичный проект без валидного студента.

## Шаги и опора на спецификации

| Шаг | Действие | Почему это разрешено спецификациями |
| --- | --- | --- |
| 1 | Работать с одной когортой | Бизнес-контекст и `Ф001` исключают несколько курсов и архивы в первой версии. |
| 2 | Создать студента без GitHub-ссылки | `Ф001` прямо разрешает создать студента без ссылки и добавить ее позже. |
| 3 | Создать связанный проект | `Т003` фиксирует связь `student 1 -> 1 project`. |
| 4 | Показать статус проекта в списке | `Ф001` и `Т007` требуют показывать последний статус проекта. |
| 5 | Не проверять доступ при сохранении формы | `Ф001` относит фактическую проверку доступности репозитория к обновлению из `Ф002`. |
| 6 | Не терять историю при редактировании | `Ф001` и `Ф005` требуют сохранять историю и не удалять ее молча. |

## Границы

Эта история не добавляет теги, группы, поиск, архивирование студентов, несколько проектов на одного студента, email, аватары или импорт списка студентов.
