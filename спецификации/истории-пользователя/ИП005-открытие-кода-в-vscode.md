# Открытие кода выбранного обновления в VS Code

## Цель

Как преподаватель, я хочу открыть код конкретного обновления в VS Code на правильном коммите, чтобы проверить проект вручную без повреждения основной локальной копии.

## Связанные спецификации

- `Ф003`: открытие кода доступно из деталей выбранного обновления.
- `Ф005`: основные репозитории и review-копии хранятся отдельно.
- `Т005`: основная локальная копия предназначена для обновления `main`.
- `Т006`: исторические версии открываются через review-копию или Git worktree.
- `Т008`: команда VS Code настраивается и диагностируется.
- `Т010`: ошибка открытия не меняет статус проверки.

## Лучший сценарий

1. Преподаватель открывает детали обновления с известным `newCommit`.
2. Нажимает действие открытия кода.
3. Приложение проверяет, что локальный репозиторий доступен.
4. Приложение создает или переиспользует review-копию в `data/review-copies/`.
5. Review-копия переводится на `newCommit` выбранного обновления.
6. Основная локальная копия в `data/repositories/` остается на `main`.
7. Приложение запускает VS Code с путем review-копии как отдельным аргументом.
8. Преподаватель проверяет код вручную.

Ожидаемый результат: преподаватель смотрит нужную версию кода, а будущие обновления не ломаются из-за переключения основной копии.

## Рабочие отклонения

### Review-копия уже существует

Приложение проверяет существующую review-копию. Если она уже соответствует нужному коммиту и рабочая директория чистая, приложение открывает ее повторно. Если в ней остались локальные изменения от прошлого просмотра, приложение принудительно восстанавливает review-копию до `newCommit` выбранного обновления и открывает ее.

### VS Code недоступен

Приложение показывает понятную ошибку и путь, который пыталось открыть. Статус проверки не меняется, review-копия не удаляется автоматически.

### У обновления нет `newCommit`

Действие открытия кода недоступно. Интерфейс объясняет, что нет известного коммита, который можно открыть.

## Худший допустимый сценарий

Нужный коммит отсутствует локально или локальный репозиторий поврежден.

Приложение сохраняет ошибку открытия, не переключает основную копию, не делает автоматический `fetch` только ради старого обновления и не удаляет данные проверки.

## Шаги и опора на спецификации

| Шаг | Действие | Почему это разрешено спецификациями |
| --- | --- | --- |
| 1 | Открывать код из деталей обновления | `Ф003` связывает действие открытия с выбранным обновлением. |
| 2 | Использовать `newCommit` | `Ф003` определяет версию обновления как новый коммит из `Ф002`. |
| 3 | Создать review-копию | `Ф005` и `Т006` требуют отделить review-копии от основной копии. |
| 4 | Не переключать основную копию | `Ф003`, `Ф005` и `Т006` защищают основную копию `main` для будущих обновлений. |
| 5 | Открыть VS Code | `Ф003` и `Т008` поддерживают открытие кода через VS Code. |
| 6 | Не менять статус проверки автоматически | `Ф003` и `Т010` требуют ручного статуса и безопасного поведения при ошибках. |
