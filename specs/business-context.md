# Pilot Tracker: бизнес-контекст

## Назначение проекта

Pilot Tracker - локальный инструмент преподавателя для отслеживания личных проектов студентов на GitHub.

Проект нужен, чтобы преподаватель мог:

- хранить список студентов и связанные с ними GitHub-репозитории;
- одной кнопкой обновлять все студенческие проекты;
- быстро видеть, кто из студентов сделал новые изменения;
- открывать конкретный проект студента для проверки;
- сохранять историю обновлений, проверок, комментариев и заметок;
- использовать локальный Codex/AI-помощник для анализа изменений и подготовки черновиков комментариев, которые преподаватель может использовать в своей проверке.

Первая версия должна работать полностью локально на компьютере преподавателя. Онлайн-версия, сервер, авторизация, облачная база данных и командная работа не входят в первую фазу, но архитектура не должна закрывать возможность добавить это позже.

## Почему это важно

Во время курса каждый студент постепенно развивает личный проект. Преподавателю нужно регулярно проверять прогресс, не терять историю и быстро понимать:

- кто обновил проект после прошлой проверки;
- что именно изменилось;
- какие файлы стоит посмотреть;
- какие комментарии уже были оставлены раньше;
- какие проблемы или улучшения повторяются от проверки к проверке.

Без отдельного инструмента эта информация распадается между GitHub, локальными папками, заметками и памятью преподавателя. Pilot Tracker должен стать единым рабочим местом для проверки студенческих проектов.

## Основные пользователи

### Преподаватель

Главный пользователь первой версии. Он добавляет студентов, подключает GitHub-репозитории, запускает обновления, проверяет изменения и пишет комментарии.

### Студент

Студент напрямую не пользуется первой локальной версией приложения. Его данные и репозиторий отображаются в системе как объект проверки.

### AI-помощник

Локальный помощник, который помогает преподавателю анализировать проекты. В первой версии AI не должен автоматически принимать решения за преподавателя. Он готовит черновики, краткие выводы и подсказки. Он может показать файлы, на которые стоит обратить внимание в текущем анализе, но проверка не обязана хранить список просмотренных файлов.

## Объекты предметной области

### Студент

Минимальные данные:

- имя;
- дополнительная информация о студенте;
- ссылка на GitHub-репозиторий;
- локальный путь к склонированному проекту;
- статус последнего обновления;
- история проверок и заметок.

Позже можно добавить группу, email, уровень, теги, аватар, ссылку на демо и другие поля.

### Проект

Проект связан с GitHub-репозиторием студента. Для первой версии один студент может иметь один основной проект. В будущем возможно расширение до нескольких проектов на одного студента.

Проект должен хранить:

- GitHub URL;
- локальную папку проекта;
- текущую ветку;
- последний известный commit;
- дату последнего обновления;
- результат последней проверки обновлений;
- историю изменений.

### Обновление

Обновление - это событие, когда система проверила репозиторий студента и обнаружила или не обнаружила новые изменения.

Для каждого обновления важно сохранять:

- дату и время;
- студента;
- проект;
- старый commit;
- новый commit;
- были ли новые изменения;
- краткое описание результата;
- ошибки, если обновление не удалось.

### Проверка

Проверка - это работа преподавателя с конкретным состоянием проекта студента.

Проверка может содержать:

- ручные комментарии преподавателя;
- AI-черновики комментариев;
- замечания;
- итог проверки, например: проверено, нужно доработать, требует повторной проверки;
- дату проверки.

## Главные сценарии первой версии

### 1. Добавить студента

Преподаватель создает запись студента, указывает имя, дополнительную информацию и GitHub-репозиторий.

Ожидаемый результат: студент появляется в списке, а система знает, откуда получать его проект.

### 2. Обновить все проекты одной кнопкой

Преподаватель нажимает одну кнопку, и система проходит по всем студентам:

- проверяет доступность репозитория;
- клонирует проект, если он еще не был загружен локально;
- делает pull/fetch, если проект уже существует;
- сравнивает предыдущий commit с новым;
- фиксирует результат в истории.

Ожидаемый результат: преподаватель видит, какие студенты обновили проекты, а какие нет.

### 3. Быстро увидеть новые изменения

После общего обновления система должна выделить студентов, у которых появились новые commits.

Ожидаемый результат: преподаватель может сразу выбрать, кого проверять первым.

### 4. Открыть проект студента для проверки

Преподаватель выбирает студента и переходит в отдельное пространство проверки.

Там должны быть доступны:

- информация о студенте;
- ссылка на GitHub;
- локальный путь к проекту;
- история обновлений;
- история проверок;
- место для комментариев и заметок;
- инструменты AI-анализа.

### 5. Сохранить комментарии и заметки

Преподаватель пишет комментарии по проекту студента и сохраняет их.

Ожидаемый результат: заметки не теряются и остаются привязанными к студенту, проекту, дате и конкретной проверке.

### 6. Использовать AI для анализа

Преподаватель запускает AI-анализ для выбранного проекта или изменения.

AI может помогать:

- кратко описать, что изменилось;
- найти важные файлы;
- обратить внимание на подозрительные или слабые места;
- подготовить черновик комментария, который преподаватель может использовать в своей проверке.

Ожидаемый результат: преподаватель получает помощь, но финальное решение и комментарии остаются за ним.

## Принципы хранения данных

Первая версия должна быть local-first:

- все данные сохраняются на компьютере преподавателя;
- история не должна теряться между запусками приложения;
- обновления, проверки и комментарии должны быть записаны явно;
- данные должны быть переносимыми, насколько это возможно;
- проект не должен зависеть от внешней базы данных в первой версии.

На старте можно рассмотреть хранение в локальных файлах или SQLite. Решение будет принято в технической спецификации.

## История и неизменность

Ключевое требование: не терять данные с момента создания проекта до конца курса.

Это означает:

- каждое обновление должно оставлять запись в истории;
- старые проверки нельзя случайно перезаписывать или удалять;
- ручные комментарии преподавателя можно редактировать, если он ошибся или передумал;
- если студент не обновил проект, это тоже может быть полезным событием истории;
- ошибки обновления тоже должны сохраняться, чтобы понимать, что произошло;
- AI-ответы должны сохраняться как черновики или артефакты проверки, если преподаватель решил их оставить.

## Роль GitHub

GitHub является источником кода студентов, но не единственным источником данных Pilot Tracker.

Pilot Tracker должен хранить собственную историю проверок, потому что:

- GitHub не знает заметки преподавателя;
- GitHub не показывает локальные AI-анализы;
- заметки и AI-анализы предназначены только для преподавателя;
- Pilot Tracker не отправляет комментарии, файлы или изменения в репозитории студентов;
- преподавателю нужна удобная сводка по всем студентам, а не только отдельные репозитории.

## Роль Codex и AI

В первой версии предполагается использование локального Codex на компьютере преподавателя.

AI-интеграция должна быть осторожной:

- AI помогает анализировать, но не заменяет преподавателя;
- все AI-выводы должны быть проверяемыми;
- каждый сохраненный AI-анализ должен быть привязан к конкретному состоянию проекта, например к commit hash, чтобы позже было понятно, какую именно версию проекта проверял AI;
- AI работает в режиме чтения: он может анализировать локальную копию проекта, но не должен изменять файлы, делать commit или push;
- преподаватель должен явно запускать анализ.

## Ограничения первой версии

Первая версия:

- работает локально;
- рассчитана на одного преподавателя;
- не требует онлайн-доступа к собственному серверу приложения;
- использует GitHub-репозитории студентов как внешний источник кода;
- должна сохранять данные между запусками;
- должна быть простой для ежедневного использования во время курса.

## Не входит в первую версию

Пока не планируется:

- публичный сайт для студентов;
- личные кабинеты студентов;
- облачная база данных;
- многопользовательский режим;
- автоматическая отправка комментариев студентам;
- сложная система оценивания;
- интеграция с LMS;
- полноценный CI/CD для студенческих проектов.

Эти возможности можно рассмотреть позже, когда локальная версия станет стабильной.

## Вопросы для следующих спецификаций

- Какой формат локального хранения выбрать: SQLite, JSON-файлы или гибрид?
- Нужен ли desktop-интерфейс, web-интерфейс на localhost или CLI в первой версии?
- Где хранить локальные копии студенческих репозиториев?
- Нужно ли поддерживать приватные GitHub-репозитории?
- Как именно Codex будет запускаться из приложения?
- Какие статусы проверки нужны преподавателю?
- Нужно ли сравнение изменений между двумя проверками или достаточно между двумя commits?
- Нужно ли сохранять AI-анализ автоматически или только после подтверждения преподавателя?
- Нужна ли история версий для отредактированных комментариев преподавателя?
- Как экспортировать историю курса в конце обучения?

## Критерии успеха первой рабочей версии

Первая рабочая версия считается полезной, если преподаватель может:

- добавить несколько студентов с GitHub-репозиториями;
- нажать одну кнопку и обновить все проекты;
- увидеть, у кого появились новые изменения;
- открыть конкретного студента и посмотреть его проект;
- сохранить комментарий по проверке;
- увидеть историю обновлений и комментариев после перезапуска приложения;
- запустить AI-анализ выбранного проекта или изменения.
