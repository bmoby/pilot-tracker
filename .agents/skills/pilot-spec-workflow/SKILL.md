# Pilot Tracker: работа через спецификации

## Назначение

Этот навык фиксирует обязательный порядок работы в проекте Pilot Tracker. Его цель - не дать агенту случайно реализовать поведение, которое не соответствует бизнес-контексту или функциональным спецификациям.

## Обязательный порядок

1. Прочитать `спецификации/бизнес-контекст.md`.
2. Определить, какие функциональные спецификации относятся к задаче.
3. Прочитать все прямо связанные спецификации до изменения файлов.
4. Если задача затрагивает границы между функциями, прочитать связанные спецификации.
5. Сравнить запрос пользователя с прочитанными спецификациями.
6. Только после этого начинать реализацию, редактирование документации или изменение требований.

## Как выбирать спецификации

- `Ф001` - студенты, проекты, список, карточки, связь студента с репозиторием и текущие статусы.
- `Ф002` - обновление проектов, Git-операции, обнаружение новых изменений и история обновлений.
- `Ф003` - страница студента, лента обновлений, комментарии преподавателя, статусы проверки и открытие кода.
- `Ф004` - локальный Codex CLI, ИИ-анализ, ИИ-рапорты, черновики комментариев и контекст pull request.
- `Ф005` - локальные JSON-данные, папка `data/`, локальные репозитории, review-копии, настройки Git, GitHub CLI и Codex CLI.

Если задача касается нескольких областей, нужно читать все соответствующие спецификации, а не только одну.

## Правило конфликта

Если запрос пользователя противоречит бизнес-контексту или функциональной спецификации, агент обязан остановиться и сообщить, что запрос выходит за рамки текущих спецификаций или конфликтует с ними.

Агент не должен:

- реализовывать противоречащее поведение;
- молча считать новый запрос новой спецификацией;
- обходить ограничение через техническую деталь;
- расширять первую версию продукта без явного решения владельца проекта.

Если владелец проекта явно просит изменить уже зафиксированное поведение, сначала нужно обновить соответствующие спецификации, а уже потом менять реализацию, документацию или другие материалы проекта.

## Если спецификации неполные

Если спецификация не покрывает запрошенный случай, неоднозначна или оставляет важный продуктовый риск, агент должен сначала указать на пробел.

Для безопасной работы нужно предложить одно из двух действий:

- дополнить или уточнить спецификацию;
- ограничить задачу только той частью, которая уже покрыта спецификациями.

## Рабочая проверка перед ответом

Перед финальным ответом агент должен убедиться, что может кратко назвать:

- какой бизнес-контекст был учтен;
- какие функциональные спецификации были прочитаны;
- почему изменение соответствует спецификациям или почему работа была остановлена;
- какие файлы были изменены, если изменения выполнялись.
