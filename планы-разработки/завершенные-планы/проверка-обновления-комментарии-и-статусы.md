## Цель

Добавить первый рабочий контур проверки конкретного обновления: преподаватель открывает событие в ленте студента, видит детали обновления, добавляет, редактирует и удаляет ручные комментарии после подтверждения, вручную меняет статус проверки и видит сохраненный результат после перезапуска.

## Контекст спецификаций

План продолжает уже реализованные каталог студентов, обновление проектов и ленту событий. `Ф003` требует, чтобы комментарии и статус проверки были связаны с конкретным обновлением, а не со студентом в целом. `Т003` задает JSON-модель `comments.json` и `review-statuses.json`, `Т007` описывает детали выбранного обновления, форму комментария и ручное изменение статуса, а `Т010` требует не показывать успешное состояние до надежного сохранения данных.

## Входит в план

- Уточнить строгие схемы и проверки связей для `comments.json`.
- Добавить сценарии приложения для получения деталей выбранного обновления, добавления комментария, редактирования комментария, удаления комментария после подтверждения и изменения статуса проверки.
- Показать в карточках ленты количество комментариев и текущий статус проверки.
- Добавить рабочую область деталей выбранного обновления на странице студента.
- Показать данные обновления: результат, ветку, старый коммит, новый коммит, количество новых коммитов и ошибку обновления.
- Добавить форму нового комментария внутри деталей обновления.
- Добавить редактирование существующего комментария.
- Добавить удаление существующего комментария после подтверждения.
- Добавить ручной выбор статуса проверки из разрешенных статусов первой версии.
- Сохранить комментарии и статусы между перезапусками приложения.
- Покрыть сценарии комментариев и статусов тестами.

## Не входит в план

- История редакций комментариев.
- Комментарии к файлам, строкам кода, коммитам или фрагментам diff.
- ИИ-анализ, ИИ-рапорты и черновики комментариев.
- Создание review-копий и открытие кода в VS Code.
- Встроенный просмотр кода или diff.
- Публикация комментариев в GitHub или отправка обратной связи студенту.

## Шаги разработки

1. Проверить текущие схемы данных и добавить строгую схему комментариев, если она еще не реализована полностью.
2. Добавить доменную и прикладную логику комментариев, удаления комментариев и статусов проверки.
3. Расширить загрузку страницы студента данными комментариев, счетчиками и выбранным обновлением.
4. Обновить интерфейс страницы студента: открытие деталей события, форма комментария, редактирование комментария, удаление комментария с подтверждением и выбор статуса.
5. Добавить тесты для добавления комментария, редактирования комментария, удаления комментария, изменения статуса и сохранения после повторного чтения данных.
6. Выполнить `pnpm check` и ручную проверку в браузере.

## Критерии готовности

- Любое событие обновления можно открыть в деталях на странице студента.
- Детали показывают сохраненные данные обновления без изменения исторического факта события.
- Новый комментарий сохраняется в `comments.json` и связан с выбранным `updateEventId`.
- Существующий комментарий можно отредактировать, при этом обновляется `updatedAt`.
- Существующий комментарий можно удалить только после подтверждения.
- После удаления комментария карточка обновления показывает обновленное количество комментариев.
- Пустой комментарий не сохраняется и показывает ошибку рядом с формой.
- Статус проверки выбранного обновления меняется вручную и сохраняется в `review-statuses.json`.
- Карточка обновления в ленте показывает актуальный статус проверки и количество комментариев.
- После перезапуска приложения комментарии и статус проверки остаются видимыми.
- `pnpm check` проходит успешно.

## Пользовательская проверка

Владелец проекта проверяет локально:

1. Создать или использовать студента с сохраненным событием обновления.
2. Открыть страницу студента и раскрыть детали события обновления.
3. Добавить комментарий к событию.
4. Изменить статус проверки на `проверяется`, затем на `проверено` или `требует доработки`.
5. Отредактировать комментарий.
6. Удалить один комментарий после подтверждения и убедиться, что он исчез из деталей обновления.
7. Обновить страницу или перезапустить приложение и убедиться, что оставшиеся комментарии, дата редактирования, счетчик комментариев и статус сохранились.

## Риски и открытые вопросы

- Детали выбранного обновления должны оставаться частью страницы студента, чтобы не усложнять навигацию отдельными маршрутами раньше необходимости.
- История редакций и история удаления комментариев сознательно откладываются, потому что текущий план хранит только актуальный набор личных заметок преподавателя.
- В этом плане нужно не задеть ИИ и VS Code: действия можно показать как недоступные или оставить для следующих планов, но не реализовывать их частично.
