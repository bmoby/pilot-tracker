## Цель

Сделать запуск ИИ-анализа надежным для ежедневной работы: преподаватель должен вручную поставить один или несколько анализов в локальную очередь, перейти на другую страницу и позже увидеть сохраненные успешные или ошибочные ИИ-рапорты. Запуск не должен зависеть от открытой страницы и не должен создавать конфликтующие записи при нескольких анализах.

## Контекст спецификаций

Бизнес-контекст сохраняет ключевое ограничение: ИИ помогает преподавателю, но не запускается автоматически после обновления и не принимает итоговое решение. `Ф004` и `Т009` уже требуют, чтобы начатая попытка ИИ-анализа завершалась успешным или ошибочным ИИ-рапортом. `Т004` требует безопасные записи JSON и сериализацию параллельных операций хранения. `ИП007` требует, чтобы сохраненные ИИ-рапорты оставались доступными после перезапуска. Текущие спецификации не описывают локальную очередь заданий и выполнение вне запроса страницы, поэтому перед реализацией нужно уточнить `Ф004`, `Т003`, `Т004`, `Т007`, `Т009`, `Т010` и пользовательские истории.

## Входит в план

- Добавить локальную модель задания ИИ-анализа.
- Добавить JSON-файл очереди ИИ-заданий в `data/app/`.
- Сохранять запрос анализа как задание до запуска Codex CLI.
- Выполнять задания через локальный worker в процессе приложения с ограничением параллелизма.
- Не запускать ИИ автоматически после обновления проектов.
- Разрешить массовую постановку заданий только по явному действию преподавателя.
- Не создавать дубликат задания для одного обновления, если по нему уже есть ожидающее или выполняющееся задание.
- Сохранять успешный или ошибочный ИИ-рапорт внутри выбранного обновления.
- Показывать состояние очереди в интерфейсе: ожидает, выполняется, готово, ошибка.
- При старте приложения помечать зависшие выполнявшиеся задания как прерванные или готовые к повторному запуску без потери старых рапортов.
- Покрыть хранение очереди и worker тестами.
- По прямой просьбе владельца проекта обновить поведение review-копий: при повторном открытии выбранного обновления принудительно восстанавливать грязную review-копию до `newCommit`, чтобы действие открытия не блокировало проверку кода.

## Не входит в план

- Серверная очередь, облачная база данных или фоновый демон вне локального Next.js-процесса.
- Продолжение анализа после полного выключения приложения, закрытия ноутбука или остановки Node.js-процесса.
- Автоматический запуск ИИ сразу после обновления проектов.
- Выбор модели, альтернативные ИИ-провайдеры или прямой вызов OpenAI API.
- Публикация комментариев в GitHub или автоматическое изменение статуса проверки.
- Полноценная система приоритетов, расписаний, повторов по таймеру и отмены уже запущенного процесса.

## Шаги разработки

1. Уточнить спецификации локальной очереди ИИ-анализа, статусов заданий и поведения после перезапуска.
2. Расширить модель данных и начальное хранилище JSON-файлом очереди.
3. Разделить текущий синхронный запуск ИИ на постановку задания и выполнение одного задания worker-ом.
4. Добавить локальный singleton worker с ограничением параллелизма.
5. Обновить действия интерфейса: запуск выбранного обновления ставит задание в очередь, массовый запуск ставит несколько заданий.
6. Показать состояние очереди на странице студента и в списке студентов там, где это помогает выбрать следующую проверку.
7. Добавить обработку зависших заданий при загрузке хранилища или старте worker-а.
8. Добавить тесты для постановки заданий, защиты от дублей, выполнения успешного задания, ошибки Codex и зависшего задания.
9. Выполнить `pnpm check`.
10. Проверить сценарий в браузере: поставить анализ в очередь, перейти на другую страницу, вернуться к сохраненному рапорту.

## Критерии готовности

- Преподаватель может поставить ИИ-анализ выбранного обновления в очередь и сразу продолжить работу с интерфейсом.
- Преподаватель может явно поставить в очередь несколько подходящих обновлений.
- Одно обновление не получает несколько одновременно ожидающих или выполняющихся заданий.
- Worker выполняет задания с ограниченным параллелизмом и сохраняет результат как ИИ-рапорт.
- Ошибка начатого задания сохраняется как ошибочный ИИ-рапорт.
- После перезапуска зависшие задания не остаются навечно в состоянии `выполняется`.
- Старые ИИ-рапорты не перезаписываются.
- Все пользовательские тексты остаются на русском языке.
- `pnpm check` проходит успешно.

## Пользовательская проверка

Владелец проекта открывает страницу студента, ставит анализ в очередь, переходит на список студентов или настройки, затем возвращается и видит обновленное состояние задания и сохраненный ИИ-рапорт. Затем владелец проекта запускает массовую постановку для нескольких обновлений и проверяет, что интерфейс не блокируется ожиданием каждого Codex-запуска.

## Риски и открытые вопросы

- Локальный worker живет только пока запущен Node.js-процесс приложения. Если приложение полностью остановлено, анализ не продолжится, но очередь должна остаться восстановимой.
- Параллельный запуск нескольких Codex CLI может быть тяжелым для компьютера преподавателя. Начальная безопасная настройка должна ограничивать параллелизм.
- Нужно аккуратно объединять результаты нескольких заданий в `ai-reports.json`, чтобы параллельные сохранения не потеряли чужие записи.
- Массовый запуск должен оставаться явным действием преподавателя, чтобы не нарушить запрет автоматического ИИ после обновления проектов.
