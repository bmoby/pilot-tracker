## Цель

Перевести Pilot Tracker к гибридной модели хранения: прикладные данные курса должны храниться в онлайн-проекте Supabase, а студенческие репозитории, review-копии, локальный ИИ-анализ, временные файлы и машинные настройки должны оставаться на компьютере преподавателя.

## Контекст спецификаций

Текущие спецификации на `main` фиксируют локальное JSON-хранилище как основное хранилище первой версии и прямо говорят, что облачная база данных не входит в первую фазу. Владелец проекта явно согласовал изменение этого решения и хочет начать разработку перехода на онлайн-проект Supabase.

Дополнительное решение от 2026-05-28: рабочей целью является онлайн-проект Supabase. Локальный Supabase через Docker не используется как окружение разработки, проверки миграций или пользовательской работы.

Поэтому первый шаг плана - не код, а обновление спецификаций: нужно добавить `Т011` и привести связанные документы к новой гибридной рамке. Только после этого можно добавлять миграции, подключение Supabase и новый слой хранилища.

## Входит в план

- Добавить и согласовать техническую спецификацию `Т011` для Supabase-хранилища данных приложения.
- Обновить бизнес-контекст и связанные спецификации, чтобы убрать конфликт со старым правилом локальных JSON-файлов.
- Добавить начальную SQL-схему Supabase для студентов, проектов, обновлений, комментариев, статусов проверки, заданий ИИ-анализа и ИИ-рапортов.
- Применить начальную SQL-схему к онлайн-проекту Supabase после настройки доступа Supabase CLI или удаленной Postgres-базы.
- Добавить локальную конфигурацию подключения к Supabase без попадания секретов в Git.
- Добавить серверный модуль подключения к Supabase.
- Добавить первый слой чтения каталога из Supabase без резкого переключения всех рабочих экранов.
- Сохранить существующее JSON-хранилище как временный рабочий механизм до отдельного этапа миграции.
- Проверить типизацию, тесты, линт и production build.

## Не входит в план

- Полное переключение всех экранов и сценариев на Supabase в одном шаге.
- Автоматическая миграция существующих JSON-данных в Supabase.
- Офлайн-режим с последующей синхронизацией.
- Командная работа нескольких преподавателей.
- Студенческий кабинет.
- Хранение кода студентов в Supabase.
- Облачный запуск ИИ.
- Публикация комментариев в GitHub.
- Локальный Supabase через Docker.

## Шаги разработки

1. Добавить `Т011` как согласованную спецификацию Supabase-хранилища.
2. Обновить бизнес-контекст, `Ф005`, `Т001`, `Т003`, `Т004`, `Т008`, `Т009`, `Т010` и технический индекс под гибридную модель.
3. Добавить зависимость Supabase SDK.
4. Создать начальную миграцию Supabase.
5. Добавить `.env.example` с локальными переменными подключения.
6. Добавить серверный модуль подключения к Supabase и тесты конфигурации.
7. Добавить первый класс Supabase-хранилища для чтения каталога.
8. Применить миграцию к онлайн-проекту Supabase.
9. Запустить `typecheck`, тесты, линт и build.
10. После проверки выбрать следующий безопасный этап: подключение списка студентов к Supabase или миграция JSON-данных.

## Критерии готовности

- `Т011` добавлен и согласован как основание для разработки.
- Старые спецификации больше не блокируют разработку Supabase-хранилища.
- В репозитории есть SQL-миграция начальной схемы Supabase.
- Начальная SQL-схема применена к онлайн-проекту Supabase или явно зафиксирован недостающий доступ для применения миграции.
- В репозитории есть пример локальных переменных окружения без секретов.
- В коде есть серверный модуль подключения к Supabase.
- В коде есть первый слой чтения каталога из Supabase.
- Существующие пользовательские сценарии не сломаны резким переключением хранилища.
- Все обязательные проверки проходят.

## Пользовательская проверка

Владелец проекта должен проверить:

- устраивает ли гибридная модель хранения;
- устраивает ли список таблиц Supabase;
- устраивает ли решение, что на новом компьютере данные видны из Supabase, но проекты нужно локально загрузить по GitHub-ссылкам;
- устраивает ли поэтапный перенос без немедленного отключения JSON-хранилища.

## Проверка онлайн-проекта

На 2026-05-28 локальный `.env` указывает на онлайн-проект Supabase. Подключение приложения через `PILOT_SUPABASE_URL` и `PILOT_SUPABASE_SERVICE_ROLE_KEY` проверено без вывода секретов.

В онлайн-проекте проверено наличие таблиц текущей начальной схемы:

- `students`;
- `projects`;
- `update_runs`;
- `update_events`;
- `review_statuses`;
- `comments`;
- `ai_analysis_jobs`;
- `ai_reports`;
- `project_ai_descriptions`.

Применение миграции через `supabase db push` пока нельзя считать полностью закрытым: текущая Supabase CLI-сессия не имеет прав на project ref из `.env`. Для синхронизации истории миграций Supabase CLI нужно войти в аккаунт, которому доступен этот проект, или использовать отдельный безопасный доступ к удаленной Postgres-базе.

## Риски и открытые вопросы

- Онлайн-проект Supabase выбран владельцем проекта и настроен в локальном `.env`, но для применения SQL-миграции нужен доступ Supabase CLI к этому проекту или отдельный пароль удаленной Postgres-базы.
- Нужно отдельно разработать сценарий миграции существующих JSON-данных.
- Нужно отдельно подключить рабочие экраны к Supabase.
- Нужно отдельно определить восстановление результата ИИ-анализа, если локальный Codex завершился, но запись в Supabase не удалась.
