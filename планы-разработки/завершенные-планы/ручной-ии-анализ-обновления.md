## Цель

Добавить первый рабочий контур ручного ИИ-анализа выбранного обновления: преподаватель запускает анализ из деталей обновления, приложение готовит безопасную review-копию, запускает локальный `codex exec` в режиме чтения, сохраняет ИИ-рапорт и показывает его внутри выбранного обновления.

## Контекст спецификаций

Прочитаны бизнес-контекст, `Ф003`, `Ф004`, `Ф005`, `Т003`, `Т006`, `Т008`, `Т009`, `Т010` и `ИП006`. План остается внутри рамок первой локальной версии: ИИ запускается только вручную, использует локальный Codex CLI, не вызывает OpenAI API напрямую, не публикует комментарии, не меняет файлы студенческого проекта и не выставляет статус проверки автоматически.

## Входит в план

- Уточнить схему `ai-reports.json` и связи ИИ-рапортов с обновлениями.
- Добавить application-сценарий запуска ИИ-анализа выбранного обновления.
- Создавать или переиспользовать review-копию выбранного обновления как рабочую папку Codex.
- Выбирать режим анализа автоматически по наличию `previousCommit`.
- Передавать промпт в `codex exec` через stdin и использовать схему результата.
- Сохранять успешный ИИ-рапорт или ошибку начатого анализа внутри выбранного обновления.
- Блокировать запуск без создания ИИ-рапорта, если заранее известно, что Codex CLI недоступен или у обновления нет нужных данных.
- Показать кнопку запуска и список ИИ-рапортов в деталях обновления.
- Добавить тесты запуска, блокировок, ошибок и сохранения нескольких рапортов.

## Не входит в план

- Автоматический запуск ИИ после обновления проекта.
- Автоматическое выставление статуса проверки.
- Публикация комментариев в GitHub или pull request.
- Ручной выбор режима, модели или типа рапорта.
- Полноценная проверка CI/CD студенческого проекта.
- Редактирование или удаление ИИ-рапортов.

## Шаги разработки

1. Уточнить текущие модули review-копий, настроек, Git-интеграции и страницы студента.
2. Добавить строгие схемы ИИ-рапортов и проверку связей в локальном хранилище.
3. Добавить интеграцию запуска Codex CLI с передачей stdin и чтением `--output-last-message`.
4. Добавить сценарий `runAiAnalysisForUpdate` с подготовкой review-копии, сбором контекста и сохранением результата.
5. Обновить страницу студента: кнопка запуска ИИ, состояния недоступности, список рапортов и ошибки.
6. Добавить тесты application-слоя и интеграции Codex CLI через заглушки.
7. Выполнить автоматическую и ручную проверку основного потока.

## Критерии готовности

- ИИ-анализ запускается только из деталей выбранного обновления.
- Для обновления без `newCommit` запуск недоступен и ИИ-рапорт не создается.
- Если последняя диагностика заранее показывает ошибку Codex CLI, запуск недоступен и ИИ-рапорт не создается.
- Начатый запуск сохраняет успешный или ошибочный ИИ-рапорт.
- Повторный запуск создает новый рапорт и не перезаписывает старые.
- Рапорт показывает режим анализа, коммиты, резюме, важные файлы, риски, вопросы и черновик комментария, если они доступны.
- Статус проверки обновления не меняется автоматически.
- Тесты и проверка качества проходят.

## Пользовательская проверка

Владелец проекта открывает страницу студента с обновлением, раскрывает детали обновления, запускает `Запустить ИИ-анализ`, проверяет появление ИИ-рапорта внутри этого обновления, перезагружает страницу и убеждается, что рапорт сохранился. Затем повторно запускает анализ и проверяет, что появился новый рапорт, а старый остался.

## Риски и открытые вопросы

- Реальный запуск `codex exec` может занять заметное время; первая версия выполняет его синхронно в серверном действии.
- Если локальный Codex CLI изменит обязательные флаги, запуск завершится ошибочным ИИ-рапортом или будет заблокирован диагностикой.
- Контекст pull request полезен, но ошибка GitHub CLI не должна блокировать основной локальный анализ.
