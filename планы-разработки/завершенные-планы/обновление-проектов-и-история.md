## Цель

Добавить первый рабочий контур обновления студенческих проектов: преподаватель запускает обновление всех проектов или одного проекта, приложение создает историю обновлений, сохраняет результаты и показывает их в списке студентов и на странице студента.

## Контекст спецификаций

План следует главному потоку первой версии: после создания каталога студентов преподаватель должен вручную обновлять проекты и видеть, где появилась новая работа. `Ф002` и `Т005` задают правила Git-операций, `Т003` и `Т004` задают формат локальной истории, `Т007` задает отображение статусов, сводки и ленты, а `Т010` требует, чтобы ошибка одного проекта не ломала весь запуск.

## Входит в план

- Расширить локальные данные и сценарии приложения для `update-runs.json`, `update-events.json` и начальных статусов проверки обновлений.
- Реализовать сервис обновления всех проектов и одного проекта по правилам ветки `main`.
- Клонировать новые репозитории в `data/repositories/<student-id>/<project-id>/`.
- Обновлять существующую чистую локальную копию через `fetch`, проверку `origin/main` и приведение к `origin/main`.
- Сохранять события для первой загрузки, новых изменений, отсутствия изменений, отсутствующей GitHub-ссылки и ошибок Git.
- Обновлять текущий статус проекта, последний известный коммит, локальный путь и последнюю ошибку.
- Показать действие общего обновления, состояние выполнения, последнюю сводку и статусы в списке студентов.
- Добавить страницу студента с верхним контекстом проекта, одиночным обновлением и лентой событий обновления.
- Покрыть доменную логику, хранилище и сценарии обновления тестами без реальных сетевых вызовов.

## Не входит в план

- Ручные комментарии преподавателя к обновлению.
- Ручное изменение статуса проверки обновления.
- ИИ-анализ, ИИ-рапорты и краткое ИИ-описание проекта.
- Создание review-копий и открытие кода в VS Code.
- Получение diff, списка измененных файлов, pull request или комментариев GitHub.
- Поддержка веток кроме `main`.
- Автоматическое восстановление поврежденных или грязных локальных копий.

## Шаги разработки

1. Проверить текущие контракты данных и расширить типы, начальные файлы и операции хранилища для запусков и событий обновления.
2. Добавить доменную логику результатов обновления, сводки запуска, статусов проекта и пользовательских сообщений ошибок.
3. Добавить Git-адаптер, который запускает команды через массив аргументов без shell-интерполяции.
4. Реализовать сценарии общего и одиночного обновления с сохранением истории и частичных ошибок.
5. Обновить экран студентов: кнопка общего обновления, сводка последнего запуска, статусы, локальный путь и ошибки.
6. Добавить страницу студента с лентой обновлений и действием одиночного обновления.
7. Добавить тесты и пройти локальную проверку.

## Критерии готовности

- Общий запуск создает один `update-run` и событие для каждого студента.
- Студент без GitHub-ссылки получает событие `skipped_no_repository`, а общий запуск продолжается.
- Первая успешная загрузка сохраняет локальный путь, ветку `main`, новый коммит и статус первой загрузки.
- Повторное обновление сохраняет событие с новыми коммитами или без изменений.
- Ошибка одного проекта сохраняется в событии и не останавливает остальные проекты.
- Список студентов показывает последнюю сводку, статусы проектов, локальные пути, коммиты и ошибки.
- Страница студента показывает верхний контекст и ленту событий после перезапуска приложения.
- Тесты проходят командой `pnpm check`.

## Пользовательская проверка

Владелец проекта проверяет локально:

1. Создать студента без GitHub-ссылки и запустить общее обновление.
2. Убедиться, что в списке появилась сводка, а у студента сохранено событие о неподключенном проекте.
3. Добавить студенту тестовую GitHub-ссылку с веткой `main`.
4. Запустить общее обновление и проверить первую загрузку в `data/repositories/`.
5. Повторно запустить обновление и проверить событие без новых изменений.
6. Открыть страницу студента и убедиться, что лента обновлений сохраняется после перезапуска приложения.

## Риски и открытые вопросы

- Реальные приватные репозитории зависят от локальной Git-аутентификации преподавателя; приложение только сохраняет ошибку доступа.
- Сетевые ошибки Git могут отличаться по тексту на разных машинах, поэтому классификация должна быть понятной, но не чрезмерно хрупкой.
- Полный диагностический журнал можно расширять в следующих планах; в этом плане критично сохранить пользовательскую ошибку в истории обновления.
