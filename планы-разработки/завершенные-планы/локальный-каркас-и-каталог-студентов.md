## Цель

Создать первый проверяемый вертикальный срез Pilot Tracker: локальное Next.js-приложение открывается на `localhost`, создает базовую структуру `data/`, показывает главный экран `Студенты`, позволяет создать, изменить и удалить студента со связанным проектом, а данные сохраняются в JSON-файлах после перезапуска приложения.

## Контекст спецификаций

План остается внутри рамок первой локальной версии из бизнес-контекста: один преподаватель, одна активная когорта, локальное хранение, без авторизации, облачной базы данных, серверного развертывания, студенческого кабинета, LMS и экспорта.

`Ф001` задает главный экран первой версии: список студентов с именем, дополнительной информацией, GitHub-ссылкой или признаком ее отсутствия, связанным проектом и текущим статусом проекта. Студента можно создать без ссылки GitHub, добавить ссылку позже, изменить карточку и удалить запись только после подтверждения.

`Ф005`, `Т003` и `Т004` требуют локальные JSON-файлы внутри `data/app/`, создание структуры `data/`, безопасную запись, чтение после перезапуска и отсутствие молчаливого перезаписывания поврежденных данных. Добавление студента должно создавать связанный проект по правилу `student 1 -> 1 project`.

`Т001` и `Т002` требуют разделить интерфейс, сценарии приложения, доменную модель и хранилище. Интерфейс Next.js должен быть адаптером над сценариями приложения и не должен напрямую читать или записывать JSON-файлы.

`Т007` делает экран `Студенты` первым рабочим экраном приложения и требует рабочие пустые состояния, форму добавления и редактирования, видимые ошибки сохранения и подтверждение удаления. `Т008` учитывается на уровне начальных настроек и структуры данных, но полноценный экран диагностики окружения в этот план не входит.

`Т010` задает обработку ошибок хранения, валидации и интерфейсных сценариев: действие не считается успешным, пока данные надежно не сохранены, пользовательские сообщения должны быть понятны преподавателю, техническая диагностика не должна содержать секреты.

Истории `ИП001`, `ИП002`, `ИП007` и `ИП008` определяют проверяемый пользовательский поток этого плана: первое открытие, создание рабочей когорты, сохранение после перезапуска и защищенное удаление студента.

## Входит в план

- Инициализировать локальное приложение на Next.js, React и TypeScript с менеджером пакетов `pnpm`.
- Настроить базовую структуру кода `app/`, `src/application/`, `src/domain/`, `src/storage/`, `src/ui/`, `src/config/`, `src/diagnostics/`, `src/shared/`.
- Добавить команды разработки `pnpm dev`, `pnpm build`, `pnpm test`, `pnpm lint`, `pnpm typecheck`, `pnpm format`, `pnpm check` в согласованном виде для текущего этапа.
- Реализовать доменные типы и Zod-схемы для студентов, проектов, начальных JSON-файлов и минимальных настроек.
- Реализовать инициализацию структуры `data/`: `data/app/`, `data/repositories/`, `data/review-copies/`, `data/logs/`, `data/backups/`.
- Создавать отсутствующие начальные JSON-файлы в `data/app/`: `students.json`, `projects.json`, `update-runs.json`, `update-events.json`, `comments.json`, `review-statuses.json`, `ai-reports.json`, `settings.json`.
- Реализовать чтение, валидацию и безопасную запись JSON-файлов с переводом строки в конце файла и отступом в два пробела.
- Реализовать сценарии приложения для списка студентов, создания студента, редактирования студента и удаления студента после подтверждения.
- При создании студента создавать связанный проект со статусом `not_connected`, если GitHub-ссылка не указана, или `never_updated`, если ссылка указана.
- Реализовать проверку имени студента и формальную проверку GitHub-ссылки без проверки доступа к репозиторию.
- Реализовать главный экран `Студенты` с пустым состоянием, списком, статусом проекта, GitHub-ссылкой или признаком отсутствующей ссылки, формой добавления и редактирования, подтверждением удаления и ошибками сохранения.
- Добавить минимальную навигацию с разделами `Студенты` и `Настройки`, где `Настройки` могут показывать только начальные пути и нейтральное состояние будущей диагностики.
- Добавить автоматические тесты для доменных правил, валидации GitHub-ссылки, создания связанного проекта, инициализации хранилища, чтения и записи JSON, сохранения студентов после повторной загрузки.
- Провести ручную проверку локального потока через браузер: создать, изменить, удалить студента и убедиться, что данные сохраняются после перезапуска.

## Не входит в план

- Клонирование, обновление и сравнение студенческих репозиториев.
- Общий запуск обновления проектов и создание реальных событий обновления.
- Страница конкретного студента с лентой обновлений.
- Комментарии преподавателя и статусы проверки обновлений.
- Review-копии и открытие кода в VS Code.
- Запуск локального ИИ-анализа через Codex CLI.
- Получение контекста pull request через GitHub CLI.
- Полноценная диагностика Git, GitHub CLI, Codex CLI и VS Code.
- Импорт списка студентов, поиск, фильтры, теги, группы, несколько курсов, архивирование и экспорт.
- Физическое удаление папок локальных репозиториев или review-копий при удалении студента.

## Шаги разработки

1. Подготовить каркас Next.js-приложения, зависимости, TypeScript, Tailwind CSS, базовую навигацию и команды `pnpm`.
2. Создать структуру слоев `src/` и зафиксировать направление зависимостей: интерфейс вызывает сценарии приложения, сценарии используют домен и хранилище.
3. Описать доменные типы, статусы проекта, начальные настройки и Zod-схемы для студентов, проектов и JSON-файлов.
4. Реализовать локальное хранилище: создание `data/`, создание начальных JSON-файлов, чтение, валидация и безопасная запись.
5. Реализовать сценарии приложения для списка студентов, создания, редактирования и удаления с сохранением связанных проектов.
6. Реализовать главный экран `Студенты`: пустое состояние, список, форма добавления и редактирования, подтверждение удаления, состояния сохранения и ошибки.
7. Реализовать начальный экран `Настройки` с отображением путей данных и нейтральным состоянием диагностики без запуска внешних инструментов.
8. Добавить модульные и интеграционные тесты для домена, сценариев и хранилища на временной папке данных.
9. Запустить проверки `pnpm check` или согласованный набор `pnpm typecheck`, `pnpm lint`, `pnpm test`, `pnpm build`.
10. Запустить локальное приложение, проверить пользовательский поток в браузере и зафиксировать результат ручной проверки.

## Критерии готовности

- Приложение запускается локально через `pnpm dev` и открывает рабочий экран `Студенты`.
- При первом запуске создается структура `data/` и начальные JSON-файлы в `data/app/`.
- Интерфейс показывает пустое состояние списка студентов без маркетингового экрана, авторизации или выбора курса.
- Преподаватель может создать студента только с именем и дополнительной информацией без GitHub-ссылки.
- Преподаватель может создать студента с GitHub-ссылкой, если ссылка выглядит как ссылка на репозиторий.
- Для каждого студента создается ровно один связанный проект.
- Студент без GitHub-ссылки получает статус проекта `not_connected`.
- Студент с GitHub-ссылкой получает статус проекта `never_updated`.
- Преподаватель может изменить имя, дополнительную информацию и GitHub-ссылку до появления локального пути или последнего известного коммита.
- Преподаватель может удалить студента только после подтверждения.
- Удаление студента меняет JSON-данные приложения, но не удаляет физические папки `data/repositories/` и `data/review-copies/`.
- После перезапуска приложения сохраненные студенты и проекты читаются из JSON-файлов и отображаются в списке.
- Поврежденный или невалидный JSON-файл не перезаписывается пустой структурой автоматически.
- Интерфейс не читает и не пишет JSON-файлы напрямую, а работает через сценарии приложения.
- Основные проверки типов, тестов, lint и сборки проходят или для каждого непрошедшего пункта зафиксирована причина.

## Пользовательская проверка

1. Запустить приложение командой `pnpm dev`.
2. Открыть локальный адрес приложения в браузере.
3. Убедиться, что первым экраном открыт список `Студенты` и он показывает пустое рабочее состояние.
4. Добавить студента без GitHub-ссылки и увидеть статус проекта `проект не подключен`.
5. Добавить студента с GitHub-ссылкой и увидеть статус проекта `проект еще не обновлялся`.
6. Попробовать сохранить студента без имени и увидеть ошибку валидации без создания записи.
7. Отредактировать имя, заметку и GitHub-ссылку существующего студента.
8. Удалить одного студента через подтверждение.
9. Остановить приложение, запустить его снова и убедиться, что оставшиеся студенты сохранились.
10. Проверить, что в `data/app/students.json` и `data/app/projects.json` есть связанные записи, а остальные начальные JSON-файлы созданы пустыми структурами.

## Риски и открытые вопросы

- На старте репозиторий пока содержит спецификации и рабочие документы, поэтому инициализация Next.js может затронуть много инфраструктурных файлов.
- Если выбранные версии Next.js, React, Tailwind CSS, ESLint или Vitest создадут конфликтующие настройки, нужно сохранить требование `Т002`: строгий TypeScript, `pnpm` и разделение слоев важнее генераторных значений по умолчанию.
- Полноценная диагностика локальных инструментов отложена, поэтому экран `Настройки` в этом плане не должен обещать готовность Git, GitHub CLI, Codex CLI или VS Code.
- Удаление студента в этом плане нужно реализовать так, чтобы не закрыть будущие требования по истории обновлений, комментариям, статусам и ИИ-рапортам.
- Файл `data/` может появиться при ручной проверке. Нужно убедиться, что в репозиторий попадают только нужные исходные файлы, а пользовательские локальные данные остаются вне коммита, если это будет закреплено в реализации.
