## Цель

Перевести стандартный ИИ-анализ выбранного обновления из технического пересказа diff и файлов в короткий человеческий обзор для преподавателя: что студент добавил как функциональность, что можно быстро проверить руками, какие вопросы задать студенту и какой комментарий можно взять за основу. Одновременно запретить запуск ИИ-анализа для повторных обновлений без новых коммитов.

## Контекст спецификаций

Прочитаны бизнес-контекст, `Ф002`, `Ф003`, `Ф004`, `Ф005`, `Т003`, `Т009`, `Т010` и `ИП006`. Спецификации уточнены в этой же ветке по явному решению владельца проекта до составления плана.

План остается внутри рамок первой локальной версии: ИИ-анализ запускается только вручную, использует локальный Codex CLI, не вызывает OpenAI API напрямую, не публикует комментарии, не меняет файлы студенческого проекта и не выставляет статус проверки автоматически.

## Входит в план

- Заблокировать запуск ИИ-анализа для повторного обновления без новых коммитов.
- Оставить запуск ИИ-анализа доступным для первой загрузки проекта.
- Оставить запуск ИИ-анализа доступным для обновления с новыми коммитами.
- Обновить причину недоступности запуска в данных страницы студента и пользовательском интерфейсе.
- Обновить JSON-схему ответа Codex под человекочитаемый рапорт без отдельных разделов `Как это влияет на проект` и `Куда движется студент`.
- Обновить промпт ИИ-анализа так, чтобы основной текст объяснял добавленную функциональность и ручные проверки простым языком.
- Оставить технические детали, важные файлы, коммиты и pull request как справку для проверяемости, а не как главный текст рапорта.
- Обновить отображение ИИ-рапортов на странице студента под новую структуру.
- Обновить тесты application-слоя для блокировки обновлений без новых коммитов и нового формата рапорта.

## Не входит в план

- Автоматический запуск ИИ после обновления проекта.
- Автоматическая оценка студента или изменение статуса проверки.
- Публикация комментариев в GitHub или pull request.
- Ручной выбор режима, модели или типа рапорта.
- Отдельный глубокий code review, проверка CI/CD или анализ качества архитектуры.
- Новые разделы рапорта про влияние обновления на проект или направление движения студента.
- Миграция старых ИИ-рапортов в новый формат, если их можно продолжать читать без потери истории.

## Шаги разработки

1. Проверить текущую реализацию `runAiAnalysisForUpdate`, схему `ai-report-output-schema.json`, промпт и отображение ИИ-рапортов на странице студента.
2. Добавить единое правило доступности ИИ-анализа: первая загрузка или обновление с новыми коммитами.
3. Обновить server action и данные страницы студента, чтобы интерфейс показывал понятную причину блокировки для обновлений без новых коммитов.
4. Обновить схему ответа Codex и нормализацию результата под поля человеческого рапорта.
5. Переписать промпт ИИ-анализа: простой язык, добавленная функциональность, быстрая ручная проверка, вопросы студенту и черновик комментария.
6. Обновить UI списка ИИ-рапортов: главный текст рапорта выше, техническая справка ниже и визуально вторична.
7. Обновить и добавить тесты для доступности запуска, сохранения нового формата, блокировки обновления без новых коммитов и отсутствия автоматического изменения статуса проверки.
8. Выполнить автоматическую проверку качества.

## Критерии готовности

- Для повторного обновления без новых коммитов кнопка ИИ-анализа недоступна и новый ИИ-рапорт не создается.
- Для первой загрузки проекта ИИ-анализ остается доступен.
- Для обновления с новыми коммитами ИИ-анализ остается доступен.
- Рапорт содержит краткое резюме, добавленную функциональность, пункты быстрой ручной проверки, вопросы студенту и черновик комментария.
- В стандартном рапорте нет отдельных разделов про влияние обновления на проект и направление движения студента.
- Технические сведения не являются главным содержанием рапорта и показываются как справка.
- Старые сохраненные ИИ-рапорты не удаляются и не перезаписываются.
- Статус проверки обновления не меняется автоматически.
- Автоматические тесты и проверка качества проходят.

## Пользовательская проверка

Владелец проекта проверяет три обновления на странице студента:

1. Первая загрузка проекта: запуск ИИ доступен, рапорт описывает проект человеческим языком.
2. Обновление с новыми коммитами: запуск ИИ доступен, рапорт объясняет, что добавлено как функциональность и что быстро проверить руками.
3. Повторное обновление без новых коммитов: запуск ИИ недоступен, интерфейс объясняет, что новой работы для анализа нет, и новый ИИ-рапорт не появляется.

После этого владелец проекта проверяет, что технические детали рапорта не занимают главное место и что черновик комментария можно использовать как основу для обратной связи.

## Риски и открытые вопросы

- Текущая модель данных уже содержит поля `importantFiles`, `changes`, `risks` и `manualReviewQuestions`; в рамках плана нужно решить, достаточно ли переиспользовать их с новым смыслом или лучше добавить новые поля.
- Если добавить новые поля в `ai-reports.json`, может понадобиться мягкая совместимость со старыми ИИ-рапортами.
- Codex может все равно возвращать слишком технический текст; промпт и схема должны явно ограничить стиль ответа.
- Интерфейс должен аккуратно показать старые и новые рапорты, если их структура отличается.
