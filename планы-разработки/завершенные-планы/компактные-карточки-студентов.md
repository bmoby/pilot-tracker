## Цель

Сделать главный список студентов более компактным и рабочим: убрать из карточек сведения, которые преподаватель не использует при выборе следующей проверки, сделать GitHub-ссылку кликабельной и сократить верхнюю сводку страницы.

## Контекст спецификаций

Прочитан бизнес-контекст: главный экран первой версии должен помогать преподавателю быстро выбрать следующую проверку, увидеть новые изменения, ошибки и неподключенные проекты.

`Ф001` обновлена так, чтобы компактная карточка общего списка показывала только полезные для выбора проверки сведения. Локальный путь, внутренний идентификатор и последний коммит не должны отображаться на карточке списка, но остаются доступными в более подробном контексте.

`Ф002` сохраняет требование показывать сводку общего обновления, но не требует крупных постоянных счетчиков студентов.

`Т007` обновлена так, чтобы интерфейс списка студентов использовал компактную сводку, кликабельную GitHub-ссылку и одну понятную зону действий карточки.

`ИП002` уточнена под компактную карточку рабочей когорты.

## Входит в план

- Обновить действующие спецификации под компактную карточку студента.
- Убрать из карточки общего списка внутренний идентификатор, локальный путь проекта и последний коммит.
- Сделать GitHub-ссылку в карточке кликабельной.
- Сохранить видимость статуса, даты последнего обновления, признаков первой загрузки, новых изменений и ошибок.
- Сократить верхние счетчики и сводку так, чтобы они не занимали крупную отдельную область без рабочей пользы.
- Убедиться, что действия открытия, редактирования и удаления остаются доступными без двух конкурирующих меню.
- Проверить результат тестами и локальным просмотром интерфейса, если приложение запускается.

## Не входит в план

- Изменение модели данных студентов и проектов.
- Удаление локального пути или коммита из сохраненных данных.
- Изменение страницы конкретного студента, кроме случаев, когда это нужно для сохранения навигации.
- Добавление поиска, тегов, групп, архивов или новых фильтров.
- Изменение логики обновления проектов через Git.
- Автоматическое открытие GitHub через GitHub CLI вместо обычной ссылки.

## Шаги разработки

1. Зафиксировать изменения спецификаций и пользовательской истории для компактной карточки.
2. Найти компоненты главного списка студентов и текущую структуру карточки.
3. Упростить верхнюю область страницы списка студентов.
4. Упростить карточку студента и убрать лишние технические строки.
5. Сделать GitHub-ссылку кликабельной и сохранить состояние отсутствующей ссылки.
6. Проверить, что действия карточки остаются доступны и не дублируются.
7. Запустить релевантные проверки проекта.
8. Открыть локальный интерфейс и проверить экран списка студентов в браузере, если dev-сервер запускается.

## Критерии готовности

- В карточке студента общего списка нет внутреннего идентификатора, локального пути проекта и хеша последнего коммита.
- GitHub-ссылка на карточке кликабельна, если репозиторий указан.
- Если GitHub-ссылки нет, карточка показывает короткое рабочее состояние отсутствующего репозитория.
- Статус проекта, дата последнего обновления, новые коммиты, первая загрузка и ошибки остаются видимыми.
- Верхняя область списка не содержит три крупные постоянные карточки `Всего студентов`, `С репозиторием`, `Без репозитория`.
- Сводка последнего обновления остается доступной в компактном виде.
- Открытие студента, редактирование и удаление остаются доступны.
- Проверки проекта проходят или все непройденные проверки описаны.

## Пользовательская проверка

Владелец проекта открывает главный список студентов и проверяет:

- карточки стали компактнее и не показывают бесполезные технические детали;
- GitHub-репозиторий на карточке можно открыть кликом;
- по карточке все еще понятно, кого нужно проверять следующим;
- верхняя часть страницы занимает меньше места и не дублирует очевидные счетчики;
- действия открытия, редактирования и удаления студента остаются понятными.

## Риски и открытые вопросы

- Если существующие данные для локального просмотра пустые, визуальную проверку нужно выполнить на пустом состоянии и, при необходимости, через уже имеющиеся тестовые данные.
- Если dev-сервер или браузерная проверка не запустятся из-за окружения, это нужно явно зафиксировать в результате работы.
