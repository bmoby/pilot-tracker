## Цель

Добавить защищенный вход администратора через Supabase Auth, чтобы преподаватель мог открыть Pilot Tracker с другого компьютера через веб-интерфейс, войти по логину и паролю и работать с теми же прикладными данными Supabase без раскрытия секретных ключей в браузере.

## Контекст спецификаций

Текущие спецификации первой версии прямо запрещают авторизацию внутри интерфейса, серверное развертывание и роли пользователей. Поэтому реализация не может начаться с кода. Сначала нужно явно обновить продуктовую и техническую рамку: Pilot Tracker остается инструментом одного преподавателя и не становится студенческим кабинетом или командным продуктом, но получает защищенный администраторский доступ к рабочим данным.

`Ф005` и `Т011` уже фиксируют Supabase как источник истины для прикладных данных и разделяют данные курса от локальных репозиториев, review-копий, Codex CLI, Git и VS Code. Новый план должен сохранить это разделение: удаленно доступны прикладные данные, а локальные действия остаются зависящими от конкретного компьютера.

## Входит в план

- Обновить бизнес-контекст и связанные спецификации под новый режим защищенного администраторского доступа.
- Добавить новую техническую спецификацию `Т012` для Supabase Auth, RLS, cookie-сессий и admin-only доступа.
- Добавить или обновить пользовательскую историю входа администратора.
- Обновить дизайн-карту для страницы входа, состояния выхода и защищенного доступа до изменения рабочего интерфейса.
- Реализовать страницу входа и выход администратора после подтверждения дизайн-карты владельцем проекта.
- Подключить Supabase Auth для Next.js SSR через cookie-сессии.
- Защитить рабочие маршруты приложения от неавторизованного доступа.
- Перевести рабочий Supabase-клиент с постоянного `service_role` потока на безопасный request-scoped доступ там, где данные открываются пользователю.
- Включить RLS и политики доступа для таблиц Pilot Tracker.
- Обновить `.env.example` без секретных значений.
- Добавить автоматические проверки для сценариев входа, выхода и отказа доступа без сессии.

## Не входит в план

- Несколько ролей пользователей.
- Командная работа нескольких преподавателей.
- Студенческий кабинет.
- Управление курсами, семестрами или архивами.
- Удаленный запуск Git, VS Code, Codex CLI или GitHub CLI на сервере.
- Хранение кода студентов в Supabase.
- Публикация комментариев студентам или в GitHub.
- Автоматическая оценка проектов.

## Шаги разработки

1. Уточнить спецификации: бизнес-контекст, `Ф005`, `Т001`, `Т002`, `Т007`, `Т010`, `Т011`, план технических спецификаций и пользовательские истории.
2. Создать `Т012` как источник истины для Supabase Auth, RLS и защищенного администраторского доступа.
3. Обновить дизайн-карту новым экраном входа и состояниями доступа, затем остановиться для подтверждения владельцем проекта.
4. После подтверждения дизайн-карты добавить зависимости и серверные утилиты Supabase SSR.
5. Реализовать страницу входа, серверные действия входа и выхода, защищенный layout или middleware/proxy.
6. Настроить Supabase Auth и RLS-политики для таблиц приложения.
7. Адаптировать слой хранения под авторизованный запрос и исключить использование `service_role` в пользовательском потоке.
8. Добавить тесты и выполнить `pnpm check`.
9. Провести ручную проверку: без входа, успешный вход, выход, перезагрузка, доступ с чистой сессией.

## Критерии готовности

- Спецификации больше не противоречат администраторскому входу и явно ограничивают область одним администратором-преподавателем.
- В дизайн-карте есть экран входа и состояния защищенного доступа, подтвержденные владельцем проекта.
- Без сессии рабочие страницы недоступны и ведут на вход.
- После входа администратор видит рабочий список студентов из Supabase.
- После выхода рабочие данные снова недоступны.
- Секретный `service_role` ключ не попадает в браузер и не используется в пользовательском клиентском потоке.
- RLS включен для таблиц Pilot Tracker, а политики не открывают данные неавторизованному пользователю.
- Публичная регистрация отсутствует.
- `pnpm check` проходит.

## Пользовательская проверка

Владелец проекта должен сначала открыть `/design-map` и подтвердить экран входа. После реализации он должен проверить: переход на вход без сессии, успешный вход администратора, открытие списка студентов после входа, выход из аккаунта и недоступность рабочих данных после выхода.

## Риски и открытые вопросы

- В спецификации выбран способ фиксации администратора через `app_metadata.pilot_tracker_role = "admin"`. Для безопасности нельзя опираться на `user_metadata`; перед пользовательской проверкой нужно вручную назначить этот признак учетной записи администратора в Supabase.
- Нужно решить, будет ли проект оставаться локально запускаемым через `localhost` или появится первое внешнее развертывание Next.js. Это влияет на cookie-настройки, Supabase Auth redirect URL и эксплуатацию локальных инструментов.
- В текущем коде рабочий слой использует `service_role` на сервере. Для защищенного веб-доступа нужно аккуратно разделить служебные миграции и пользовательский authenticated-доступ.
- Удаленный компьютер сможет видеть данные Supabase, но не сможет выполнять локальные операции с Git, VS Code и Codex CLI, пока на нем не настроены локальные инструменты и репозитории.
