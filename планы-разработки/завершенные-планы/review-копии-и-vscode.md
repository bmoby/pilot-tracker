## Цель

Добавить рабочий контур открытия кода выбранного обновления в VS Code: преподаватель нажимает действие в деталях обновления, приложение создает или переиспользует безопасную review-копию на сохраненном `newCommit`, открывает именно эту папку в VS Code и не переключает основную локальную копию студенческого репозитория с `main`.

## Контекст спецификаций

Бизнес-контекст требует, чтобы преподаватель мог открыть код выбранного обновления для ручной проверки, но Pilot Tracker не должен отправлять изменения в студенческие репозитории и не должен автоматически принимать решение за преподавателя.

`Ф003` и `ИП005` задают пользовательское действие открытия кода из деталей выбранного обновления. Версией обновления считается сохраненный `newCommit`; если его нет, действие недоступно и интерфейс объясняет причину.

`Т006` уже полностью описывает нужное поведение: открывать не основную копию из `data/repositories/`, а review-копию в `data/review-copies/<student-id>/<update-id>/`, предпочтительно через Git worktree в detached `HEAD`. Поэтому отдельное изменение спецификаций для этого плана не требуется.

`Т005` защищает основную локальную копию как техническую копию ветки `main`. План не должен добавлять checkout исторических коммитов, commit, push, merge или rebase в студенческих репозиториях.

`Т003` и `Т004` требуют сохранять историю обновлений как исторический факт. Путь review-копии можно вычислять из `studentId` и `updateEventId`, не редактируя завершенное событие обновления. Если реализации потребуется служебное сохранение состояния открытия, оно должно быть отдельным техническим состоянием или диагностикой, а не изменением смысла события обновления.

`Т008` задает настройки команд `git` и `code`, а `Т010` требует понятные пользовательские ошибки, безопасный запуск процессов без shell-интерполяции и тесты для review-копий и открытия VS Code через заглушку команды.

## Входит в план

- Добавить прикладной сценарий открытия кода выбранного обновления в VS Code.
- Добавить Git-операции для подготовки review-копии через worktree: проверка основного репозитория, проверка доступности `newCommit`, создание worktree в detached `HEAD`, чтение `HEAD` и проверка чистоты существующей копии.
- Вычислять путь review-копии внутри `data/review-copies/<student-id>/<update-id>/` и проверять, что он не выходит за пределы базовой папки review-копий.
- Переиспользовать существующую review-копию, если она чистая и указывает на нужный `newCommit`.
- Безопасно пересоздавать только чистую несоответствующую review-копию.
- Блокировать открытие грязной или нераспознанной review-копии без удаления и показывать путь для ручного решения.
- Добавить адаптер запуска VS Code через настроенную команду `code` с путем review-копии как отдельным аргументом.
- Показать действие открытия кода в деталях выбранного обновления на странице студента.
- Отключать действие или показывать понятную ошибку, если у обновления нет `newCommit`, нет локального репозитория, коммит отсутствует локально, Git недоступен, review-копия некорректна или VS Code недоступен.
- Сохранить ручное изменение статуса проверки: открытие VS Code не должно автоматически менять статус обновления.
- Покрыть сценарий модульными и интеграционными тестами с управляемыми Git- и VS Code-заглушками.
- Проверить результат командой `pnpm check` и ручным локальным сценарием в браузере.

## Шаги разработки

5. Добавить нормализацию ошибок review-копии и VS Code в пользовательские сообщения на русском языке.

7. Добавить тесты для успешного создания review-копии, повторного открытия чистой копии, отсутствующего `newCommit`, отсутствующего локального репозитория, отсутствующего коммита, грязной копии, ошибки создания worktree и ошибки запуска VS Code.
8. Выполнить `pnpm check`.
9. Запустить локальный интерфейс и вручную проверить открытие обновления в VS Code или понятную ошибку, если команда `code` недоступна в окружении.

## Критерии готовности

- В деталях выбранного обновления есть действие открытия кода в VS Code.
- Для обновления с известным `newCommit` приложение создает review-копию внутри `data/review-copies/<student-id>/<update-id>/`.
- Review-копия указывает на сохраненный `newCommit` выбранного обновления.
- Основная локальная копия студенческого проекта остается на `main` и не переводится в detached `HEAD`.
- Повторное открытие чистой review-копии с правильным `HEAD` переиспользует ее.
- Чистая review-копия с неправильным `HEAD` безопасно пересоздается внутри базовой папки review-копий.
- Грязная review-копия не удаляется и не перезаписывается автоматически.
- Если у обновления нет `newCommit`, действие недоступно или возвращает понятную причину без создания review-копии.
- Если нужного коммита нет локально, приложение показывает понятную ошибку и не выполняет автоматический `fetch` только ради открытия старого обновления.
- Если VS Code недоступен, приложение показывает ошибку и путь review-копии, если она была подготовлена.
- Открытие VS Code не меняет статус проверки и не создает новое событие обновления.
- Процессы Git и VS Code запускаются через массив аргументов без сборки shell-строки из пользовательских данных.
- Проверки проекта проходят или все непройденные проверки явно описаны.

## Пользовательская проверка

Владелец проекта проверяет локально:

1. Открыть студента с успешным событием обновления, у которого есть `newCommit`.
2. Открыть детали этого обновления и нажать действие открытия кода в VS Code.
3. Убедиться, что открыта папка из `data/review-copies/<student-id>/<update-id>/`, а не основная копия из `data/repositories/`.
4. Убедиться, что открытая папка соответствует нужному коммиту выбранного обновления.
5. Убедиться, что основная локальная копия студенческого проекта осталась на `main`.
6. Повторно нажать действие открытия и убедиться, что чистая существующая review-копия переиспользуется.
7. Открыть обновление без `newCommit` или событие ошибки и убедиться, что действие недоступно или показывает понятную причину.
8. Проверить, что после открытия VS Code статус проверки обновления не изменился автоматически.

## Риски и открытые вопросы

- На машине преподавателя команда `code` может быть недоступна в `PATH`; в этом случае реализация должна показать полезную ошибку и путь, который можно открыть вручную.
- Git worktree может не создаться, если основной репозиторий поврежден, не содержит нужный коммит или имеет ограничение локального Git; ошибка должна быть безопасной и не должна менять основную копию.
- Грязные review-копии могут содержать ручную работу преподавателя, поэтому их нельзя удалять автоматически даже если они мешают повторному открытию.
- Если текущая реализация настроек еще не поддерживает редактирование команды VS Code, план должен использовать значение по умолчанию `code` и оставить расширение экрана настроек для отдельного плана, если оно окажется крупнее безопасного объема.
