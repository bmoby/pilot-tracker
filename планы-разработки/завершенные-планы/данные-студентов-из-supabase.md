## Цель

Сделать так, чтобы рабочий интерфейс использовал онлайн-проект Supabase как источник прикладных данных, а не локальные JSON-файлы.

## Контекст спецификаций

Бизнес-контекст, `Ф005` и `Т011` уже фиксируют онлайн-проект Supabase как целевой источник истины для прикладных данных курса. `Т011` отдельно требует сначала мигрировать старые JSON-данные, сверить количество записей и только после успешной проверки переключать приложение на Supabase.

`Ф001` требует сохранить рабочий сценарий списка студентов: создание, редактирование, удаление после подтверждения, отображение GitHub-ссылки, статуса проекта, последнего обновления и статуса проверки последнего обновления. Из-за того, что список студентов зависит от истории обновлений, комментариев, статусов проверки и ИИ-заданий, переключение должно выполняться на уровне общего `AppStorage`, а не отдельного read-only каталога.

## Входит в план

- Добавить сценарий миграции текущих JSON-данных из `data/app/` в Supabase с пробным режимом.
- Сохранить исходные идентификаторы студентов, проектов, обновлений, комментариев, статусов проверки, ИИ-заданий и ИИ-рапортов.
- Записать данные в Supabase в порядке, который не нарушает внешние ключи.
- Сверить количество записей после миграции.
- Подключить общий слой `AppStorage` к Supabase для чтения и записи прикладных данных.
- Сделать так, чтобы список студентов, карточка студента, обновления проектов, комментарии, статусы проверки, ИИ-задания и ИИ-рапорты работали через Supabase по умолчанию.
- Оставить локальное JSON-хранилище как источник миграции и временную поддержку тестов.
- Проверить, что секретный ключ Supabase не попадает в клиентский код.

## Не входит в план

- Автоматический офлайн-режим с последующей синхронизацией.
- Отдельный пользовательский интерфейс для различения онлайн-статуса проекта и отсутствующей локальной копии на новом компьютере.
- Изменение схемы Supabase сверх того, что нужно для уже согласованной миграции.
- Хранение кода студентов в Supabase.
- Авторизация внутри интерфейса.

## Шаги разработки

1. Проверить доступность таблиц Supabase и текущее количество записей.
2. Добавить mapper из локальной `AppData`-модели в строки Supabase.
3. Добавить CLI-сценарий миграции с пробным режимом, режимом `--write` и режимом строгого выравнивания `--replace`.
4. Выполнить пробную миграцию и затем запись в онлайн-проект Supabase.
5. Добавить Supabase-хранилище для общей `AppData`-модели.
6. Переключить application-сценарии на Supabase по умолчанию через `getDefaultStorage()`.
7. Проверить миграционный mapper пробным запуском и записью в онлайн-проект.
8. Запустить `pnpm check` и ручную проверку главного экрана.

## Критерии готовности

- В Supabase есть текущие студенты и связанные проекты из локальных JSON-данных.
- Количество перенесенных записей сверено с локальными JSON-файлами.
- Главный экран читает студентов из Supabase.
- Создание, редактирование и удаление студента записываются в Supabase.
- Страница студента, статусы проверки, комментарии, обновления проектов и ИИ-очередь используют общий Supabase-backend.
- При недоступном Supabase приложение показывает ошибку, а не молча пишет в JSON.
- Локальные JSON-файлы не удалены и остаются архивом старого состояния.
- `pnpm check` проходит.

## Пользовательская проверка

Владелец проекта должен открыть главный экран, убедиться, что список студентов загружается из Supabase, добавить тестового студента, обновить его карточку, удалить его после подтверждения и проверить, что изменения отражаются в онлайн-проекте Supabase.

## Риски и открытые вопросы

- Если в Supabase уже есть часть рабочих данных, миграция должна быть идемпотентной и не создавать дубли.
- Если Data API для новых таблиц ограничен настройками проекта, серверный `service_role` клиент должен явно показать ошибку чтения или записи.
- После записи JSON-снимка без `--replace` в Supabase осталось больше данных, чем в локальном архиве: 36 студентов против 35, 36 проектов против 35, 11 запусков против 10, 171 событие против 136 и 171 статус проверки против 136.
- Дополнительный онлайн-студент: `Janna`, GitHub-ссылка `https://github.com/janna-web1111/-specs-deco`, статус проекта `repository_unavailable`.
- Один дополнительный онлайн-запуск обновления от 2026-05-28 содержит 35 событий, которых нет в локальном JSON-архиве. Эти данные не удалены, потому что `--replace` не запускался без отдельного решения владельца проекта.
